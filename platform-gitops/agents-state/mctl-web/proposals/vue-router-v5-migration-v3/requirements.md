# Vue Router v5 Migration: 4.6.4 to 5.0.6

## Context
mctl-web currently uses vue-router 4.6.4 alongside Nuxt 4.x and Vue 3.5.x (see `context/architecture.md`). Vue Router 5.0.6 was released on 2026-04-22 and represents a major version milestone: v5 integrates `unplugin-vue-router` into core, delivering typed file-based routing as a first-class feature. For mctl-web, which already uses Nuxt's file-based routing convention for its three routes (`/`, `/docs`, `/privacy`), v5 provides native TypeScript type inference for route names and params without requiring a separate plugin.

The upstream project claims no breaking changes from v4 to v5, making this a lower-risk major bump than typical. ADR 0001 does not prohibit router upgrades. However, because this is a major version change, a deliberate evaluation and compatibility confirmation step is warranted before the change is applied to production.

## User stories
- AS a developer I WANT vue-router upgraded to v5 SO THAT route navigation calls are fully type-checked and mistyped route names are caught at compile time rather than at runtime.
- AS a developer I WANT file-based routing types generated by the router core SO THAT I do not need to maintain a separate `unplugin-vue-router` configuration or manually declare route types.
- AS a developer I WANT confidence that the migration does not break the existing tenant form flow or OAuth redirect handling SO THAT I can adopt v5 without risking production incidents.

## Acceptance criteria (EARS)
- WHEN vue-router is upgraded to 5.0.6 THE SYSTEM SHALL resolve all existing `<RouterLink>`, `useRouter()`, and `useRoute()` usages without TypeScript errors.
- WHEN `nuxt build` is executed after the upgrade THE SYSTEM SHALL complete without errors and produce correct prerendered output for all three routes (`/`, `/docs`, `/privacy`).
- WHEN a user navigates to any route in the deployed application THE SYSTEM SHALL serve the correct page with no JavaScript console errors related to the router.
- IF vue-router v5 introduces a behavioral change in programmatic navigation (e.g., `router.push`) THEN THE SYSTEM SHALL have explicit compatibility shims or code updates applied before the upgrade is merged.
- WHILE the Cloudflare Worker handles the GitHub OAuth callback (`/api/github/callback`) THE SYSTEM SHALL correctly redirect to the intended post-login route, unaffected by any router v5 changes (Worker routing is independent of vue-router).
- WHEN the upgrade is applied THE SYSTEM SHALL be compatible with Nuxt 4.4.x (the version targeted by the nuxt-minor-upgrade-v2 proposal).

## Out of scope
- Adopting new vue-router v5 APIs beyond what is required for compatibility (e.g., new navigation guard APIs, typed route params).
- Enabling typed file-based routing in the Nuxt config if it requires non-trivial configuration changes — that is a follow-up enhancement.
- Changes to the Cloudflare Worker routing logic.
- Upgrading Nuxt or Vue core as part of this proposal (those are tracked in nuxt-minor-upgrade-v2).
- Migrating from the existing three-route structure to a new route organization.
