## Top-3 (for spec-writer)

### 1. upgrade-to-2026-4-29: Upgrade openclaw to v2026.4.29 stable
**Category:** security
**Impact:** 5
**Effort:** 2
**Rationale:** v2026.4.29 became stable on 2026-04-30 and supersedes the existing `upgrade-to-2026-4-8` proposal — it carries all 7 April-28 CVEs (including three SSRF-class and one eval-bypass) plus security hardening (HTML tag sanitization, timing-safe credential comparison) and gateway slow-startup / stale-session recovery fixes that directly reduce restore-state probe failure risk under ADR-0002. Upgrading from 2026.3.14 eliminates the full outstanding CVE surface in one rollout through the labs → admins → ovk sequence mandated by ADR-0001. No new library dependencies are introduced, so no memory footprint increase is expected for labs.
**Source:** https://github.com/openclaw/openclaw/releases/latest

### 2. cve-addendum-ssrf-eval-bypass: Requirements addendum for CVE-2026-42423, CVE-2026-41912, CVE-2026-41914
**Category:** security
**Impact:** 4
**Effort:** 1
**Rationale:** Three CVEs published 2026-04-28 were absent from yesterday's `upgrade-to-2026-4-8` proposal requirements: CVE-2026-42423 (strictInlineEval approval-timeout bypass, allows unsanctioned eval execution), CVE-2026-41912 (SSRF policy bypass, allows navigation past configured SSRF guards), and CVE-2026-41914 (QQ Bot media SSRF, affects a channel in active use). All three are SSRF-class or sandbox-escape class threats; they must be explicitly tracked in proposal requirements for compliance and audit traceability even though the fix overlaps with the upgrade. This addendum also informs the spec for finding 1 above.
**Source:** https://www.redpacketsecurity.com/cve-alert-cve-2026-42423-openclaw-openclaw/ ; https://www.redpacketsecurity.com/cve-alert-cve-2026-41912-openclaw-openclaw/ ; https://www.redpacketsecurity.com/cve-alert-cve-2026-41914-openclaw-openclaw/

### 3. nodejs-heap-cap-labs: Cap Node.js heap in labs pod via --max-heap-size (Node 24.15.0 LTS)
**Category:** performance
**Impact:** 3
**Effort:** 1
**Rationale:** Node.js 24.15.0 LTS "Krypton" promotes `--max-heap-size` to a stable CLI option, enabling an explicit heap ceiling on the openclaw process without requiring a Node.js major version bump (the separate `nodejs-runtime-upgrade` proposal handles that). The labs tenant is documented as close to its memory limit; setting a hard heap cap prevents the pod from consuming headroom needed for OS/channel buffers and avoids OOMKill-triggered auth loss that would cascade to the restore-state probe. This is the lowest-effort mitigation available today for a live constraint. Note: the cap value must be profiled carefully — setting it too low risks V8 GC thrashing; flag this sizing step as a required task in the spec.
**Source:** https://github.com/nodejs/node/releases (Node.js 24.15.0 LTS, released 2026-04-15)
