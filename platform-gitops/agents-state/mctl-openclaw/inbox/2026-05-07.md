## Top-3 (for spec-writer)

### 1. openclaw-upgrade-cve-batch: Upgrade openclaw to 2026.5.6 to fix six open CVEs
**Category:** security
**Impact:** 5
**Effort:** 2
**Rationale:** We are running 2026.3.14 against which six CVEs are confirmed open: CVE-2026-32922 (CVSS 9.9, privilege escalation), CVE-2026-25253 "ClawBleed" (CVSS 8.8, actively exploited CSWSH), CVE-2026-44115 (CVSS 8.8, shell expansion bypass), CVE-2026-44116 (CVSS 8.6, SSRF in Zalo plugin), CVE-2026-42436 (improper access control in browser routes), and CVE-2026-43573 (SSRF policy bypass in browser interaction routes). The upstream release 2026.5.6 is available and the ADR-0001 rollout order (labs -> admins -> ovk) must be followed; the labs tenant is near its memory limit so the upgrade itself does not increase memory usage, but resource headroom should be verified before rollout. The effort is low because this is a version bump through the existing Helm/ArgoCD pipeline.
**Source:** https://github.com/openclaw/openclaw/releases/latest; CVEs at thehackerwire.com, armosec.io, blink.new, redpacketsecurity.com (see inbox entries above).

### 2. baileys-supply-chain-audit: Audit and harden Baileys dependency against lotusbail supply-chain attack
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** A malicious npm package `lotusbail` with 56,000+ downloads impersonates Baileys and exfiltrates WhatsApp credentials, session keys, and media to an attacker-controlled endpoint, then persists as a trusted device. Because mctl-openclaw uses Baileys for the WhatsApp Web channel, a developer or CI environment accidentally installing `lotusbail` instead of the legitimate `@whiskeysockets/baileys` would result in live customer session compromise in all three tenants. The mitigation requires locking the Baileys package reference in lockfiles, adding an allowlist check in CI, and publishing an advisory to the team — low effort but high blast-radius if ignored.
**Source:** https://www.bleepingcomputer.com/news/security/malicious-npm-package-steals-whatsapp-accounts-and-messages/

### 3. typescript-v6-migration: Migrate build toolchain to TypeScript 6.0.3
**Category:** refactor
**Impact:** 3
**Effort:** 3
**Rationale:** TypeScript 6.0.3 removes `moduleResolution: classic` and forbids `esModuleInterop: false`, both of which are legacy settings that many Node.js projects still carry; leaving them unaddressed means the project will eventually be blocked from upgrading at all, since TypeScript v7 will be a Go rewrite with potentially larger migration surface. The reported 20-50% build time improvement has a direct positive effect on CI pipelines for all three tenants. This finding does not conflict with any accepted ADR and has no memory impact in labs.
**Source:** https://github.com/microsoft/TypeScript/releases/latest
