## Top-3 (for spec-writer)

### 1. upgrade-to-2026-4-8: Patch four openclaw CVEs (batch, min safe version 2026.4.8)
**Category:** security
**Impact:** 5
**Effort:** 2
**Rationale:** Four CVEs published 2026-04-28 affect all three tenants running 2026.3.14: two CVSS 8.8 issues (token-role bypass, node.pair.approve auth bypass allowing exec-node privilege escalation) and two CVSS 7.1 issues (missing plugin integrity verification, gateway HTTP auth privilege escalation). The minimum upstream release that closes all four is 2026.4.8, which is a stable tag. Rolling out labs -> admins -> ovk following ADR-0001 procedure, with the s3-sync canary and restore-state probe per ADR-0002, makes this achievable at low effort.
**Source:** CVE-2026-42422, CVE-2026-42426, CVE-2026-42428, CVE-2026-42429 — https://cve.threatint.eu/CVE/CVE-2026-42422

### 2. baileys-registry-lockdown: Pin Baileys to official package and enforce registry allowlist
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** The `lotusbail` typosquatting package (56,000+ downloads) actively steals WhatsApp auth tokens and session keys — exactly the credentials we store in S3 per ADR-0002. A targeted lockfile pin to `whiskeysockets/baileys` at an exact version, combined with an npm allowlist or `publishConfig` registry scope enforcement, closes the substitution vector without touching the runtime. This is more specific than the existing `npm-supply-chain-audit` proposal (general audit) and directly protects the most sensitive state in the system (WhatsApp channel auth for ovk).
**Source:** lotusbail disclosure — https://www.bleepingcomputer.com/news/security/malicious-npm-package-steals-whatsapp-accounts-and-messages/

### 3. slack-socket-mode-update: Update @slack/socket-mode to 2.0.7 for reconnection reliability
**Category:** performance
**Impact:** 2
**Effort:** 1
**Rationale:** Version 2.0.7 terminates stale closing WebSocket connections earlier when a normal close handshake fails, reducing reconnection latency and suppressing repeated warning log noise on all three tenants. The change is a patch-level dependency bump with no API surface changes, making it low-effort and low-risk. This is not memory-additive, so the labs tenant memory constraint is not affected.
**Source:** @slack/socket-mode@2.0.7 — https://github.com/slackapi/node-slack-sdk/releases
