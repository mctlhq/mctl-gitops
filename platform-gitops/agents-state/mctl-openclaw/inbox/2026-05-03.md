## Top-3 (for spec-writer)

### 1. upgrade-to-2026-5-2: Upgrade openclaw to v2026.5.2 (security + memory + gateway)
**Category:** security
**Impact:** 5
**Effort:** 2
**Rationale:** openclaw v2026.5.2 (published 2026-05-02) supersedes the already-proposed `upgrade-to-2026-4-29`. It includes all seven CVE fixes from 2026.4.29 PLUS a new CWE-532 log-sanitization security fix (credentials currently leak into logs on our 2026.3.14 deployment), a gateway startup latency reduction that directly lowers restore-state probe failure risk (ADR-0002), and the externalization of ACPX and diagnostics-otel into opt-in packages that shrinks the base memory footprint — directly benefiting the `labs` tenant which is near its limit. Plugin runtime-deps stale roots are also pruned on upgrade, cleaning up leftover artefacts from prior versions.
**Source:** https://github.com/openclaw/openclaw/releases/tag/v2026.5.2

### 2. cwe532-log-credential-sanitization: Interim mitigation for CWE-532 credential leak in logs
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** openclaw 2026.3.14 does not redact `?password=`, `?token=` query parameters, or `Authorization:` headers before writing to logs. These values are present in live traffic for OAuth flows, Slack tokens, webhook auth, and plugin auth routes, meaning operator logs and log-aggregation pipelines (Loki, CloudWatch, etc.) may contain plaintext credentials today. The fix is delivered by the 2026.5.2 upgrade, but if the upgrade is delayed (e.g., waiting on the mandatory labs -> admins -> ovk soak cycle), an interim mitigation at the log-shipper layer should be deployed immediately. This proposal tracks that interim measure as a standalone deliverable independent of the upgrade timeline.
**Source:** https://github.com/openclaw/openclaw/releases/tag/v2026.5.2 (CWE-532 in release notes)

### 3. git-plugin-install-allowlist: Policy and allowlist for new git: plugin install feature
**Category:** security
**Impact:** 3
**Effort:** 2
**Rationale:** openclaw 2026.5.2 introduces first-class `git:` plugin installs, allowing operators to install plugins directly from arbitrary git repository URLs. Without a corresponding allow-list policy, any operator (or compromised credential) could install a malicious plugin from an attacker-controlled git repository, bypassing npm registry security controls. The existing `clawhub-skills-allowlist` proposal addresses ClawHub skill sources but does not cover git: URLs. A new policy is needed: only allowlisted git hosts/orgs (e.g., `github.com/mctlhq/*`, `github.com/openclaw/*`) should be permitted, enforced at the openclaw config level before the feature is enabled on any tenant. Not flagged as risky for `labs` memory — the policy change itself adds no runtime footprint.
**Source:** https://github.com/openclaw/openclaw/releases/tag/v2026.5.2 (git: plugin installs feature)
