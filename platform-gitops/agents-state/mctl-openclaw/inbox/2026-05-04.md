## Top-3 (for spec-writer)

### 1. emergency-cve-patch: Emergency upgrade of openclaw core to 2026.4.8
**Category:** security
**Impact:** 5
**Effort:** 3
**Rationale:** All three tenants run 2026.3.14. At least 10 CVEs published in the last 7 days affect this version, including CVE-2026-41386 (CVSS 9.1 CRITICAL — pairing privilege escalation), CVE-2026-41394 (CVSS 8.2 HIGH — unauthenticated HTTP routes with operator write scope), CVE-2026-42422/42426 (CVSS 8.8 HIGH — token/pairing role bypasses), and CVE-2026-41914 (CVSS 8.5 HIGH — SSRF in QQ Bot). All are fully remediated in 2026.4.8. Target is 2026.4.8 rather than 2026.5.2 because 2026.5.2 has known regressions (#77095 gateway crash at 12 h, #77116 Feishu incompatibility). Rollout must follow the established labs → admins → ovk pipeline.
**Source:** CVE-2026-41386, CVE-2026-41394, CVE-2026-42422, CVE-2026-41378, and 6 others — https://cve.threatint.eu/CVE/CVE-2026-41386; https://cve.threatint.eu/CVE/CVE-2026-41394

### 2. ws-dos-rate-limit: Network-level rate limit for unauthenticated WebSocket upgrades (CVE-2026-41399 + defence-in-depth)
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** CVE-2026-41399 describes unbounded concurrent unauthenticated WebSocket upgrades as a DoS vector; the patched version boundary is unconfirmed, so even after the 2026.4.8 upgrade a network-layer control is prudent. Adding a pre-authentication connection budget at the ingress (Kubernetes NetworkPolicy + ingress rate limit annotation) is low-effort, requires no openclaw code change, zero memory overhead in `labs`, and provides defence-in-depth regardless of upstream patch status.
**Source:** CVE-2026-41399 — https://www.redpacketsecurity.com/cve-alert-cve-2026-41399-openclaw-openclaw/

### 3. channel-lib-updates: Bump discord.js to 14.26.4 and @slack/socket-mode to 2.0.7
**Category:** feature
**Impact:** 3
**Effort:** 2
**Rationale:** discord.js 14.26.4 (2026-05-01) fixes silent loss of DMs in uncached DMChannels — a correctness regression that silently drops user messages. `@slack/socket-mode` 2.0.7 fixes stale WebSocket connections that stay open after Slack server-side close, causing reconnect delays and "pong wasn't received" noise. Both are point-release dependency bumps with no API surface changes and no expected memory footprint increase in `labs`. No ADR conflicts.
**Source:** https://github.com/discordjs/discord.js/releases/latest; https://github.com/slackapi/node-slack-sdk/releases/latest
