## Top-3 (for spec-writer)

### 1. openclaw-upgrade-2026-5-12: Upgrade openclaw to v2026.5.12 to patch four disclosed CVEs
**Category:** security
**Impact:** 5
**Effort:** 3
**Rationale:** Four openclaw CVEs were publicly disclosed in May 2026 — CVE-2026-44112 (CVSS 9.6, sandbox write escape), CVE-2026-44113 (CVSS 7.7, sandbox read escape), CVE-2026-44115 (CVSS 8.8, shell injection in OpenShell), and CVE-2026-44118 (CVSS 7.8, gateway impersonation) — all present in our running version 2026.3.14. The stable upstream v2026.5.12 is the fix target and also moves WhatsApp, Slack, Bedrock, and Anthropic Vertex dependency cones out of the core runtime, which directly reduces memory footprint in `labs` (currently near its limit) — making this upgrade a security fix and a memory win simultaneously. Rollout must follow ADR 0001 order: `labs` first with observation, then `admins`, then `ovk`; the s3-sync canary and restore-state probe (ADR 0002) must be honoured throughout.
**Source:** https://thehackernews.com/2026/05/four-openclaw-flaws-enable-data-theft.html ; https://github.com/openclaw/openclaw/releases

### 2. nodejs-dos-cve-patch: Upgrade Node.js to v22.22.3 to fix two High-severity remote DoS CVEs
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** CVE-2026-21637 (missing try/catch in `loadSNI()`) and CVE-2026-21710 (`__proto__` header crash in `req.headersDistinct`) both allow unauthenticated remote DoS against Node.js 22.x, which is our runtime. Both are fixed in v22.22.3 (released May 13, 2026), the current stable LTS. Upgrading the base image Node.js version is a low-effort Dockerfile/gitops change that closes two crash vectors across all three tenants with no memory footprint increase.
**Source:** https://nodejs.org/en/blog/vulnerability/march-2026-security-releases ; https://github.com/nodejs/node/releases

### 3. lotusbail-supply-chain-audit: Audit all tenant lock files and CI pipelines for the malicious `lotusbail` package
**Category:** security
**Impact:** 3
**Effort:** 1
**Rationale:** The npm package `lotusbail` (a typosquatting Baileys fork with 56,000+ downloads) exfiltrates WhatsApp credentials, session keys, and message content. We use WhiskeySockets/Baileys for the WhatsApp channel; any developer or CI script that accidentally named the wrong package could have introduced `lotusbail` into a lock file. A one-time grep of `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` across all three tenant gitops repos and the CI pipeline is sufficient to close this risk, with no runtime changes required.
**Source:** https://www.securityweek.com/npm-package-with-56000-downloads-steals-whatsapp-credentials-data/
