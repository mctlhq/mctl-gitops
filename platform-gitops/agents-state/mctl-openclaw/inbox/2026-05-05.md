## Top-3 (for spec-writer)

### 1. upgrade-to-2026-5-3: Upgrade all tenants to openclaw v2026.5.3 to patch six open HIGH CVEs
**Category:** security
**Impact:** 5
**Effort:** 3
**Rationale:** We are on 2026.3.14 and are confirmed vulnerable to six CVEs with CVSS scores ranging from 5.4 to 8.8 HIGH, including an auth bypass on unauthenticated plugin-auth routes (CVE-2026-41394), a role bypass in token rotation (CVE-2026-42422), a directory-traversal + command-injection in device pairing (CVE-2026-33579), an exec allowlist bypass (CVE-2026-41390), a webhook replay bypass (CVE-2026-41395), and a Slack prompt injection (CVE-2026-41358). v2026.5.3 is the latest stable release and resolves all of them; v2026.5.2 had known regressions but v2026.5.3 is a follow-up patch addressing those. Rollout must follow labs → admins → ovk per ADR0001, with S3 canary and restore-state probe kept intact per ADR0002; note that the lazy-load startup optimization in v2026.5.3 may reduce pod startup memory pressure in labs but this must be validated before the labs rollout is marked safe.
**Source:** CVE-2026-42422, CVE-2026-41394, CVE-2026-41395, CVE-2026-41390, CVE-2026-33579, CVE-2026-41358 — https://github.com/openclaw/openclaw/releases/latest

### 2. baileys-lockfile-audit: Audit lockfile to confirm Baileys dependency is not the malicious "lotusbail" package
**Category:** security
**Impact:** 4
**Effort:** 1
**Rationale:** The "lotusbail" npm package (a Baileys fork with ~56 000 downloads) exfiltrates WhatsApp auth tokens, messages, contacts, and media via a socket wrapper; our architecture stores WhatsApp auth tokens in S3 (ADR0002) and these would be directly at risk if the wrong package were resolved. The audit requires only a lockfile inspection to confirm the resolved package name and integrity hash for the Baileys dependency across all three tenants, making effort minimal relative to the potential impact of a full WhatsApp session compromise. If "lotusbail" is found in any lockfile it must be removed immediately before any other rollout proceeds.
**Source:** https://www.theregister.com/2025/12/22/whatsapp_npm_package_message_steal/

### 3. discord-js-dm-fix: Update discord.js to v14.26.4 to restore DM receipt in uncached DMChannels
**Category:** performance
**Impact:** 2
**Effort:** 1
**Rationale:** We use `discord.js` from `discordjs/discord.js` (confirmed in architecture) and v14.26.4 is a targeted bug-fix release that restores DM receipt handling for uncached DMChannels — a regression that silently drops incoming direct messages, making the Discord channel unreliable for `ovk` and `labs` users. The change is a single-PR patch with minimal blast radius and no known memory footprint increase, so it is safe to roll out through labs first without triggering the labs memory risk flag.
**Source:** https://github.com/discordjs/discord.js/releases/latest (PR #11495)
