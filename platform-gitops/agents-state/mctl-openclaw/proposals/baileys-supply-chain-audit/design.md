# Design: baileys-supply-chain-audit

## Current state

All three tenants (`admins`, `labs`, `ovk`) share a Node.js + TypeScript workspace rooted at the `mctl-openclaw` service directory. `@whiskeysockets/baileys` is declared as a direct dependency (current resolved version: 2026.3.14). The dependency is pulled in via npm install during Docker image builds and during local development.

There is presently no CI step that:
- Verifies the package name resolves only to `@whiskeysockets/baileys` and not to any alias or impersonator.
- Validates that the integrity hash in the lockfile has not been tampered with or replaced.
- Rejects lockfile entries whose package names partially match known Baileys-impersonating patterns.

The WhatsApp auth tokens and linked-device session keys generated by Baileys are persisted to S3 per ADR-0002. There is no documented procedure for revoking those sessions in the event of a supply-chain compromise. The absence of a revocation runbook means the blast radius of a compromise extends for an indeterminate time after detection.

Reference: `context/architecture.md`, ADR-0001 (rollout order), ADR-0002 (S3 session storage).

## Proposed solution

### 1. Lockfile audit

Before merging any pull request and as a scheduled daily job, a script (`scripts/audit-baileys-lockfile.sh`) inspects `package-lock.json` (or `yarn.lock` if present) and asserts:

- The resolved package for the Baileys module identifier is exactly `@whiskeysockets/baileys`.
- No entry in the lockfile has a name matching the regex `/(lotusbail|lotus-bail|baileys(?!.*whiskeysockets))/i` or any similar known alias.
- The integrity field for `@whiskeysockets/baileys` is non-empty and matches the expected hash stored in `config/expected-integrity.json` (a file checked into the repo).

The script exits non-zero on any failure, causing the CI job to fail and emitting a structured error message with the offending lockfile line number and field value.

### 2. CI guard (lockfile-lint step)

A new CI job named `supply-chain-guard` is added to the pipeline (GitHub Actions workflow or equivalent) and runs in parallel with existing lint and test jobs. It executes:

1. `npm ci --ignore-scripts` to install from the lockfile without running lifecycle hooks (prevents malicious `postinstall` scripts from executing during the CI audit step itself).
2. `scripts/audit-baileys-lockfile.sh` as described above.
3. `npx lockfile-lint --path package-lock.json --allowed-hosts npm --validate-package-names` to catch unexpected registry sources.

The job is marked `required` so it blocks merges. It runs on every push to `main`, every pull request targeting `main`, and on a nightly schedule.

### 3. Revocation runbook

A Markdown file `ops/runbooks/baileys-compromise-revocation.md` documents the following procedure:

1. Identify the affected tenant(s) by checking S3 session paths (per ADR-0002 naming convention).
2. For each affected tenant, navigate to the WhatsApp linked-devices management page (or equivalent admin UI) and revoke all linked devices.
3. Restart the openclaw pod for the affected tenant to clear in-memory session state.
4. Trigger a fresh QR-code pairing flow for the tenant.
5. Rotate the S3 bucket credentials used by that tenant (IAM role or access key) to invalidate any exfiltrated S3 credentials.
6. File an incident, capture the timeline, and notify affected customers per the incident response policy.

The runbook is linked from the main `README.md` and from the `context/decisions/` index.

### 4. Dependency pinning

The `package.json` entry for `@whiskeysockets/baileys` is changed from any semver range (e.g., `^2026.3.14`) to the exact version `2026.3.14`. The lockfile is committed with the full `integrity` field populated. A note is added to `CONTRIBUTING.md` explaining that Baileys must always be pinned and that any version bump requires:

- Manually verifying the new version's changelog and git tag signature on `github.com/whiskeysockets/baileys`.
- Updating `config/expected-integrity.json` with the new hash.
- Following the `labs` → `admins` → `ovk` rollout sequence per ADR-0001.

No wildcard (`*`) or open-ended ranges are permitted for this package. The CI guard enforces this.

## Alternatives

### Option A: Replace Baileys with an alternative WhatsApp Web library

Considered briefly. No production-grade alternative with equivalent feature coverage exists at this time. A migration would require re-implementing the session-management and media-handling layers, introduce significant regression risk, and take substantially longer than the targeted remediation window. Deferred out of scope.

### Option B: Do nothing / accept the risk

Unacceptable. Lotusbail has been downloaded approximately 56,000 times, demonstrating real-world reach. A compromise of the `ovk` tenant S3 session store would expose production customer data. The cost of remediation is low relative to the potential breach impact.

### Option C: Use a private npm registry mirror with allowlist

Feasible as a longer-term hardening measure, but adds operational complexity (mirror maintenance, authentication). The CI guard described in the proposed solution delivers equivalent protection at the point of package resolution with far less operational overhead. The private mirror option is tracked as a future enhancement in the broader supply-chain initiative.

## Platform impact

### Migrations

- `package.json`: change `@whiskeysockets/baileys` version specifier to exact pin `2026.3.14`.
- `package-lock.json`: re-lock to confirm no extraneous entries; commit the updated lockfile.
- Add `config/expected-integrity.json` to the repository.
- Add `scripts/audit-baileys-lockfile.sh`.
- Add `ops/runbooks/baileys-compromise-revocation.md`.
- Add the `supply-chain-guard` CI job definition.

### Backward compatibility

The Baileys API surface does not change. Pinning the exact version (which is already the installed version) introduces no runtime behavior change.

### Resource impact

The CI guard step runs only in the pipeline and does not execute inside the deployed pods. There is zero memory or CPU impact on the `labs` tenant or any other running deployment. This proposal is safe for `labs`.

### Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| CI guard produces false positives on legitimate package names | Low | Regex pattern reviewed against npm registry; only exact known aliases targeted |
| Lockfile hash changes due to a legitimate upstream patch | Medium | `config/expected-integrity.json` update is a required step in the version-bump process; CI error message instructs operator clearly |
| Revocation runbook becomes stale | Low | Runbook is linked from the incident-response checklist and reviewed quarterly |
| `labs` memory limit breached | None | No new runtime components introduced |
