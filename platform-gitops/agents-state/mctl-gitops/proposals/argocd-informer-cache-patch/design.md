# Design: argocd-informer-cache-patch

## Current state

ArgoCD is pinned to a version prior to v3.3.8 in the bootstrap chart located under
`platform-gitops/bootstrap/` (see `context/architecture.md`). The bootstrap chart is the
App-of-Apps root; ArgoCD reconciles itself via the `apps` ApplicationSet which watches
`platform-gitops/services/*/*`.

Three bugs are present in the currently pinned version:

| Bug | Observable symptom | Risk level |
|-----|--------------------|------------|
| Stale informer cache in RevisionMetadata handler | ArgoCD may trigger a sync or a no-op based on a cluster-state snapshot that is several seconds to minutes old | High — can cause spurious rollbacks across all tenant apps |
| Core-mode sync failure when `server.secretkey` absent | Applications stuck in `OutOfSync` indefinitely after a secret rotation or fresh cluster bootstrap | Medium |
| Autosync event message format regression | Alert rules / log parsers that match the event string produce false positives or miss events | Low |

The platform uses an App-of-Apps pattern (ADR `0001-app-of-apps-pattern.md`): the bootstrap
chart spawns three ApplicationSets (`apps`, `tenants`, `openclaw-skills`). An incorrect
sync decision by ArgoCD cascades to all downstream Applications generated by these sets.

## Proposed solution

Pin ArgoCD to **v3.3.8** in the bootstrap chart. This is a single-line version bump; no
API changes, no CRD schema migrations, no new components.

### Change surface

```
platform-gitops/bootstrap/
  Chart.yaml            # or values.yaml — bump appVersion / image tag
  values.yaml           # argocd.image.tag: "v3.3.8"  (exact key depends on chart)
```

If ArgoCD is managed via the official `argo-cd` Helm chart as a dependency:

```yaml
# platform-gitops/bootstrap/Chart.yaml
dependencies:
  - name: argo-cd
    version: "x.y.z"   # chart version that packages appVersion v3.3.8
    repository: https://argoproj.github.io/argo-helm
```

If ArgoCD is pinned via an image override in a values file:

```yaml
# platform-gitops/bootstrap/values.yaml
argo-cd:
  global:
    image:
      tag: v3.3.8
```

The exact mechanism is confirmed during task 1 (audit). The principle is identical in both
cases: change one version reference, commit to git, ArgoCD self-reconciles.

### Reconciliation flow after the commit

1. ArgoCD detects a drift between the desired state (v3.3.8) and the running state.
2. The bootstrap Application performs a Helm sync; the ArgoCD Deployment receives the new
   image tag.
3. Kubernetes performs a rolling update of the ArgoCD server and application-controller
   pods.
4. Once healthy, the new ArgoCD pods take over reconciliation with the patched informer
   cache and core-mode sync logic.

### Why self-reconciliation is safe here

ArgoCD supports in-place self-upgrades for patch bumps. The rolling update strategy ensures
at least one pod is running at all times. In-flight sync operations that were already
dispatched before the rolling update are completed or retried by the new pod transparently.

## Alternatives

### A. Skip v3.3.8 and wait for v3.4.x

Rejected. The stale informer cache bug is actively risky on the App-of-Apps topology.
Waiting for the next minor release (unknown ETA) extends the exposure window unnecessarily.
The v3.3.8 release notes explicitly describe these as targeted bug fixes with no breaking
changes.

### B. Patch the informer cache bug with a custom image overlay

Rejected. Maintaining a custom ArgoCD image introduces an ongoing build and security-scan
burden. The upstream patch is already available in v3.3.8; building a custom image for a
one-line config change is disproportionate effort and increases supply-chain risk.

### C. Disable RevisionMetadata calls to avoid stale cache hits

Rejected. RevisionMetadata is used internally for sync-wave ordering and diff computation.
Disabling it would degrade sync fidelity and is not a supported configuration.

## Platform impact

### Migrations

None. v3.3.8 is a patch release. No CRD schema changes, no API version bumps, no
migration jobs required.

### Backward compatibility

Fully backward compatible. The only observable difference is the correction of the three
bugs listed above. The revert of the autosync event message format restores the format that
was in place before the regression — existing alert rules remain valid.

### Resource impact

**`admins` tenant:** neutral. Pod count and resource requests/limits are unchanged.

**`labs` tenant:** neutral. This patch upgrade does not introduce new sidecars, init
containers, or persistent volumes. `labs` memory usage is unaffected. No risk of pushing
`labs` toward its memory limit.

### Risks and mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| ArgoCD self-sync loop during rolling update causes a double-sync of child apps | Low | Rolling update keeps one pod alive; ArgoCD's leader-election prevents dual reconciliation |
| v3.3.8 Helm chart version is not yet indexed in the repo cache | Low | Pin the chart digest or use `helm repo update` in the CI pipeline before sync |
| Rollback increases stale-cache exposure again | Low | Rollback is fast (one git revert + sync); exposure window is bounded by time-to-detect |
