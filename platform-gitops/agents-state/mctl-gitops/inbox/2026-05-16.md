## Top-3 (for spec-writer)

### 1. eso-confused-deputy-patch: ESO Privilege Escalation via Confused-Deputy (CVE-2026-42876)
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** CVE-2026-42876 affects ESO versions 0.1.0–2.4.0, meaning the target version of the existing `eso-cross-namespace-bypass-patch` proposal (helm-chart-2.4.1) remains in the vulnerable range. A user with only ExternalSecret create permissions can force the ESO controller to generate a Kubernetes Secret containing a long-lived Service Account token, bypassing the principle of least privilege and potentially enabling cross-tenant privilege escalation. The fix is ESO v2.5.0, which also ships the new healthz liveness probe endpoint — the upgrade is a drop-in with no CRD migrations needed.
**Source:** CVE-2026-42876 — https://dailycve.com/external-secrets-privilege-escalation-cve-2026-42876-moderate/

### 2. argocd-v3-4-2-stability-patch: ArgoCD v3.4.2 Permission Validator Panic and Go Security Updates
**Category:** security
**Impact:** 3
**Effort:** 1
**Rationale:** ArgoCD v3.4.2 (released 2026-05-12) fixes a permission validator panic that can cause the ArgoCD server to crash under specific RBAC evaluation conditions, and it bundles updated Go runtime binaries that address known Go-level CVEs. The existing `argocd-v3-4-upgrade-plan-v2` proposal targets v3.4.0 GA; v3.4.2 is two patch releases ahead and is the correct pinned target to avoid shipping a known crasher into production. The effort is minimal — a one-line version bump in the bootstrap manifest — and the stability gain directly protects reconciliation availability across both tenants.
**Source:** argoproj/argo-cd v3.4.2 release — https://github.com/argoproj/argo-cd/releases

### 3. argo-rollouts-v1-9-0-upgrade: Argo Rollouts v1.9.0 BlueGreen Analysis Premature-Success Fix
**Category:** security
**Impact:** 3
**Effort:** 2
**Rationale:** Argo Rollouts v1.9.0 corrects a correctness bug where a BlueGreen analysis step is incorrectly marked as successful when the new ReplicaSet becomes unsaturated mid-rollout, allowing a degraded revision to be promoted to stable. This is a silent data-plane risk: a bad deployment can be considered healthy and fully promoted without any human or automated gate catching the regression. No existing proposal addresses Argo Rollouts at all, and the platform uses Argo Rollouts for progressive delivery. The fix is a version pin bump with no CRD schema changes in this release.
**Source:** argoproj/argo-rollouts v1.9.0 release — https://github.com/argoproj/argo-rollouts/releases
