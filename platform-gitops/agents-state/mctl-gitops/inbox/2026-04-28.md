## Source: cve
- **CVE-2026-40886 (HIGH 7.7) — Argo Workflows controller crash loop via malformed pod annotation** — An unchecked array index in `podGCFromPod()` causes a controller-wide panic when a workflow pod carries a malformed `workflows.argoproj.io/pod-gc-strategy` annotation; panic occurs outside recover() scope, crashing the entire controller and causing a restart loop until the pod is manually deleted. Affects versions 3.6.5–3.6.19 and 3.7.0–3.7.13; fixed in v3.7.14 and v4.0.5. https://advisories.gitlab.com/golang/github.com/argoproj/argo-workflows/v3/CVE-2026-40886/
- **CVE-2026-31892 (HIGH) — Argo Workflows WorkflowTemplate security bypass via podSpecPatch** — Authenticated users can bypass Strict/Secure reference mode restrictions using the `podSpecPatch` field, enabling privileged container execution and potential Kubernetes node compromise. https://advisories.gitlab.com/pkg/golang/github.com/argoproj/argo-workflows/CVE-2026-31892/
- **CVE-2026-28229 — Argo Workflows unauthorized access to WorkflowTemplate/ClusterWorkflowTemplate** — Any client presenting `Authorization: Bearer nothing` can retrieve WorkflowTemplates and ClusterWorkflowTemplates, potentially leaking sensitive template content including embedded Secret manifests. https://advisories.gitlab.com/pkg/golang/github.com/argoproj/argo-workflows/v3/CVE-2026-28229/
- **CVE-2026-5052 / HCSEC-2026-06 — Vault PKI ACME SSRF via attacker-controlled DNS** — Vault's PKI ACME http-01 and tls-alpn-01 challenge validation did not reject local network targets, allowing an attacker with DNS control to redirect validation requests to internal services and cause information disclosure. Affects Vault CE and Enterprise 1.14.0–1.21.4; fixed in v2.0.0 (CE), v1.21.5, v1.20.10, v1.19.16 (Enterprise); published April 16. https://discuss.hashicorp.com/t/hcsec-2026-06-vault-vulnerable-to-server-side-request-forgery-in-acme-challenge-validation-via-attacker-controlled-dns/77343
- **CVE-2026-42151 — Prometheus OAuth client_secret exposed via /-/config endpoint** — AzureAD remote-write `client_secret` was logged in plaintext through the config API endpoint; fixed in Prometheus v3.11.3. https://github.com/prometheus/prometheus/releases/tag/v3.11.3
- **CVE-2026-42154 — Prometheus remote-read DoS via oversized snappy-compressed request** — Server did not reject snappy-compressed remote-read requests whose declared decoded length exceeded the decode limit, enabling a DoS condition; fixed in Prometheus v3.11.3. https://github.com/prometheus/prometheus/releases/tag/v3.11.3
- **Helm GHSA-hr2v-4r36-88hr — Chart extraction output directory collapse** — Malicious `Chart.yaml` name containing dot-segments could collapse the extraction output directory; fixed in Helm v4.1.4. https://github.com/helm/helm/releases/tag/v4.1.4
- **Helm GHSA-q5jf-9vfq-h4h7 — Plugin verification fails-open when .prov is missing** — Unsigned plugins could be installed without error when the provenance file is absent; fixed in Helm v4.1.4. https://github.com/helm/helm/releases/tag/v4.1.4
- **Helm GHSA-vmx8-mqv2-9gmg — Path traversal in plugin metadata version allows arbitrary file write** — Malformed plugin metadata version field enables writing files outside the Helm plugin directory; fixed in Helm v4.1.4. https://github.com/helm/helm/releases/tag/v4.1.4

## Top-3 (for spec-writer)

### 1. argo-workflows-cve-patch-v2: Argo Workflows — three HIGH CVEs patched in v3.7.14 / v4.0.5
**Category:** security
**Impact:** 5
**Effort:** 2
**Rationale:** Three simultaneous HIGH-severity CVEs affect the Argo Workflows controller we run for all platform pipelines: a crash-loop via malformed pod annotation (CVE-2026-40886, CVSS 7.7), a privilege-escalation bypass via podSpecPatch (CVE-2026-31892), and unauthenticated WorkflowTemplate read exposing embedded secrets (CVE-2026-28229). Combined, these allow denial-of-service, node compromise, and secret leakage — the highest possible blast radius on our CI/CD layer. A version bump to v3.7.14 or v4.0.5 closes all three.
**Source:** https://advisories.gitlab.com/golang/github.com/argoproj/argo-workflows/v3/CVE-2026-40886/ ; https://advisories.gitlab.com/pkg/golang/github.com/argoproj/argo-workflows/CVE-2026-31892/ ; https://advisories.gitlab.com/pkg/golang/github.com/argoproj/argo-workflows/v3/CVE-2026-28229/

### 2. vault-ssrf-acme-patch: Vault PKI ACME SSRF via attacker-controlled DNS (CVE-2026-5052)
**Category:** security
**Impact:** 4
**Effort:** 3
**Rationale:** Vault is the single secret store for all tenants (`secrets.mctl.ai`, `vault-backend` ClusterSecretStore); a confirmed SSRF in PKI ACME challenge validation allows an attacker with DNS control to probe internal services through Vault's outbound HTTP calls. The fix ships in Vault CE v2.0.0 which carries breaking changes (Docker helper, auth endpoint requirements), so the upgrade path needs careful review — higher effort than a routine patch but non-negotiable given our full Vault dependency.
**Source:** https://discuss.hashicorp.com/t/hcsec-2026-06-vault-vulnerable-to-server-side-request-forgery-in-acme-challenge-validation-via-attacker-controlled-dns/77343

### 3. argocd-informer-cache-patch: ArgoCD v3.3.8 — stale informer cache and core-mode sync fixes
**Category:** performance
**Impact:** 3
**Effort:** 1
**Rationale:** The stale informer cache bug in RevisionMetadata can cause ArgoCD to act on outdated cluster state, leading to incorrect sync decisions on our App-of-Apps setup; the core-mode sync fix is relevant because we rely on ArgoCD as the sole reconciliation engine for all tenant services. This is a low-effort patch bump with no breaking changes and directly stabilises the reconciliation loop that underpins every service deployment.
**Source:** https://github.com/argoproj/argo-cd/releases/tag/v3.3.8
