# Design: scaffolder-secret-leak

## Текущее состояние
Согласно `context/architecture.md`, mctl-portal использует Backstage с плагином
`plugin-scaffolder-backend`. Текущая версия пакета относится к диапазону 3.1.0–3.1.4
(до исправления), уязвимому согласно CVE-2026-32237. Dry-run endpoint scaffolder'а
доступен аутентифицированным пользователям через Dex JWT SSO. Vault-секреты смонтированы
в backend-pod как переменные среды и файлы через ExternalSecret — это означает, что
`process.env` в рантайме содержит Vault token, Postgres DSN и GitHub App credentials.

Проблема заключается в том, что при сериализации ответа dry-run endpoint вложенные
JSON-объекты не проходят рекурсивную редакцию: только top-level ключи маскируются, но
значения внутри вложенных структур возвращаются в открытом виде.

## Предлагаемое решение

### Обновление пакета (один PR с CVE-2026-24046)

CVE-2026-32237 и CVE-2026-24046 закрываются одним и тем же релизом:
`plugin-scaffolder-backend` >= 3.1.1. Таким образом, оба CVE закрываются единым PR,
который также обновляет `@backstage/backend-defaults` до ^0.12.2 (требование
CVE-2026-24046).

Обновляемые пакеты в рамках одного PR:
```
@backstage/backend-defaults  ^0.12.2   (закрывает CVE-2026-24046)
plugin-scaffolder-backend    ^3.1.1    (закрывает CVE-2026-24046 + CVE-2026-32237)
```

Апстрим-фикс в 3.1.1 реализует рекурсивный redactor для dry-run ответа: функция
обходит весь JSON-граф ответа и заменяет значения по списку известных чувствительных
ключей (`token`, `password`, `dsn`, `credentials`, `secret` и т.д.) на маркер
`[REDACTED]`. Дополнительно — фильтрация по значениям, соответствующим паттернам
известных форматов токенов.

### Почему один PR, а не два
- Тот же пакет (`plugin-scaffolder-backend`) закрывает оба CVE.
- Два отдельных PR означали бы два production-деплоя, два прогона playwright, два
  ArgoCD sync — при идентичном изменении `yarn.lock`.
- Атомарный патч снижает риск незакрытого окна между деплоями.

### Дополнительная мера: Vault secret rotation
После деплоя патча рекомендуется ротировать секреты, смонтированные в backend-pod, на
случай если dry-run endpoint эксплуатировался до патча. Это находится за рамками данного
PR, но должно быть выполнено параллельно как incident response action.

## Альтернативы

**A. Добавить собственный middleware для фильтрации dry-run ответов**
Потребует поддержки custom-кода для отслеживания списка чувствительных ключей и паттернов.
Апстрим-патч решает задачу надёжнее и без бремени maintenance. Отклонено.

**B. Отключить dry-run endpoint полностью**
Dry-run используется разработчиками для отладки шаблонов. Отключение снижает DX без
необходимости, поскольку апстрим-патч доступен. Отклонено.

**C. Ограничить доступ к dry-run через Backstage permission framework**
Снижает поверхность атаки, но не устраняет уязвимость для пользователей с легитимным
доступом. Может применяться как дополнительный контроль, но не как основной фикс.
Отклонено как единственная мера.

## Влияние на платформу

### Migration/миграции
Нет миграций схемы или данных. Изменения ограничены `yarn.lock` и `package.json`.

### Backward compatibility
`plugin-scaffolder-backend` 3.1.1 — patch-релиз; публичный API dry-run endpoint не
меняется. Ответ будет содержать `[REDACTED]` вместо значений секретных полей — это
намеренное изменение поведения, ожидаемое по spec.

### Resource impact
Рекурсивный redactor добавляет незначительную CPU-нагрузку при dry-run вызовах
(обход JSON-графа). Влияние на память пренебрежимо мало. Тенант `labs` не затронут.

### Риски и митигации
| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Секреты уже утекли до деплоя патча | Неизвестна | Ротировать все секреты backend-pod после деплоя; проверить audit log Vault |
| Peer-dependency конфликт при совместном обновлении пакетов | Средняя | `yarn backstage-cli versions:check` перед merge |
| Регрессия dry-run для легитимных шаблонов | Низкая | Playwright тест dry-run с эталонным шаблоном |
