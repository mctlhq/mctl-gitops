## Top-3 (for spec-writer)

### 1. catalog-unprocessed-security-patch: Backstage v1.50.4 catalog security fixes
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** Backstage v1.50.4 ships security fixes directly in `@backstage/plugin-catalog-backend-module-unprocessed` and its companion packages, which are part of the catalog stack we actively use. No existing proposal covers these specific catalog-module CVEs, and a targeted patch upgrade (staying on the 1.50.x line) is low-risk with no breaking changes. Leaving a known-patched security release unapplied violates platform security SLA.
**Source:** https://github.com/backstage/backstage/releases/tag/v1.50.4

### 2. catalog-facets-perf-fix: Backstage v1.50.3 catalog facets performance regression fix
**Category:** performance
**Impact:** 3
**Effort:** 2
**Rationale:** The v1.50.3 release resolves a confirmed performance regression in the catalog facets endpoint, which backs the search and filtering UI used by every developer on the platform. Slow facet queries degrade the catalog browsing experience and can increase backend CPU under load. The fix is a patch-level bump on the same 1.50.x line as the security patch above, so both can be applied together, reducing deployment overhead.
**Source:** https://github.com/backstage/backstage/releases/tag/v1.50.3

### 3. yarn-enable-scripts-default: Yarn v4.14.0 enableScripts default change evaluation
**Category:** refactor
**Impact:** 3
**Effort:** 2
**Rationale:** Yarn Berry v4.14.0 flips `enableScripts` to `false` by default, which silently prevents install-time scripts from running on `yarn install`. Backstage's monorepo relies on several packages with postinstall scripts (e.g., native bindings, playwright browser downloads). If the team upgrades Yarn without auditing this change, CI pipelines and local builds can break with opaque errors. A deliberate review and explicit opt-in or per-package allowlist is needed before upgrading.
**Source:** https://github.com/yarnpkg/berry/releases/tag/4.14.0

## Dropped
- **CVE-2026-24046 (Scaffolder symlink traversal):** Fully covered by the accepted `scaffolder-symlink-traversal-v2` proposal — no new content.
- **CVE-2026-23869 (React Server Components DoS):** Affects `react-server-dom-webpack/turbopack/parcel` (Next.js App Router RSC). Backstage uses React as a client-side SPA and does not use React Server Components or App Router; the vulnerable packages are not in our dependency tree.
- **CVE-2026-31431 (Linux kernel LPE):** OS/infrastructure-level kernel vulnerability; remediation is a node OS patch managed by the platform infra team, not the mctl-portal service. Out of scope for this service.
- **Node.js March 2026 CVEs (CVE-2026-21710 et al.):** Already addressed by the `nodejs22-security-batch` proposal targeting Node.js v22.22.2.
- **TypeScript v6.0 / v6.0.3:** Already covered by the `typescript-6-upgrade` proposal.
- **Backstage v1.51.0-next.1:** Pre-release / next channel; per ADR-0001, do not propose major bumps on or shortly after release day. Community-plugins compatibility lag applies.
- **Node.js v22.22.1:** Performance-only release (buffer ops, assertion comparisons); superseded in priority by the security-focused `nodejs22-security-batch` (v22.22.2), which also incorporates these changes.
- **Playwright v1.59.x:** Adds Screencast API and fixes a Windows codegen regression. e2e tooling improvement; low platform impact and no security relevance. Nice-to-have, does not make Top-3.
- **Prettier v3.8.3:** SCSS `if()` trailing-comma fix; minimal impact on this codebase. Nice-to-have.
- **Yarn v4.14.1 (EBADF fix):** Only relevant for Node.js v24.15+ compatibility; we currently run Node.js 22 LTS, so no immediate exposure.
