## Top-3 (for spec-writer)

### 1. backstage-cve-44374: Upgrade Backstage to v1.50.4 to patch catalog permission bypass
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** CVE-2026-44374 (CVSS 4.3) causes the catalog-unprocessed-entities plugin to expose entity data to any authenticated user, bypassing ownership checks and leaking cross-tenant data. Both affected plugins (`plugin-catalog-backend-module-unprocessed`, `plugin-catalog-unprocessed-entities`) are part of the Backstage stack we run. v1.50.4 is a security-only patch released April 29 — today is May 7, giving 8 days of community-plugins soak time, satisfying the ADR's "wait ~a week" constraint.
**Source:** https://advisories.gitlab.com/npm/@backstage/plugin-catalog-unprocessed-entities/CVE-2026-44374/ / https://github.com/backstage/backstage/releases/latest

### 2. node22-security-patch: Upgrade Node.js runtime to v22.22.0 (6-CVE security batch)
**Category:** security
**Impact:** 4
**Effort:** 1
**Rationale:** Node.js 22.22.0 patches six CVEs including a TLS error-handler bypass (CVE-2025-59465) and an HTTP/2 memory leak on excessive WINDOW_UPDATE frames (CVE-2026-21637); both vectors are exploitable in a network-accessible backend like Backstage. The upgrade is a drop-in patch within the already-pinned LTS line (22.x) with no API removals, making effort minimal. This concerns the tenant `admins` only; no impact on `labs` memory.
**Source:** https://nodejs.org/en/blog/release/v22.22.0 / https://nodejs.org/en/blog/vulnerability/march-2026-security-releases

### 3. react-cve-23864-audit: Verify and remediate React transitive dependency for CVE-2026-23864
**Category:** security
**Impact:** 3
**Effort:** 2
**Rationale:** CVE-2026-23864 (CVSS 7.5, High) allows DoS via memory exhaustion against React 19.x when Server Components are enabled; Backstage itself does not use RSC, but the transitive React version in the monorepo must be confirmed to be outside the affected range (19.0.4–19.2.4) or locked to a patched version. If the React version falls inside the affected range, a `resolutions` entry in `package.json` is the minimal fix. The CVSS score is high enough to warrant an explicit verification task even absent confirmed exploitation.
**Source:** https://www.akamai.com/blog/security-research/cve-2026-23864-react-nextjs-denial-of-service
