## Repo: mctl-gitops  (prod-version: unverified)

### Commit 1e3f42f — fix(base-service): add command/args support to rollout.yaml
- **Date:** 2026-05-02
- **Type:** fix (parity fix — new user-visible config knob for Rollout users)
- **User-visible effect:** The base-service Helm chart's `rollout.yaml` template now
  supports `command` and `args` values (matching parity with `deployment.yaml`, which
  already had them). Operators who run services under Argo Rollouts (canary/blue-green
  strategies) can now override the container entrypoint via standard Helm values:
  ```yaml
  command: ["/app/my-binary"]
  args: ["--port=8080"]
  ```
  Previously these fields were silently ignored in Rollout mode — the container would
  start with the default image entrypoint regardless. The diff is 8 lines added to
  `platform-gitops/helm-charts/base-service/templates/rollout.yaml`.
- **Docs:** gap — `docs/guides/gitops-workflows.md` does not mention service-level
  Helm values at all (command/args, rollout vs deployment strategy, etc.). The
  `docs/guides/services.md` page may cover deployment configuration but the omission
  in rollout.yaml was a silent footgun.
- **Suggested doc location:** `docs/guides/gitops-workflows.md` — add a "Service
  Configuration Values" section noting `command`, `args`, and the rollout/deployment
  parity guarantee. Alternatively `docs/guides/services.md`.
- **Diff highlight:**
  ```yaml
  # platform-gitops/helm-charts/base-service/templates/rollout.yaml
  # Lines added at container spec level:
  {{- with .Values.command }}
  command:
    {{- toYaml . | nindent 12 }}
  {{- end }}
  {{- with .Values.args }}
  args:
    {{- toYaml . | nindent 12 }}
  {{- end }}
  ```
