### Commit `10448a0` analysis — gap

- **Date:** 2026-05-02
- **Type:** fix
- **User-visible effect:** OpenClaw agents can be configured with an `active-hours`
  schedule (start/end times + timezone, e.g. `Asia/Shanghai`). Before this fix, the
  heartbeat phase-seeking algorithm was UTC-naive: even when `active-hours` was set,
  heartbeat ticks could fire during the configured quiet-hours window (e.g. during night
  for a `4h` heartbeat). After this fix, the scheduler seeks the next in-window phase
  slot and respects the configured timezone. Users who configure `active-hours` will now
  see their agents go genuinely quiet outside those hours rather than firing spuriously.
- **Docs:** gap — `docs/platform/openclaw.md` exists but the `context/docs-tree.md`
  snapshot does not list any page covering OpenClaw agent scheduling or `active-hours`
  configuration. The feature is user-configurable (per-agent YAML/JSON config with a
  timezone field) but is not mentioned on `docs.mctl.ai`.
- **Suggested doc location:** `docs/platform/openclaw.md` — add an "Agent scheduling
  (active hours)" subsection explaining the `active-hours` configuration key, the
  timezone field, and the expected heartbeat behaviour. Alternatively, a new
  `docs/guides/scheduling.md` page if the scope warrants it.
- **Diff highlight (from PR summary):**
  > "The PR adds active-hours-aware heartbeat phase seeking, wires runner scheduling and
  > config reloads through it. Reproducibility: a `4h` heartbeat with an `Asia/Shanghai`
  > active-hours window — before fix, main arms raw UTC-phase slots and only skips when
  > the timer fires."
- **Files touched:** `heartbeat-active-hours.ts` (+1/-1 core logic),
  `heartbeat-runner.ts` (+62/-3 scheduling wiring),
  `heartbeat-schedule.ts` (+38/-0 phase-seek logic),
  `CHANGELOG.md` (+1/-0 user-visible note).
- **Version status:** in production (mctl-openclaw 2026.5.2-mctl.1, deployed 2026-05-04).
- **Caveat:** The exact config key syntax (e.g. `activeHours`, `active_hours`, or YAML
  equivalent) is not confirmed from commit message alone — the `proposed-content.md` will
  need a `<TODO: confirm config key name with author of 10448a0>` marker.
