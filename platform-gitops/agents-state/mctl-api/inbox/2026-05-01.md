## Top-3 (for spec-writer)

### 1. mcp-rce-server-hardening: MCP Write-Tool Server-Side Hardening Against Systemic RCE (CVE-2025-49596 Family)
**Category:** security
**Impact:** 5
**Effort:** 3
**Rationale:** mctl-api exposes 13 write tools via mark3labs/mcp-go (Streamable HTTP) and Anthropic has explicitly declined to fix the protocol-level weakness; server-side mitigations are the only available defence. The CVE-2025-49596 family (CVSS 9.4) demonstrates that malicious or compromised MCP clients can trigger arbitrary command execution through tool invocations, making hardening of input validation, allow-listing, and per-tool authorization checks directly applicable to mctl-api's threat model. No existing proposal addresses server-side MCP tool hardening; the mcp-go-upgrade-v0.50 proposal covers the library bump but not the defensive mitigations.
**Source:** https://thehackernews.com/2026/04/anthropic-mcp-design-vulnerability.html — CVE-2025-49596, CVSS 9.4

### 2. argocd-token-scope-audit: Audit and Restrict ARGOCD_TOKEN Scope to Mitigate CVE-2025-55190
**Category:** security
**Impact:** 4
**Effort:** 2
**Rationale:** CVE-2025-55190 (CVSS 10.0) allows any holder of a project-scoped ArgoCD token to read all repository credentials (usernames, passwords) from the project details API without needing explicit secrets permission. mctl-api retrieves its ARGOCD_TOKEN from Vault and uses it for application status queries; if that token is project-scoped, it is sufficient to leak every repo credential in the ArgoCD project, representing a direct supply-chain risk. This is an mctl-api configuration risk that must be audited and remediated (token scope review + principle-of-least-privilege enforcement in Vault) independent of the ArgoCD server upgrade.
**Source:** https://github.com/argoproj/argo-cd/security/advisories/GHSA-786q-9hcg-v9ff — CVE-2025-55190, CVSS 10.0

### 3. go-oidc-dep-bump: Bump coreos/go-oidc to v3.18.0 (oauth2 + go-jose freshness)
**Category:** security
**Impact:** 2
**Effort:** 1
**Rationale:** mctl-api uses go-oidc for all three bearer-token verification paths (Dex JWT, GitHub OAuth, OAuth JWT with HMAC-SHA256); keeping its transitive dependencies current — particularly go-jose/v4 (4.1.3 → 4.1.4) and golang.org/x/oauth2 (0.28.0 → 0.36.0) — closes any latent vulnerabilities in those libraries before they are disclosed. This is a low-effort, zero-breaking-change bump that maintains security hygiene on the JWT verification stack without touching application logic. No existing proposal covers this specific dependency update.
**Source:** https://github.com/coreos/go-oidc/releases/latest — v3.18.0 released 2026-04-08
