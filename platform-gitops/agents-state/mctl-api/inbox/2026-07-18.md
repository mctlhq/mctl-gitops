## Top-3 (for spec-writer)

### 1. client-go-oidc-impersonation-bypass: client-go v0.32 OIDC impersonation-header bypass (CVE-2026-23990)
**Category:** security
**Impact:** 5
**Effort:** 3
**Rationale:** CVE-2026-23990 is a privilege-escalation bug where client-go omits impersonation headers when an OIDC provider's claims/CEL expressions come back empty, causing Kubernetes API calls to run with the calling service account's elevated privileges instead of the authenticated user's. mctl-api combines exactly the two ingredients this CVE needs — Dex/GitHub OIDC auth plus client-go for all Kubernetes API calls — and the architecture doc flags "mctl-api itself authorizes tenant scope — a bug here = cross-tenant leak" as a known risk area. This is a materially new angle versus the existing `client-go-version-drift` proposal, which explicitly stated "no direct CVEs affecting v0.32 are known" — that proposal is generic version-drift hygiene, not a fix for this specific vulnerability class.
**Source:** CVE-2026-23990, https://dbugs.ptsecurity.com/vulnerability/PT-2026-3873 (inbox `## Source: cve`)

### 2. argo-workflows-podspecpatch-bypass: argo-workflows podSpecPatch security-setting override (CVE-2026-31892 / CVE-2026-42296)
**Category:** security
**Impact:** 4
**Effort:** 3
**Rationale:** A Workflow submitted against a WorkflowTemplate can override security-relevant pod fields (hostNetwork, serviceAccountName, securityContext) via `podSpecPatch` even under `templateReferencing: Strict`; the first fix (CVE-2026-31892) was incomplete, and CVE-2026-42296 shows the bypass still works through other WorkflowSpec fields. mctl-api's `trigger_workflow` MCP write tool submits directly into this Argo Workflows cluster (architecture.md lists Argo Workflows as a tracked, actively-used integration), so this is a real privilege-escalation surface reachable through our own write tool. No existing proposal in `proposals/` touches argo-workflows at all, so this is a genuinely new, uncovered topic — recommend upgrading the controller to 3.7.14/4.0.5+ and adding server-side validation/rejection of security-relevant fields in `podSpecPatch` payloads submitted via `trigger_workflow` as defense-in-depth.
**Source:** CVE-2026-31892, CVE-2026-42296 (inbox `## Source: cve`)

### 3. admins-cpu-headroom: investigate CPU limit headroom for mctl-api in tenant admins
**Category:** performance
**Impact:** 3
**Effort:** 2
**Rationale:** Today's mctl metrics show tenant `admins` consuming ~75.5% of its 10-core CPU *limit* (7550m/10000m) while only 5/12 pods are scheduled and CPU *requests* usage is just ~37% (740m/2000m) — a wide gap between requested and limit-consumed CPU that points to bursty throttling risk rather than steady baseline load, and is worth investigating (profiling, `GOMAXPROCS`/limit tuning) before it produces a CPU-throttling incident. This is a fresh, first-time signal with no existing proposal covering it, sits squarely in the analyst's "metrics as input for optimisation proposals" remit, and does not touch tenant `labs` (mctl-api does not run there), so it carries no `labs` memory risk.
**Source:** mctl metrics — `Resource usage (tenant admins)` (inbox `## Source: mctl metrics`)
