## Top-3 (for spec-writer)

### 1. pgx-upgrade-v592: Upgrade pgx/v5 to 5.9.2 to fix three CVEs
**Category:** security
**Impact:** 5
**Effort:** 2
**Rationale:** The service is on pgx v5.8, which is affected by three separate vulnerabilities all fixed in v5.9.2: CVE-2025-54236 (SQL injection via placeholder confusion), CVE-2026-33815 (improper array index validation, memory-safety), and CVE-2026-33816 (incorrect comparison, memory-safety). A SQL injection risk in the database driver is a critical threat for a platform API handling tenant data and audit logs, and the fix is a single minor-version bump with no breaking changes expected within pgx v5.
**Source:** https://pkg.go.dev/github.com/jackc/pgx/v5 ; https://dailycve.com/pgx-go-sql-injection-via-placeholder-confusion-cve-2025-54236-low-severity/ ; https://nvd.nist.gov/vuln/detail/CVE-2026-33815 ; https://nvd.nist.gov/vuln/detail/CVE-2026-33816

### 2. mcp-go-cve-waf-bypass: Upgrade mcp-go to 1.3.1 to fix CVE-2026-27896 WAF bypass
**Category:** security
**Impact:** 4
**Effort:** 4
**Rationale:** CVE-2026-27896 (CVSS 7.0, High) allows an attacker to bypass WAF rules by capitalizing JSON-RPC keys, since Go's `encoding/json` is case-insensitive; mcp-go 1.3.1 fixes this by enforcing strict-case parsing. The fix requires a major version jump from 0.31 to 1.3.1, which means all 24 MCP tools must be re-validated through MCP Inspector per ADR-0001 — high effort but warranted given the MCP endpoint is public-facing with write tools. This is a version upgrade of the existing library, not a replacement, so it is consistent with ADR-0001.
**Source:** https://cvereports.com/reports/CVE-2026-27896

### 3. chi-upgrade-v525: Upgrade chi to v5.2.5 to harden against CVE-2025-69725
**Category:** security
**Impact:** 3
**Effort:** 2
**Rationale:** CVE-2025-69725 (CVSS 4.7, Medium) is an open redirect in the `RedirectSlashes` middleware affecting chi >= 5.2.2; the service is currently on 5.2.1 and therefore not yet in the vulnerable range, but upgrading directly to 5.2.5 (which hardens the handler) eliminates the risk of accidentally introducing the vulnerability in any future partial upgrade and brings atomic-type refactors. The upgrade is a minor version bump within the same chi/v5 major, so no breaking API changes are expected.
**Source:** https://github.com/advisories/GHSA-4h76-926q-wxxw ; https://github.com/go-chi/chi/releases/latest
