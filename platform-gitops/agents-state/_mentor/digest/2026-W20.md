### 1. `mctl-openclaw/openclaw-upgrade-2026-5-12` — direct hop to 2026.5.12 closes the "Claw Chain" CVE cluster and finally trims the `labs` core footprint

- **What:** bump every tenant (`labs` → `admins` → `ovk`) from
  **2026.3.14** straight to **v2026.5.12** (released 2026-05-14).
  Closes four "Claw Chain" CVEs disclosed the same week:
  CVE-2026-44112 (CVSS 9.6 — TOCTOU lets OpenShell writes escape the
  sandbox mount root), CVE-2026-44115 (8.8 — incomplete here-document
  allowlist enables shell-expansion bypass), CVE-2026-44118 (7.8 —
  non-owner loopback clients impersonate gateway owner),
  CVE-2026-44113 (7.7 — TOCTOU read counterpart of 44112). Beyond
  the CVE set, v2026.5.12 **externalises the heavy provider cones**
  (WhatsApp/Baileys, Slack, Amazon Bedrock, Anthropic Vertex) out of
  the core process — the expected RSS delta for `labs` is *negative*.
- **Why:** a 9.6 CVSS sandbox escape is a security emergency on its
  own. The simultaneous core-footprint reduction is the second
  consecutive openclaw release that *creates* `labs` RAM headroom
  (after the 2026.5.2 / 2026.5.4 sequence), and it is the upgrade
  that finally retires the running 2026.3.14 baseline — the gap is
  now eight feature releases. The proposal explicitly supersedes
  the W19-tracked `openclaw-upgrade-cve-batch` (which targeted
  2026.5.6) and the entire 2026.4-x ladder W18 already closed in
  principle.
- **Impact / Effort:** **5 / 3.**
- **Conflicts:**
  - **Direction shift.** Supersedes `openclaw-upgrade-cve-batch`
    (W19 anchor) and `upgrade-to-2026-5-2` / `upgrade-to-2026-5-3`
    / `upgrade-to-2026-5-4` from earlier in the openclaw ladder.
    Owner must close all four slugs the moment the 5.12 PR opens,
    citing every CVE ID they enumerate in the canonical PR
    description.
  - `git-plugin-install-allowlist` (W18 Top-5 #4) **still binds**
    — the 5.2-era `git:` plugin scheme is present in 5.12, so the
    allowlist gate must merge before any tenant flips
    `gitPluginInstall.enabled: true` on the new version.
  - The W19 carry-forward `nodejs-heap-cap-labs` proposal is finally
    fundable in evidence terms: the `labs` RSS reading taken during
    the 5.12 soak is the third corroborating measurement — see
    Platform risks #3.
- **Pre-flight:**
  - Pause the s3-sync Argo CronWorkflow per tenant before pod
    restart and resume only after the restore-state readiness probe
    passes (ADR-0002).
  - Record the `labs` RSS baseline *before* pod restart and again
    after the 15-minute settle window; write both numbers into the
    PR description so they are durable beyond review.
  - Verify the existing `device.token.rotate` response-parsing tools
    against the 2026.4.26+ format change *before* the `ovk` step
    (prerequisite gate inherited from the 5.2 design).
  - Confirm `gitPluginInstall.enabled: false` in every tenant's
    Helm values at merge time.
- Link: `mctl-openclaw/proposals/openclaw-upgrade-2026-5-12/`.

### 2. `mctl-api/mcp-go-upgrade-v2` — `mcp-go` v0.54.0 closes a server-wide validation bypass across all 24 MCP tools

- **What:** bump `github.com/mark3labs/mcp-go` from the currently
  pinned v0.31 directly to **v0.54.0** (released 2026-05-13).
  Supersedes the W19 anchor target of v0.52.0 and the older
  `mcp-go-upgrade-v0.50` proposal. The advisory in scope is
  CVE-2026-27896: `mcp-go`'s JSON parser handles field names
  case-insensitively, so requests with `"Method"` / `"method"` /
  `"METHOD"` variants silently bypass server-side validation **on
  every one of the 24 MCP tools** mctl-api exposes. The bump also
  picks up six months of panic-safety fixes in handler/goroutine
  cleanup paths and adds OpenTelemetry tracing hooks per tool
  invocation.
- **Why:** the case-variant bypass is a server-wide security defect
  — every authorised-tool gate, every tenant-scoping check, every
  argument validator is in scope. Pair this with the running
  MCP-hardening cluster (the W19 carry-forward triple
  `mcp-rce-server-hardening` / `mcp-command-injection-hardening` /
  `mcp-input-validation` and the CSRF-origin pair) and v0.54.0
  becomes the **floor version** under all of them — the hardening
  primitives are pointless if the library that dispatches them is
  itself case-bypassable. OpenTelemetry hooks also unblock per-tool
  observability that has been a documented blind spot since W18.
- **Impact / Effort:** **5 / 3.**
- **Conflicts:**
  - **Floor under the W19 MCP-hardening release-train.** Land the
    library bump first; merging any of the six hardening proposals
    on top of v0.31 leaves the bypass underneath them. Pair this PR
    with the ADR called out in W19 Platform risks #3 (shared
    schema-registry, `MCP_ALLOWED_*` env-var family, shared audit
    table, shared rate-limit thresholds) so the two endpoints
    (`mctl-api` and `mctl-agent`) cannot drift.
  - **Supersedes** `mctl-api/mcp-go-upgrade-v0.50`,
    `mctl-api/mcp-go-v051-upgrade`, `mctl-api/mcp-go-v052-upgrade`,
    `mctl-api/mcp-go-upgrade`, `mctl-api/mcp-go-oauth-upgrade`,
    `mctl-api/mcp-go-cve-waf-bypass` (six dup-cluster slugs in
    `mctl-api/proposals/`). Close all six when the v0.54.0 PR opens.
  - ADR-0001 requires re-validating all 24 tools through MCP
    Inspector before deployment — this is the dominant effort cost
    and is non-negotiable. Schedule it before the W20 hardening PRs
    queue behind the library bump.
- **Pre-flight:**
  - Run MCP Inspector against the v0.54.0 server and diff schema
    output against the v0.31 baseline; capture the diff in the PR.
  - Confirm `mcp-go` v0.54.0 still serves Streamable HTTP (POST + GET)
    so Claude.ai connectors do not need reconfiguration (ADR-0001).
  - Wire the new OpenTelemetry hooks into the existing collector
    only if the `OTEL_*` env-vars are already populated for
    `mctl-api`; otherwise stage the wiring as a follow-up.
- Link: `mctl-api/proposals/mcp-go-upgrade-v2/`.

### 3. `mctl-gitops/argocd-v3-4-2-stability-patch` — patches the W19-planned v3.4.0 GA before the platform inherits its crasher

- **What:** change the pinned ArgoCD image tag in the bootstrap
  manifest from **v3.4.0** (the W19-planned GA target) to **v3.4.2**
  (released 2026-05-12). v3.4.2 fixes a permission-validator panic
  that can crash the ArgoCD server process under specific RBAC
  evaluation conditions — i.e. crash the entire GitOps control
  plane until the pod restarts. The bundled Go runtime in v3.4.2
  also closes the Go-level CVEs left open in v3.4.0. No CRD changes,
  no ApplicationSet schema changes between v3.4.0 and v3.4.2.
- **Why:** the W19 plan already commits the platform to the v3.4
  line. Promoting v3.4.0 as planned would intentionally take a
  known crash bug into production for the sake of release timing.
  v3.4.2 is the **same release window, two patch numbers ahead, no
  schema change** — the cost/benefit is unambiguous. The crasher
  hits *every* tenant the moment any RBAC policy is evaluated in
  the wrong way, so the blast radius is platform-wide.
- **Impact / Effort:** **4 / 1.**
- **Conflicts:**
  - **Replaces, does not add to, the v3.4.0 pin** in
    `argocd-v3-4-upgrade-plan-v2`. The `labs` pre-stage and 24-hour
    soak from that plan still applies — just at v3.4.2 instead of
    v3.4.0.
  - **Bundles** with `argocd-v3-4-1-cluster-gen-migration`
    (cluster-generator label-format migration) and the W19-carried
    `argocd-cluster-label-drift` in `mctl-agent`. v3.4.2 is a strict
    superset of v3.4.1, so collapse `labs` → `admins` into a single
    soak rather than the two W19 had pencilled in.
  - Distinct from the older `argocd-informer-cache-patch` (v3.3.x
    line, holding pattern only if v3.4 promotion slips) and from
    `argocd-token-scope-audit` (independent — the client-token
    rotation work from W19).
- **Pre-flight:**
  - Walk every `platform-gitops/apps/*.yaml` for ApplicationSet
    fields deprecated between v3.3 and v3.4; surface the diff
    *before* the version bump PR opens (carries from W19 — still
    not done at the time this digest was assembled).
  - Confirm rollback path: revert the pin commit, force a sync,
    observe the v3.3.x pod return without ApplicationSet drift.
- Link: `mctl-gitops/proposals/argocd-v3-4-2-stability-patch/`.

### 4. `mctl-gitops/eso-confused-deputy-patch` — first concrete remediation of the cross-tenant privilege escalation surface W19 flagged

- **What:** upgrade External Secrets Operator from the chart-2.4.x
  line (still in the vulnerable range 0.1.0–2.4.0) to **ESO v2.5.0**
  via the chart bump. Closes **CVE-2026-42876**: a tenant user who
  holds only `create` permissions on `ExternalSecret` can craft a
  manifest that causes the ESO controller — which acts with elevated
  privileges on behalf of the requesting user — to mint a Kubernetes
  Secret containing a long-lived Service Account token outside the
  caller's RBAC scope. Classic confused-deputy. No CRD schema
  changes; the upgrade is a values-file pin bump with a Helm sync.
- **Why:** W19 Platform risks #5 listed cross-tenant privilege
  boundaries as "no longer overdue — structurally absent." This is
  the first proposal of W20 that actually *closes* one of those
  layers rather than adding another. It is also the only proposal
  this week that supersedes a W19 carry-forward target version
  (`helm-chart-2.4.1`) for a cleaner reason: the W19 target was
  still inside the vulnerable range. Land this and the W19
  cross-tenant block shrinks from seven open layers to six.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - **Supersedes** `mctl-gitops/eso-cross-namespace-bypass-patch`
    (W18 / W19 anchor) and folds in
    `mctl-gitops/eso-cross-namespace-secret-access`,
    `mctl-gitops/eso-cve-patch`, `mctl-gitops/eso-dns-exfil-patch`
    (four-slug dup-cluster). Close the other three when the v2.5.0
    PR opens.
  - **Bundle** into the cluster-security ApplicationSet sync window
    alongside the ArgoCD v3.4.2 pin (Top-5 #3), Prometheus v3.11.3
    (W18 Top-5 #5 carry-forward), Vault v2.0.0, Helm v4.1.4, and
    cert-manager v1.20.2 — one ApplicationSet sync per release
    window, one rollback plan.
  - Distinct from Vault PKI work and from `argocd-token-scope-audit`
    — all three are complementary layers under the same
    cross-tenant umbrella.
- **Pre-flight:**
  - Replay the CVE-2026-42876 PoC manifest against `labs` after the
    sync and confirm the resulting `Secret` does not embed the SA
    token. Capture the test in the PR description.
  - Verify `vault-backend` ClusterSecretStore reconciliation
    survives the controller restart without CRD migration noise.
  - Stage the new `/healthz` liveness probe in the alerting layer as
    a follow-up — not a blocker for this PR.
- Link: `mctl-gitops/proposals/eso-confused-deputy-patch/`.

### 5. Go 1.26.3 cross-repo (`mctl-agent/go-toolchain-1263-patch` + `mctl-api/go-runtime-upgrade-v2`) — five new CVEs since 1.26.2, including an HTTP/2 infinite-loop DoS on every chi + MCP-streaming entrypoint

- **What:** bump the Go toolchain from 1.24 / 1.26.2 to **Go 1.26.3**
  on both `mctl-agent` and `mctl-api`. The 2026-05-07 patch cycle
  added five CVEs that neither service is otherwise protected
  against: **CVE-2026-33814** (HTTP/2 transport enters an infinite
  loop when a peer sends `SETTINGS_MAX_FRAME_SIZE=0` — directly
  exploitable against `mctl-agent`'s chi webhook server and against
  `mctl-api`'s MCP streaming transport), **CVE-2026-39826** +
  **CVE-2026-39823** (two html/template XSS bypasses, the second a
  bypass of the 1.26.1 fix for CVE-2026-27142), **CVE-2026-42499**
  + **CVE-2026-39820** (quadratic `net/mail` DoS via crafted
  email-address headers), **CVE-2026-42501** (malicious module
  proxies can serve altered toolchains by bypassing checksum
  validation in `cmd/go` — a CI-tampering vector). `mctl-api`'s
  proposal also calls out CVE-2026-27140 (CVSS 8.8) at build time
  via cgo/SWIG, and the Green Tea GC improvements that documentably
  reduce p99 latency on MCP streaming by 10–40%.
- **Why:** the W19 anchor "Go 1.26.2 cross-repo" was a security
  upgrade; closing it on 1.26.2 instead of 1.26.3 means the platform
  rolls out a Go-runtime hop and ships with five known CVEs already
  open. Treating it as a single coordinated bump (both repos in the
  same release window) costs one extra `toolchain` directive change
  per repo and is the only way the two services share a stdlib
  surface for the rest of W20.
- **Impact / Effort:** **4 / 2.** Combined across both repos.
- **Conflicts:**
  - **Supersedes** the W19 "Go 1.26.2 cross-repo" anchor. Close
    `mctl-agent/go-upgrade-1-26`, `mctl-agent/go-toolchain-upgrade`,
    `mctl-agent/go-runtime-upgrade`, `mctl-agent/go-runtime-upgrade-v2`,
    `mctl-agent/go-runtime-patch`, plus the `mctl-api` parallels
    (`go-runtime-upgrade`, `go-runtime-cve-upgrade`,
    `go-runtime-cve-dos`, `go-toolchain-ace-cve-27140`,
    `go-upgrade-1262`, `go-upgrade-stdlib-cves`,
    `go-upgrade-stdlib-cves-v2`, `go-upgrade`) once the canonical
    1.26.3 PR is in flight.
  - **Order with mcp-go v0.54.0 (Top-5 #2):** in `mctl-api`, bump Go
    first, then mcp-go, then the MCP-hardening triple. Reversing
    that order means the hardening PRs ship against a stdlib that
    still has the HTTP/2 infinite-loop bug under them.
  - The two repos can ship independently — the only shared dependency
    is the runtime base image, which both services pin separately.
- **Pre-flight:**
  - Run `govulncheck` against the bumped binaries; capture the diff
    against the 1.26.2 baseline in the PR.
  - Smoke-test the chi webhook server against a peer that sends
    `SETTINGS_MAX_FRAME_SIZE=0` — assert clean error rather than
    hang.
  - Confirm CI's module-proxy configuration cannot bypass checksum
    validation (CVE-2026-42501 mitigation is largely behavioural —
    the proxy URL must not be a third-party mirror).
- Links: `mctl-agent/proposals/go-toolchain-1263-patch/`,
  `mctl-api/proposals/go-runtime-upgrade-v2/`.

   the W19 Top-5 rollouts (openclaw 2026.5.4, Backstage v1.50.4
   twelve-bundle, ArgoCD v3.4 GA, mcp-go v0.52.0, Go 1.26.2,
   Loki v3.7.1, Helm v4.1.4, cert-manager v1.20.2, Prometheus
