## Топ-5 предложений к ревью

### 1. `mctl-web/wrangler-cve-0933` — CVE-2026-0933 в wrangler (CVSS 9.9)
- **Что:** поднять wrangler до v4.59.1+ в `deploy.yml` и `package.json`; добавить
  `wrangler --version` и `npm audit --audit-level=critical` в CI.
- **Почему:** OS command injection в `wrangler pages deploy` через `--commit-hash`.
  Любой контроль над commit metadata (имя ветки/тега) → произвольное выполнение в
  GitHub Actions runner с доступом к Cloudflare API Token, GITHUB_CLIENT_SECRET,
  BACKSTAGE_LANDING_TOKEN.
- **Impact / Effort:** **5 / 1**
- **Конфликты:** нет.
- **Готовность к merge:** высокая. Точечный bump патч-версии в рамках мажора v4,
  обратно совместимо. Дополнительно стоит явно решить: ротировать ли затронутые
  Worker secrets (proposal явно держит это out-of-scope, но платформенно — это
  отдельный вопрос).
- Ссылка: `mctl-web/proposals/wrangler-cve-0933/`

### 2. `mctl-openclaw/upgrade-to-2026-4-25` — закрытие 5 CVE openclaw
- **Что:** обновить тег образа openclaw `2026.3.14 → 2026.4.25` по маршруту
  labs → admins → ovk (ADR 0001). Закрывает CVE-2026-41349 (CVSS 8.8, agentic
  consent bypass через `config.patch`), 41361 (SSRF IPv6), 41359 (Telegram privesc),
  41353 (allowProfiles bypass), 41348 (Discord auth bypass).
- **Почему:** в agentic-среде openclaw consent bypass = неконтролируемое действие
  без апрува пользователя; остальные 4 CVE — рядом. Все пять закрываются одним
  upstream-релизом, двойной rollout не оправдан.
- **Impact / Effort:** **5 / 3**
- **Конфликты:** мягкая координация с `clawhub-skills-allowlist` (см. ниже) — оба
  меняют один и тот же Helm values слой каждого тенанта. Если идут параллельно —
  делать через отдельные PR, иначе rebase-боль.
- **Pre-flight (обязательно):** RAM-check тестовым подом 2026.4.25 c labs-лимитами
  до коммита в labs-манифест. Если delta > 50MB — блок и тикет на поднятие лимита,
  до admins/ovk не идти. **Это критичный момент — `labs` исторически близок к
  лимиту памяти, и я не могу подтвердить это через MCP в этой сессии.**
- Ссылка: `mctl-openclaw/proposals/upgrade-to-2026-4-25/`

### 3. `mctl-openclaw/npm-supply-chain-audit` — пиннинг Baileys/discord.js
- **Что:** одноразовый аудит `package-lock.json` + постоянный CI-шаг, который
  блокирует запрещённые имена (`baileys` без namespace, `discord.js-user`) и
  не-`registry.npmjs.org` resolved URLs для monitored пакетов.
- **Почему:** активные poisoned форки — `discord.js-user` (GHSA-69r6-7h4f-9p7q,
  CVSS 9.8) сливает Discord-токены, поддельный Baileys перехватывает WhatsApp
  auth/сообщения/медиа. Сейчас нет автоматической защиты от случайной подмены при
  будущем bump'е зависимостей.
- **Impact / Effort:** **4 / 1**
- **Конфликты:** нет.
- **Готовность к merge:** очень высокая. Effort минимальный (один bash-скрипт),
  риска регрессии нет. Если одноразовый аудит обнаружит уже установленный
  poisoned пакет — это автоматически становится **инцидентом P0 с ротацией
  WhatsApp/Discord-токенов всех затронутых тенантов**, дайджест об этом
  предупреждает заранее.
- Ссылка: `mctl-openclaw/proposals/npm-supply-chain-audit/`

### 4. `mctl-openclaw/clawhub-skills-allowlist` — fail-closed allowlist для Layer 3 skills
- **Что:** `skills.remoteAllowlist` в Helm values каждого тенанта + enforcement в
  openclaw config (или nginx middleware, если upstream не поддерживает нативно)
  + CI-проверка на новый skill-источник в diff. Default = deny-all.
- **Почему:** активная кампания ClawHavoc, 341+ malicious skills в официальном
  ClawHub marketplace. CVE пока не назначен. Layer 3 (remote/HTTP) skills
  регистрируются через REST API без ограничений по источнику. Если хоть один
  тенант когда-то ставил skills с ClawHub — вектор уже открыт.
- **Impact / Effort:** **4 / 2**
- **Конфликты:** мягкая координация с `upgrade-to-2026-4-25` (общий Helm values
  слой). По смыслу — наоборот, стоит **сначала** включить allowlist (deny-all
  не сломает то, что уже работает, но запретит новые регистрации), а потом
  обновлять образ.
- **Pre-flight:** инвентаризация уже зарегистрированных Layer 3 skills на каждом
  тенанте — иначе fail-closed reset поломает легитимные skills. Начинать раскат
  с `admins` (минимальный blast radius), затем `labs`, затем `ovk`.
- Ссылка: `mctl-openclaw/proposals/clawhub-skills-allowlist/`

### 5. `mctl-web/nuxt-upgrade-4-4-2` — Nuxt 4.3.1 → 4.4.2 + vue-router v5
- **Что:** минорный bump Nuxt в рамках мажора 4 + переход с vue-router v4
  (EOL) на v5. 28× ускорение роутинга, typed layout props, `useAnnouncer`.
- **Почему:** vue-router v4 EOL = нет security patches. Технический долг растёт.
  Сервис маленький (3 страницы), riska bumpа умеренные.
- **Impact / Effort:** **3 / 3**
- **Конфликты:** **зависит от `vue-patch-3-5-33`** (см. «Отложено»). Не блокирует,
  но порядок важен — патч Vue делается отдельным PR _до_ этого, чтобы изолировать
  переменные при отладке. Это явно прописано в design.md обоих предложений.
- **Pre-flight:** аудит прямых импортов `from 'vue-router'` в коде; smoke-проверка
  prerender для `/`, `/docs`, `/privacy`; smoke-проверка vee-validate-формы.
- Ссылка: `mctl-web/proposals/nuxt-upgrade-4-4-2/`

## Тематические группы

- **Supply-chain hardening** (Web и openclaw двигаются параллельно):
  `wrangler-cve-0933` (CI-инструмент) + `npm-supply-chain-audit` (runtime deps) +
  `clawhub-skills-allowlist` (extension marketplace). Это три разных слоя одного
  тренда — стоит обсудить общий «supply-chain checklist» для всех будущих
  сервисов: pin to official registry, CI guard на запрещённые имена, fail-closed
  allowlist для любых external-сорсов.
- **openclaw security baseline** (`upgrade-to-2026-4-25` + `clawhub-skills-allowlist`):
  оба идут в один и тот же Helm values слой. Координировать по PR, не делать
  одновременно в одном тенанте.
- **mctl-web frontend stack** (`vue-patch-3-5-33` → `nuxt-upgrade-4-4-2`):
  строго последовательно, отдельными PR.

## Платформенные риски

- **`labs` и лимит RAM.** CLAUDE.md обоих сервисов прямо называет `labs` узким
  местом. `upgrade-to-2026-4-25` это явно учитывает (pre-flight RAM check, блок
  при delta > 50MB). Но без `mcp__mctl__*` я не могу подтвердить _текущий_
  headroom — рекомендую перед началом рассмотрения proposal #2 один раз
  получить актуальные RSS/limit из MCP вручную и приложить к PR-описанию как
  baseline. Иначе «delta > 50MB» считать не от чего.
- **Три параллельных supply-chain вектора.** На неделе зафиксированы атаки
  одновременно на CI-инструмент (wrangler), npm-зависимости (Baileys/discord.js)
  и marketplace (ClawHub). Это не совпадение, а тенденция — стоит
  институализировать allowlist-by-default как платформенный инвариант, а не
  делать его per-сервис каждый раз.
- **Нет MCP-видимости в самом дайджесте.** Если ментор регулярно не может
  свериться с реальным состоянием (incidents/canary/metrics), смысл дайджеста
  падает — он становится литературным обзором proposals. Стоит зафиксировать
  как платформенный todo: убедиться, что `mcp__mctl__*` доступен в окружении
  ментора (в скилл `_mentor/CLAUDE.md` он явно упомянут как инструмент работы).
- **Out-of-scope из proposal #1, который реально in-scope для платформы.**
  Wrangler-CVE proposal оставляет ротацию Worker secrets out-of-scope как
  «отдельное мероприятие безопасности». Поскольку CVSS=9.9 и уязвимость могла
  эксплуатироваться до патча, решение «ротировать или нет» это _платформенный_
  вопрос, а не вопрос автора proposal. Поднять явно.
- **ADR не обновлён под новые риски.** ADR 0001 закрепляет rollout-маршрут
  labs → admins → ovk для openclaw. Для `clawhub-skills-allowlist` маршрут
  логично инвертировать (admins первыми — минимальный blast radius). Решить
  явно: делаем new ADR или временное отступление с пометкой в proposal.
