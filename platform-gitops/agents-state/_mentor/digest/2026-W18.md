### 1. `mctl-web/wrangler-upgrade-security` тАФ CVSS 9.9 in CI/CD + workerd UaF in one bump

- **What:** pin wrangler to **4.87.0** in `deploy.yml` (or via `devDependencies`
  with `npm ci`). 4.87.0 closes CVE-2026-0933 (`--commit-hash` OS command
  injection) **and** ships workerd `v1.20260430.1` with the use-after-free fix.
- **Why:** the CVE turns any controllable git context (branch/tag name on a
  fork PR) into arbitrary command execution inside a GitHub Actions runner
  that holds the Cloudflare API Token, `GITHUB_CLIENT_SECRET`, and
  `BACKSTAGE_LANDING_TOKEN`. The workerd UaF affects request-time memory
  safety for production traffic. Pinning to 4.87.0 collapses what was
  previously a two-PR plan (`wrangler-cve-0933` тЖТ 4.59.1, then
  `wrangler-full-upgrade` тЖТ 4.86.0) into one supportable change.
- **Impact / Effort:** **5 / 1**.
- **Conflicts:** **supersedes** `mctl-web/wrangler-cve-0933` and
  `mctl-web/wrangler-full-upgrade` тАФ close both as superseded; do not ship
  the older minimum-fix version. Worker-secret rotation remains out of
  scope of the patch and **must** be opened as a separate platform ticket
  the moment this merges (CVE 9.9 + CI runner exposure).
- **Pre-flight:** confirm the `deploy.yml` flag set is unchanged in the 4.59
  тЖТ 4.87 changelog; run a staging `wrangler pages deploy`; then chain a
  `wrangler.toml` `compatibility_date` review (orthogonal to the bump).
- Link: `mctl-web/proposals/wrangler-upgrade-security/`
  (close `mctl-web/proposals/wrangler-cve-0933/` and
  `mctl-web/proposals/wrangler-full-upgrade/` as superseded).

## Platform risks

1. **Duplicate / superseded proposals are now the norm, not an exception.**
   This week the count climbed past five duplicate clusters:
   - **pgx**: `pgx-cve-sql-injection` vs `pgx-sql-injection-fix` vs
     `pgx-sqli-cve-2025-54236` (three proposals, one CVE).
   - **chi**: `chi-security-patch` vs `chi-open-redirect-ghsa-mqqf`
     (same target, different threat model framing).
   - **mcp-go**: `mcp-go-oauth-upgrade` (v0.49) vs `mcp-go-upgrade-v0.50`.
   - **Go upgrade**: `go-upgrade` vs `go-upgrade-stdlib-cves` vs
     `go-upgrade-stdlib-cves-v2` (three proposals, one toolchain bump).
   - **openclaw image**: `upgrade-to-2026-4-8` vs `upgrade-to-2026-4-25`
     vs `upgrade-to-2026-4-26` vs `upgrade-to-2026-4-27` (four proposals,
     one image tag тАФ 4.27 covers everything below).
   - **Argo Workflows**: `mctl-gitops/argo-workflows-cve-patch` (v4.0.5)
     vs `argo-workflows-cve-patch-v2` (v3.7.14) vs
     `mctl-agent/argo-workflows-cve-patch` (cross-service duplicate).
   - **wrangler**: `wrangler-cve-0933` vs `wrangler-full-upgrade`
     vs `wrangler-upgrade-security` (three proposals, one pin).
   - **nuxt**: `nuxt-upgrade-4-4` vs `nuxt-upgrade-4-4-2` vs
     `nuxt-upgrade-4-4-4` (three proposals, one Nuxt minor).
   - **openclaw outbound docs**: `openclaw-outbound-sanitization` vs
     `openclaw-outbound-security` (two proposals, one doc subsection).

   Without a preflight dedup step the reviewer reads ~22 proposals for
   ~9 fixes. **Recommendation:** add a `supersedes:` field to the
   proposal template and require services to close superseded proposals
   before they submit.

2. **Cross-service duplicate ownership of shared components.** Argo
   Workflows arrived independently from `mctl-agent` and `mctl-gitops`.
   The same pattern will recur on ESO, Helm, Backstage, ArgoCD, and any
   other cluster component. **Recommendation:** a platform ADR with an
   explicit owner registry for shared components, plus a
   `shared-component-owner:` field in the proposal template. Without it,
   this duplication recurs every week.

3. **`labs` RAM is a shared gate with no assigned measurer.** At least
   six active proposals reference labs OOM risk: `upgrade-to-2026-4-27`,
   `nodejs-runtime-upgrade`, `egress-network-policy`, `eso-cve-patch`,
   `vault-ssrf-acme-patch` (Path B), and the agent-side
   `go-upgrade-1-26`. None is designated as the **first measurer**.
   **Resolution:** Top-5 #2 (openclaw 2026.4.27) takes the first reading
   and publishes the baseline (e.g. in `_mentor/` or in mctl-gitops
   state); every later proposal compares against it. Without that
   anchor, "no more than +50 MB" is unanchored.

4. **Supply-chain risk is being addressed three times in three services.**
   This week's wave hits CI tooling (wrangler), npm transitive deps
   (Baileys / `discord.js-user` / `lotusbail` тАФ `baileys-registry-lockdown`
   is the new narrow control), the marketplace (ClawHub, with 341+
   malicious skills reported upstream), and axios (WAVESHAPER.V2). Three
   to five independent CI guards across repos will drift.
   **Recommendation:** a single shared CI step in `mctl-gitops`
   (allowlist `resolved` URLs, deny-list names, fail-closed default)
   referenced from every service's Dockerfile/CI; the new
   `baileys-registry-lockdown` `.npmrc` scope pin is the first concrete
   primitive that should be promoted upward.

5. **Backstage scaffolder pod is a credential concentration point.**
   Top-5 #3 shows that one pod mounts the Vault token, Postgres DSN, and
   GitHub App credentials, and that any path-traversal or env-leak flaw
   lifts them all at once. Architectural, not patchable. **Backlog ADR
   (not this week):** least-privilege per scaffolder action, or split
   the scaffolder pod from co-resident secrets.

6. **CVE clustering signals release-train coordination.** A single week
   carries patches for Argo Workflows, ArgoCD, Helm, ESO, Backstage,
   openclaw (├Ч4 candidates), pgx, chi, mcp-go, Go toolchain, Node.js,
   wrangler, sqlite, Vault, Nuxt, vue-router, Vue, sass. Rolled
   individually, `admins` will see 6+ sync events in 7 days.
   **Recommendation:** `mctl-gitops` packages a "W18 cluster security
   release" тАФ one ApplicationSet sync, defined window, single rollback
   plan, bundling Argo Workflows + ArgoCD + Helm + ESO + Backstage. Cuts
   the blast-window and consolidates canary observation.

7. **MCP visibility is missing.** Without `mcp__mctl__*` tools the
   mentor cannot confirm (a) actual production versions, (b) currently
   open incidents, (c) labs RAM headroom, (d) deduplication against
   patches that already shipped. This digest is therefore a sanity-check
   of proposal text, not a reconciliation against reality. Restoring
   MCP visibility is a prerequisite for the mentor cycle to be
   meaningful; until then, every "pre-flight" line above is the service
   owner's responsibility.

8. **Proposals tree read-path is intermittent.** `ls` of per-service
   `proposals/` is still blocked because the directories resolve to
   gitops state-tree symlinks the sandbox refuses to traverse. `Glob`
   and `Read` against the file paths under them work, so this run
   was able to enumerate. The orchestrator should still mirror the
   gitops state into the mentor's working tree (or resolve symlinks at
   mount) so a future run does not silently degrade to "no items"
   whenever the indirection breaks.

9. **MCP write-tool surface is a single layer thick.** Top-5 #5's
   companion `mcp-go-upgrade-v0.50` proposes the first
   framework-level argument validation hook ever applied to the 13 MCP
   write tools. Until that lands, every write tool relies on
   per-handler validation written in 2025. CVE-2026-27896 (JSON-RPC
   field-name smuggling, confirmed in the official `go-sdk`) is an open
   question against `mark3labs/mcp-go` тАФ the upgrade PR's Step 3 must
   answer it before merge.
