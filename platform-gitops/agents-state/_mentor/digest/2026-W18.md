# Mentor digest — 2026-W18

_Дата сборки: 2026-04-27._

**Активные сервисы платформы (7):** `mctl-web`, `mctl-openclaw` (2026.3.14, все
три тенанта), `mctl-docs`, `mctl-api`, `mctl-portal`, `mctl-agent`, `mctl-gitops`.

Это первый полноценный дайджест в `_mentor/digest/`. Ранее по этому же пути
существовал частичный черновик, покрывавший только `mctl-web` и `mctl-openclaw`
и ошибочно утверждавший, что остальные 5 сервисов «промолчали» — фактически в
их `proposals/` лежит 13 свежих предложений. Текущий файл заменяет тот черновик
и охватывает все 19 proposals + addendum к openclaw.

> **Замечание о верификации.** В этой сессии тулзы `mcp__mctl__*` недоступны
> (нет в deferred-tool регистре, ToolSearch их не находит). Сверка с _реальным_
> состоянием платформы (живые версии, открытые инциденты, точные метрики RAM в
> `labs`) выполнена косвенно — по `context/` каждого сервиса, по
> `cve-addendum-2026-04-27.md` openclaw и по cross-references из самих proposals.
> Каждое требование «pre-flight» в этом дайджесте — это шаг, который я не смог
> сделать сам и который должен сделать владелец сервиса перед merge.

## Топ-5 предложений к ревью

Сортировка: impact / effort × срочность security-сигнала × локализация blast
radius.

### 1. `mctl-web/wrangler-cve-0933` — CVSS 9.9 в CI/CD
- **Что:** поднять wrangler до v4.59.1+ в `deploy.yml` и `package.json`,
  добавить `wrangler --version` и `npm audit --audit-level=critical` в CI.
- **Почему:** OS command injection в `wrangler pages deploy` через
  `--commit-hash`. Любой контроль над commit metadata (имя ветки/тега) →
  произвольное выполнение в GitHub Actions runner с доступом к Cloudflare API
  Token, GITHUB_CLIENT_SECRET, BACKSTAGE_LANDING_TOKEN.
- **Impact / Effort:** **5 / 1**.
- **Конфликты:** нет.
- **Платформенный апдейт к scope:** автор оставил ротацию Worker secrets
  out-of-scope. При CVSS 9.9 это не его решение, а платформенное —
  поднять отдельным тикетом сразу после merge.
- Ссылка: `mctl-web/proposals/wrangler-cve-0933/`.

### 2. `mctl-openclaw/upgrade-to-2026-4-25` — закрытие 12+ CVE одним апгрейдом
- **Что:** обновить тег образа openclaw 2026.3.14 → 2026.4.25 по маршруту
  `labs → admins → ovk` (ADR 0001).
- **Почему:** исходный proposal закрывает 5 CVE (CVE-2026-414xx, в т.ч. 41349
  agentic consent bypass, CVSS 8.8). Аддендум от 2026-04-27 добавляет ещё 4
  critical/high и 9 medium из апрельского batch (CVE-2026-356xx), включая
  CVE-2026-35639 (privilege escalation, 8.7) и CVE-2026-35641 (ACE через
  `.npmrc` локального plugin/hook, 8.4). Все 12+ закрываются одним релизом.
- **Impact / Effort:** **5 / 3**.
- **Конфликты:** общий Helm values слой с `clawhub-skills-allowlist` —
  обязательно отдельные PR. CVE-2026-35649 («settings allowlist bypass при
  пустом allowlist») пересекается с allowlist-предложением: ревьюер должен
  убедиться, что после апгрейда allowlist не оказался пустым (deny-all
  семантика обязательна).
- **Pre-flight (обязательно):** RAM-замер тестового пода 2026.4.25 c
  labs-лимитами до коммита labs-манифеста. Если delta > 50MB — стоп, тикет
  на лимит, до admins/ovk не идти. Этот замер становится **общим baseline**
  для остальных openclaw-предложений недели (см. «Платформенные риски»).
- Ссылка: `mctl-openclaw/proposals/upgrade-to-2026-4-25/` +
  `cve-addendum-2026-04-27.md`.

### 3. `mctl-portal` — Backstage scaffolder bundle (3 CVE → 1 деплой)
- **Что:** свести три proposals (`scaffolder-path-traversal`,
  `scaffolder-secret-leak`, `integration-scm-credentials`) в один апгрейд
  Backstage до **v1.50.3** + `plugin-scaffolder-backend` ≥ 3.1.1 +
  `@backstage/backend-defaults` ≥ 0.12.2 + `@backstage/integration` ≥ 1.20.1.
- **Почему:** все три CVE атакуют один и тот же pod, в котором ExternalSecret
  монтирует Vault token, Postgres DSN, GitHub App credentials.
  - CVE-2026-24046 (path traversal в `debug:log`/`fs:delete`/archive
    extraction) — чтение/запись/удаление файлов pod'а.
  - CVE-2026-32237 (env-leak через dry-run endpoint) — те же секреты в JSON
    ответе аутентифицированного пользователя.
  - CVE-2026-29185 (encoded path traversal в SCM URL) — утечка GitHub App
    token через catalog-import / scaffolder git actions / github-actions
    plugin.
  Авторы первых двух proposals сами называют общий патч; третий уже включает
  первые два.
- **Impact / Effort:** **5 / 3**.
- **Конфликты:** между собой не конфликтуют, наоборот — **должны быть слиты
  в один PR**, иначе три отдельных production-деплоя на тенант `admins` без
  выигрыша. Тенант `labs` не затронут.
- **Pre-flight:** ротация GitHub App installation token и Vault token в
  `admins` сразу после merge — оба могли утечь до патча, и dry-run endpoint
  доступен авторизованным пользователям.
- Ссылки:
  - `mctl-portal/proposals/scaffolder-path-traversal/`
  - `mctl-portal/proposals/scaffolder-secret-leak/`
  - `mctl-portal/proposals/integration-scm-credentials/`

### 4. `mctl-gitops/argo-workflows-cve-patch` — Argo Workflows → v4.0.5 (с разрешением owner-конфликта)
- **Что:** обновить Argo Workflows controller до v4.0.5 (или v3.7.14, если
  кластер ещё на v3-ветке).
- **Почему:** три CVE одним релизом.
  - CVE-2026-40886 (CVSS 7.7) — controller crash loop через malformed
    annotation `pod-gc-strategy`. Обрушивает все Workflow в кластере, в
    частности `cwft-rotate-github-token` каждые 30 минут — это рушит
    GitHub-токен mctl-agent и его self-healing PR.
  - CVE-2026-28229 — утечка embedded secrets из WorkflowTemplate по Bearer
    token.
  - CVE-2026-31892 — bypass Strict mode WorkflowTemplate через podSpecPatch.
- **Impact / Effort:** **5 / 2**.
- **Owner-конфликт:** этот же CVE независимо подал и `mctl-agent`. Argo
  Workflows controller — кластерный компонент в `admins`, владелец —
  **`mctl-gitops`**. `mctl-agent` должен закрыть свой
  `proposals/argo-workflows-cve-patch/` как duplicate и оставить только
  consumer-side smoke-test: `cwft-rotate-github-token` отрабатывает после
  апгрейда без модификации Workflow definitions.
- **Конфликты с другими предложениями:** нет.
- Ссылки:
  - `mctl-gitops/proposals/argo-workflows-cve-patch/` (owner)
  - `mctl-agent/proposals/argo-workflows-cve-patch/` (duplicate, закрыть)

### 5. `mctl-api/pgx-cve-sql-injection` — SQL injection на публичном endpoint
- **Что:** обновить `github.com/jackc/pgx/v5` 5.8.0 → 5.9.2.
- **Почему:** CVE-2025-54236 (GHSA-j88v-2chj-qfwx) — placeholder confusion в
  simple query protocol: `$N` внутри dollar-quoted string interpretируется как
  bind parameter. mctl-api держит identity и audit log за этим драйвером;
  endpoint публичный (`https://api.mctl.ai`). Фикс — patch-bump без breaking
  changes в публичном API.
- **Impact / Effort:** **5 / 1**.
- **Конфликты:** должен идти **отдельно** от `mctl-api/go-upgrade` — иначе
  security-фикс становится заложником более крупного toolchain bump'а.
- **Pre-flight:** `govulncheck` после bump'а; integration tests против
  реального Postgres.
- Ссылка: `mctl-api/proposals/pgx-cve-sql-injection/`.

## Платформенные риски

1. **Дубликат owners на общих компонентах.** Argo Workflows-патч приехал
   независимо от `mctl-agent` и `mctl-gitops`. Тот же сценарий повторится с
   ESO, Helm, Backstage, ArgoCD, любым кластерным компонентом. **Предложение:**
   ввести в платформенный ADR явный реестр owner'ов общих компонентов и
   маркировать в proposal-шаблоне поле `shared-component-owner`. Без этого
   раз в неделю будет дубль.

2. **`labs` RAM-лимит — общий gate, никем не назначенный.** Минимум четыре
   активных предложения (`upgrade-to-2026-4-25`, `nodejs-runtime-upgrade`,
   `egress-network-policy`, `eso-cve-patch`) явно учитывают риск OOM в labs.
   Сейчас никто из них не назначен «первым замерщиком». **Решение:** Top-5 #2
   (openclaw upgrade) проводит RAM-замер первым и публикует baseline в
   `_mentor/` или в state mctl-gitops; остальные ссылаются. Без этого фразы
   «не увеличивает RAM более чем на 20MB» не от чего считать.

3. **Supply-chain как класс.** На неделе зафиксированы атаки одновременно
   на CI-инструмент (wrangler), npm-зависимости (Baileys/discord.js),
   marketplace (ClawHub, 341+ malicious skills) и npm-форки openclaw
   (`lotusbail`). Три отдельных proposals из разных сервисов независимо
   адресуют это: `npm-supply-chain-audit`, `clawhub-skills-allowlist`,
   `nodejs-runtime-upgrade`. Будут три похожие, но несинхронизированные
   CI-проверки в трёх репо. **Предложение:** общий шаблон CI-шага
   (allowlist resolved URL, deny-list имён, fail-closed default) в
   `mctl-gitops`, переиспользуемый из всех Dockerfile/CI каждого сервиса.

4. **Backstage scaffolder pod как concentration point credentials.** Top-5 #3
   показывает, что в одном pod монтируются Vault token, Postgres DSN, GitHub
   App credentials, и что любая path-traversal или env-leak уязвимость
   немедленно вытаскивает их все. Это архитектурный gap, который не лечится
   патчем версий. **Предложение в backlog (не на эту неделю):** ADR на
   разделение секретов scaffolder-pod по принципу least-privilege per action,
   либо вынести scaffolder из pod'а с другими секретами.

5. **MCP-видимость отсутствует.** Без `mcp__mctl__*` ментор не может
   подтвердить (a) фактические версии в production, (b) текущие открытые
   инциденты, (c) headroom RAM в labs, (d) deduplication уже выкаченных
   патчей. Дайджест в этом виде — sanity-check текстов proposals, а не
   сверка с реальностью. Доступ к MCP — обязательное условие осмысленности
   ментор-цикла; до восстановления каждое требование `pre-flight` в этом
   файле должен закрыть владелец сервиса.

6. **CVE clustering сигналит на координацию релизов.** В одну неделю
   приходят патчи по Argo Workflows, Helm, ESO, Backstage, openclaw, pgx,
   chi, Go toolchain, Node.js, wrangler, Nuxt. Это не «много мелочи», а
   синхронный апрельский batch upstream'ов. Если катить каждый отдельно,
   admins-tenant получит 5–6 деплоев за неделю. **Предложение:**
   `mctl-gitops` собирает «кластерный security release W18» — один
   ApplicationSet sync, окном, с rollback-планом, объединяющий Argo
   Workflows + Helm + ESO + Backstage. Снижает blast-window и упрощает
   координацию canary.
