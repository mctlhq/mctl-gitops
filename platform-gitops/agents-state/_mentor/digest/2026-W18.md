## Top 5 proposals ready for review

Ranked by impact / effort × urgency of the security signal × blast-radius
locality.

### 1. `mctl-web/wrangler-cve-0933` — CVSS 9.9 inside CI/CD

- **What:** raise wrangler to ≥ 4.59.1 in `deploy.yml` and
  `cloudflare-worker/package.json`; add `wrangler --version` and
  `npm audit --audit-level=critical` gates in CI.
- **Why:** OS command injection in `wrangler pages deploy` via `--commit-hash`.
  Anything controllable in commit metadata (branch/tag name) becomes arbitrary
  code execution inside a GitHub Actions runner that holds the Cloudflare API
  Token, `GITHUB_CLIENT_SECRET`, and `BACKSTAGE_LANDING_TOKEN`.
- **Impact / Effort:** **5 / 1**.
- **Conflicts:** none. Author kept Worker-secret rotation out of scope; at
  CVSS 9.9 that decision is **platform-level**, not service-level — open a
  rotation ticket immediately after merge.
- **Pre-flight:** confirm staging deploy via `wrangler pages deploy` after the
  bump; then chain `wrangler-full-upgrade` (4.59.1 → 4.86.0) as a follow-up,
  not as the same PR.
- Link: `mctl-web/proposals/wrangler-cve-0933/`.

### 2. `mctl-openclaw/upgrade-to-2026-4-26` — 9+ CVEs in a single image bump

- **What:** move openclaw image tag 2026.3.14 → **2026.4.26**, route
  `labs → admins → ovk` per ADR 0001, with a 24h minimum observation per tenant.
- **Why:** 2026.4.26 supersedes the earlier `upgrade-to-2026-4-25` proposal.
  The 4.25 base closed 5 CVEs (CVSS 8.8 agentic-consent bypass CVE-2026-41349
  among them). The 4.26 release adds 4 more high-severity items including
  privilege escalation via Telegram channel and the bearer-token-echo fix in
  `device.token.rotate`. The 2026-04-27 CVE addendum on `upgrade-to-2026-4-25`
  documents 9 additional `CVE-2026-356xx` items (incl. CVE-2026-35639 priv-esc
  CVSS 8.7, CVE-2026-35641 ACE via local plugin `.npmrc` CVSS 8.4, and
  CVE-2026-35649 settings-allowlist bypass on empty allowlist) — these are
  **already rolled into 2026.4.26**, so 4.26 is now the only path to take.
  → **Close `upgrade-to-2026-4-25` as superseded.**
- **Impact / Effort:** **5 / 3**.
- **Conflicts:** shares Helm values surface with `clawhub-skills-allowlist`
  (must be separate PRs). CVE-2026-35649 intersects directly with the allowlist
  proposal: the reviewer must verify the allowlist is **not empty** after the
  upgrade — empty means deny-all-bypass, the opposite of the intent.
- **Pre-flight (mandatory):** RAM measurement of a test pod running 2026.4.26
  with `labs` limits **before** the labs manifest commit. If delta > 50 MB,
  stop, file a limit ticket, do not proceed to admins/ovk. This single
  measurement becomes the **shared baseline** for all other openclaw and
  runtime proposals this week (see "Platform risks").
- Links:
  - `mctl-openclaw/proposals/upgrade-to-2026-4-26/`
  - `mctl-openclaw/proposals/upgrade-to-2026-4-25/cve-addendum-2026-04-27.md`

### 3. `mctl-portal` — Backstage 1.50.3 bundle (5 CVEs → 1 deploy)

- **What:** collapse five portal proposals into one Backstage 1.50.3 upgrade
  PR: `scaffolder-path-traversal` + `scaffolder-secret-leak` +
  `integration-scm-credentials` + `techdocs-path-traversal-fix` +
  `integration-credential-leak-fix`. Underlying bumps:
  `@backstage/backend-defaults` ≥ 0.12.2, `plugin-scaffolder-backend` ≥ 3.1.1,
  `@backstage/integration` ≥ 1.20.1, `@backstage/plugin-techdocs-node` ≥ 1.13.11.
- **Why:** all five vulnerabilities live in the same Backstage backend pod,
  whose ExternalSecret mounts a Vault token, the Postgres DSN, and GitHub App
  credentials. One pod, one credential blast radius:

    pod filesystem).
  - CVE-2026-32237 — dry-run endpoint leaks Vault secrets via non-recursive
    redaction in nested JSON.
  - CVE-2026-29185 — encoded path traversal in SCM URL handling steers
    GitHub App token to attacker host.
  - CVE-2026-23947 — TechDocs MkDocs symlink escape.
  - Unnamed integration credential-leak — `https://github.com%2F@evil.example`
    style host spoofing in the integration library.
- **Impact / Effort:** **5 / 3** (combined; each in isolation is 4/2).
- **Conflicts:** **must be one PR**, not five. Five separate prod deploys to
  `admins` cost five sync windows for the same risk. `labs` is unaffected.
  `mctl-portal/nodejs-security-patch` (Node.js 22 LTS base image) can ride the
  same image rebuild — combine.
- **Pre-flight:**
  - Rotate Vault token, Postgres DSN, GitHub App installation token in
    `admins` immediately after merge — any of them could already have leaked
    via the dry-run endpoint or the URL-decoding flaw.
  - TechDocs symlink validation is now strict: run a build against the
    highest-traffic TechDocs repos before promotion; expect intentional
    failures on out-of-tree symlinks, communicate in changelog.
  - `yarn backstage-cli versions:check` after the bulk bump.

### 5. `mctl-api/pgx-cve-sql-injection` — SQL injection on a public endpoint

- **What:** bump `github.com/jackc/pgx/v5` 5.8.0 → **5.9.2**.
- **Why:** CVE-2025-54236 (GHSA-j88v-2chj-qfwx) — placeholder confusion in the
  simple query protocol: `$N` inside a dollar-quoted string is interpreted as
  a bind parameter. mctl-api drives identity and audit-log behind this driver,
  and the endpoint is public (`https://api.mctl.ai`). Same bump also closes
  CVE-2026-4427 (panic DoS), CVE-2026-33815 and CVE-2026-33816 (out-of-bounds
  reads). Patch-bump only, public API preserved, transitives auto-resolve.
- **Impact / Effort:** **5 / 1**.
- **Conflicts:** there are **two near-duplicate proposals** —
  `mctl-api/pgx-cve-sql-injection` and `mctl-api/pgx-sql-injection-fix`. Both
  target the same version. Pick one (the `cve-` prefix matches naming used in
  other services), close the other as duplicate. **Must ship separately** from
  any Go-toolchain bump — do not let a critical CVE wait for a 1.24 → 1.26
  toolchain change.
- **Pre-flight:** `govulncheck` after the bump; integration tests against a
  real Postgres.
- Link: `mctl-api/proposals/pgx-cve-sql-injection/`
  (close `mctl-api/proposals/pgx-sql-injection-fix/` as duplicate).

## Platform risks

1. **Duplicate proposals inside a single service.** Three pairs are visible in
   the inventory:
   - `mctl-api/pgx-cve-sql-injection` vs `mctl-api/pgx-sql-injection-fix`
     (same target, same version).
   - `mctl-api/go-upgrade` vs `mctl-api/go-upgrade-stdlib-cves`
     (both 1.24 → 1.26.2).
   - `mctl-gitops/argo-workflows-cve-patch` (v4.0.5) vs
     `mctl-gitops/argo-workflows-cve-patch-v2` (v3.7.14) — two **incompatible**
     fix paths for the same CVE set.

   Without a preflight dedup step the reviewer ends up reading 6 proposals
   for 3 fixes. **Recommendation:** add a `supersedes:` field to the proposal
   template; require services to close superseded proposals before submission.

2. **Duplicate owners on shared components.** Argo Workflows arrived
   independently from `mctl-agent` and `mctl-gitops`. The same pattern will
   repeat on ESO, Helm, Backstage, ArgoCD, and any other cluster component.
   **Recommendation:** a platform ADR with an explicit owner registry for
   shared components, plus a `shared-component-owner:` field in the proposal
   template. Without it, this duplication recurs every week.

3. **`labs` RAM is a shared gate with no assigned measurer.** At minimum five
   active proposals reference labs OOM risk: `upgrade-to-2026-4-26`,
   `nodejs-runtime-upgrade`, `egress-network-policy`, `eso-cve-patch`,
   `vault-ssrf-acme-patch` (Path B). None of them is designated as the **first
   measurer**. **Resolution:** Top-5 #2 (openclaw 2026.4.26 upgrade) takes the
   first reading and publishes the baseline (e.g. in `_mentor/` or in
   mctl-gitops state); every later proposal compares against it. Without that
   anchor, "no more than +20 MB" is unanchored.

4. **Supply-chain risk is being addressed three times in three services.**
   This week's wave hits CI tooling (wrangler), npm transitive deps
   (Baileys / `discord.js-user` / `lotusbail`), the marketplace (ClawHub, with
   341+ malicious skills reported upstream), and axios (WAVESHAPER.V2). Three
   independent CI guards in three repos will drift. **Recommendation:** a
   single shared CI step in `mctl-gitops` (allowlist `resolved` URLs, deny-list
   names, fail-closed default) referenced from every service's Dockerfile/CI.

5. **Backstage scaffolder pod is a credential concentration point.** Top-5 #3
   shows that one pod mounts the Vault token, Postgres DSN, and GitHub App
   credentials, and that any path-traversal or env-leak vulnerability lifts
   them all at once. This is architectural, not patchable. **Backlog ADR
   (not this week):** least-privilege per scaffolder action, or split the
   scaffolder pod from co-resident secrets.

6. **CVE clustering signals release-train coordination.** A single week
   carries patches for Argo Workflows, ArgoCD, Helm, ESO, Backstage, openclaw,
   pgx, chi, mcp-go, Go toolchain, Node.js, wrangler, sqlite, Vault, Nuxt,
   Vue, sass. Rolled individually, `admins` will see 5–6 sync events in 7
   days. **Recommendation:** `mctl-gitops` packages a "W18 cluster security
   release" — one ApplicationSet sync, defined window, single rollback plan,
   bundling Argo Workflows + ArgoCD + Helm + ESO + Backstage. Cuts the
   blast-window and consolidates canary observation.

7. **MCP visibility is missing.** Without `mcp__mctl__*` tools the mentor
   cannot confirm (a) actual production versions, (b) currently open
   incidents, (c) labs RAM headroom, (d) deduplication against patches that
   already shipped. This digest is therefore a sanity-check of proposal text,
   not a reconciliation against reality. Restoring MCP visibility is a
   prerequisite for the mentor cycle to be meaningful; until then, every
   "pre-flight" line above is the service owner's responsibility.

8. **Proposals tree read-path is intermittent.** The 2026-04-30 re-run
   could not enumerate any `proposals/` directory from the mentor's
   execution context — the per-service symlinks point into the gitops
   state tree, which the sandbox refuses to traverse, and pattern-based
   reads returned empty across all seven services. The 2026-04-28 and
   2026-04-29 assemblies did not hit this. **Recommendation:** the
   orchestrator should either stage the gitops state into the mentor's
   working tree before the SDK launch (mirror it under `agents/`) or
   resolve symlinks during mount. Without a fix, future weekly runs will
   silently degrade to "no items" whenever the indirection breaks, and
   real fresh proposals will be missed.
