### 1. `mctl-openclaw/upgrade-to-2026-5-2` — direct hop to 2026.5.2 supersedes the 4.29 path

- **What:** bump every tenant (`labs` → `admins` → `ovk`) from
  **2026.3.14** straight to **2026.5.2** (released 2026-05-02). Closes
  all ten outstanding CVEs in one rollout — the seven that motivated
  `upgrade-to-2026-4-29` (CVE-2026-42422 / 42426 / 42428 / 42429 /
  42423 / 41912 / 41914) **plus** the three new ones folded into the
  prior W18 cve-addenda (CVE-2026-41394 unauth → `operator.write`,
  CVE-2026-41395 Plivo replay, CVE-2026-41390 exec allowlist bypass) —
  and additionally fixes the **CWE-532** credential leak that ships
  password / token / Authorization values into pod logs.
- **Why:** 2026.5.2 is now the canonical security target. It
  externalises `@openclaw/acpx` and `@openclaw/diagnostics-otel` as
  opt-in peer deps, so the **expected RSS delta for `labs` is negative**
  — this is the first openclaw upgrade in the series that *creates*
  rather than consumes RAM headroom (Platform risks #2). It also
  retires the entire `upgrade-to-2026-4-x` ladder (five superseded
  proposals: 4-8, 4-25, 4-26, 4-27, 4-29). One soak cycle, one
  rollback plan.
- **Impact / Effort:** **5 / 3.**
- **Conflicts:**
  - **Supersedes** the prior W18 Top-5 #2 (`upgrade-to-2026-4-29`).
    This is a **direction change** — do not roll 4.29 first; jump
    directly to 5.2. Close `upgrade-to-2026-4-25/26/27/29/8`.
  - The prior W18 cve-addenda (`cve-addendum-auth-bypass-exec-bypass`,
    `cve-addendum-ssrf-eval-bypass`) are subsumed: 2026.5.2 already
    carries every CVE they enumerate. Cite their CVE IDs in the
    canonical PR description; close the addenda.
  - `cwe532-log-credential-sanitization` (Block A below) becomes a
    **time-bounded interim** that only matters if the soak cycle
    extends beyond ~3 days. If 2026.5.2 lands cleanly on `labs`
    inside the first 24-hour soak, the log-shipper transform may be
    skipped entirely — but stage it ready to deploy on day 2 if the
    `admins` step slips.
  - `git-plugin-install-allowlist` (Top-5 #4 below) **must** be
    merged before any tenant flips `gitPluginInstall.enabled: true`
    on 2026.5.2. The proposal itself defaults to `enabled: false`, so
    landing the upgrade without the allowlist is safe; landing the
    feature flag without it is not.
- **Pre-flight:**
  - Audit `device.token.rotate` response-parsing tooling **before**
    the `ovk` step (response format changed in 2026.4.26 — explicit
    prerequisite gate per the proposal).
  - Take the `labs` RSS baseline at the start of the soak. The
    "expected negative delta" is the first time this digest can
    *de-risk* `labs`; the measurement must be recorded so the
    `nodejs-heap-cap-labs` proposal can finally be sized in W19.
  - Audit log-based monitoring rules that pattern-match
    `?token=` / `Authorization:` strings — they will stop firing
    after the upgrade, and that is correct.
  - Confirm `gitPluginInstall.enabled: false` in every tenant's
    Helm values at merge time. The new attack surface must not be
    activated by the same PR that introduces it.
- Link: `mctl-openclaw/proposals/upgrade-to-2026-5-2/`.

### 2. `mctl-web/cloudflare-vite-plugin-cve-fix` — dev-port secret exfil at 1-line fix cost

- **What:** pin `@cloudflare/vite-plugin` to **≥ 1.6.0** in `mctl-web`,
  closing **CVE-2025-59427 / GHSA-4pfg-2mw5-f8jx**. Versions below 1.6.0
  do not configure `server.fs.deny`; Vite's default `/@fs/` route then
  serves any reachable file — including `.dev.vars` and `.env` — over
  the dev port to anyone with network access to it.
- **Why:** the four secrets in `.dev.vars` are **production tokens**:
  `GITHUB_CLIENT_SECRET`, `TELEGRAM_BOT_TOKEN`, `RESEND_API_KEY`,
  `BACKSTAGE_LANDING_TOKEN`. Anyone on a coffee-shop Wi-Fi, a shared
  dev VM, a Codespace, or a CI runner that exposes the dev port could
  silently exfiltrate all four. The blast radius spans GitHub OAuth
  identity impersonation, bot-as-Telegram messaging, outbound email
  via Resend, and Backstage landing-tenant provisioning. The fix is a
  single dev-dep version bump; the cost/benefit is unambiguous.
- **Impact / Effort:** **5 / 1.**
- **Conflicts:**
  - Independent of every other mctl-web proposal in flight. Land as
    its own focused PR — do not bundle with `nuxt-444-formdata-fix`
    or `wrangler-node22-ci-upgrade`; the audit trail is more
    valuable than the rollout efficiency.
  - Out of scope (per the proposal, deliberately): **rotation** of the
    four tokens currently in `.dev.vars`. Rotation is incident response
    and must run in parallel — flag this loud in the PR description so
    it is not implicitly punted.
- **Pre-flight:**
  - Add the CI smoke test from the proposal's tasks: start the dev
    server, assert `GET /.dev.vars` and `GET /.env` are non-200.
  - Open a separate rotation ticket for the four tokens; treat any
    workstation that has run an unpatched dev server on a non-loopback
    interface as a possible exposure event.
- Link: `mctl-web/proposals/cloudflare-vite-plugin-cve-fix/`.

### 4. `mctl-openclaw/git-plugin-install-allowlist` — gate the new `git:` plugin attack surface

- **What:** add a tenant-scoped `gitPluginInstall.allowlist` Helm
  value that maps to openclaw's `pluginPolicy.gitInstallAllowlist`
  config key (introduced in 2026.5.2). Default `enabled: false`; CI
  step `check-git-plugin-allowlist` blocks the unsafe combination of
  `enabled: true` + empty allowlist. URL prefix matching, with SSH
  forms normalised to HTTPS canonical before comparison.
- **Why:** 2026.5.2 introduces `openclaw plugins install git:<url>`,
  which **bypasses every existing supply-chain control** —
  `npm-supply-chain-audit` (npm scope), `clawhub-skills-allowlist`
  (ClawHub marketplace), `egress-network-policy` (IP-level only).
  Without an URL-level allowlist, any compromised operator credential
  is one command away from arbitrary code execution inside the
  openclaw pod, and any allowlisted org compromise repeats the trick
  on `plugins update`. The platform-level gate stays `enabled: false`
  until enabling is a deliberate, gitops-PR-reviewed decision per
  tenant.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - **Companion** to Top-5 #1. Same release window. Order: land
    `git-plugin-install-allowlist` (defaults all tenants to deny),
    then land `upgrade-to-2026-5-2`, then — only after the soak is
    clean — open a separate per-tenant PR to enable the feature
    if/when it is needed.
  - Distinct from `egress-network-policy` (IP-level) and
    `npm-supply-chain-audit` (npm-only). All three are
    complementary; none substitutes for the others. The CVE
    classification table in the next mentor digest should reflect
    this triplet.
- **Pre-flight:**
  - Confirm the openclaw 2026.5.2 changelog entry for
    `pluginPolicy.gitInstallAllowlist` matches the proposal's
    schema (the proposal cites the upstream key by name; verify
    spelling against the release notes).
  - Add the CI step that fails on `enabled: true` + empty
    allowlist.
  - Stage the `enabled: false` block in every tenant's
    `helm/openclaw/<tenant>/values.yaml` even if the feature is
    never enabled — explicit beats implicit.
- Link: `mctl-openclaw/proposals/git-plugin-install-allowlist/`.

### 5. `mctl-gitops/prometheus-security-patch` — three Prometheus vulns, values-file audit

- **What:** pin Prometheus to **v3.11.3** (latest patch) across every
  values file in `platform-gitops/services/` and
  `platform-gitops/helm-charts/`. Audit — and disable —
  `auth.credentials_in_debug_log` (or the chart equivalent) at the
  configuration layer. Closes:
  - **AzureAD OAuth credential exposure** in stdout when debug logging
    is enabled.
  - **Snappy-decompress DoS** on the remote-write endpoint.
  - **Stored XSS** in the heatmap chart UI for users with metric-label
    write access.
- **Why:** Prometheus is the platform's only metrics backend. The DoS
  alone is sufficient motivation: any remote-write client that can
  craft a malformed snappy frame can crash the scraper, blinding alert
  rules across every tenant simultaneously. The credential-in-log path
  also intersects with the openclaw CWE-532 work (Top-5 #1) — both go
  to the same log aggregator. Patch shape is a values-file audit, no
  CRD or schema changes.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - **Bundle** into the W18 cluster security release alongside
    ArgoCD / Argo Workflows / Helm / ESO / Vault / Backstage / Grafana
    (Platform risks #6 carried). Single ApplicationSet sync, single
    rollback plan.
  - The `auth.credentials_in_debug_log` audit overlaps the
    openclaw CWE-532 mitigation philosophically — both stop secrets
    flowing into log-aggregation. Note in the PR description that
    the platform now has *two* layers of credential-in-log defence:
    application-level (openclaw 2026.5.2) and config-level
    (Prometheus debug-log flag).
  - No conflict with `argocd-v3-4-upgrade-plan` — different
    component, different reconciliation pass.
- **Pre-flight:**
  - `grep` every values file under `platform-gitops/services/` and
    `platform-gitops/helm-charts/` for the current Prometheus tag;
    surface the audit list in the PR description.
  - Verify the v3.11.3 changelog has no breaking config field
    rename between the prior pinned version and v3.11.3.
  - Apply alert-suppression for the brief scrape gap during the
    rolling update.
- Link: `mctl-gitops/proposals/prometheus-security-patch/`.

## Platform risks

1. **The openclaw direction just changed.** Two W18 drafts ago the
   canonical was `upgrade-to-2026-4-29`; this digest replaces it with
   `upgrade-to-2026-5-2`. The cve-addenda already triaged in the prior
   draft (`cve-addendum-auth-bypass-exec-bypass`,
   `cve-addendum-ssrf-eval-bypass`) **fold into** the 5.2 image rather
   than decorating 4.29. This is a major reversal mid-week.
   **Recommendation:** owners must explicitly close the 4-x proposal
   ladder (5 proposals) when the 5.2 PR opens. Without that clean-up
   step the inbox carries six contradictory signals about what the
   intended target version is.

2. **`labs` RAM gate may finally close — twice in one rollout.**
   Both Top-5 #1 (openclaw 2026.5.2 with externalised acpx +
   diagnostics-otel; expected *negative* RSS delta) and Top-5 #3
   (ArgoCD v3.4 RC7 staging) require a `labs` memory measurement.
   Combined, they produce the first quantitative RAM headroom number
   the platform has had since the gate was first flagged. The
   measurement is the prerequisite for sizing
   `nodejs-heap-cap-labs` (deferred to W19) and for green-lighting
   any future feature proposal that touches `labs`. **Owners must
   record the RSS baseline in `_mentor/digest/` so it is durable
   beyond the PR review window.**

3. **Multi-tenant isolation: the threat surface keeps adding layers.**
   Carried from the prior draft — ESO cross-namespace, ArgoCD project
   tokens, Vault token forwarding. This pass adds a fourth surface:
   the openclaw `git:` plugin scheme, which would let a single
   compromised operator credential bypass every existing supply-chain
   control. The `git-plugin-install-allowlist` (Top-5 #4) closes
   that fourth leg — but the systemic issue is that controls keep
   accruing per-component while the *isolation model itself* has not
   been re-audited end-to-end. The post-mortem-grade re-validation
   recommended last week is now overdue.

4. **Supply-chain risk hits a sixth layer this week.** The prior draft
   listed five (CI tooling / npm transitive / npm typosquats /
   marketplace / axios). This pass adds **dev-time secret exfil**
   (`cloudflare-vite-plugin-cve-fix`): a CVE class that lives
   **between** the developer's keyboard and the production token
   store. The mitigation pattern is the same — explicit pin + CI
   smoke test. The shared CI step in `mctl-gitops` flagged in the
   prior draft is now urgent: one place to enforce "supply-chain
   pin + lockdown + dev-time fs-deny" rather than per-service.

5. **`git:` is a new plugin attack surface, not an existing one.**
   Unlike the npm and ClawHub plugin paths (already gated), the
   `git:` scheme is brand new in 2026.5.2 and ships with no
   default platform-level allowlist. Activation is *one Helm value
   away* from arbitrary code execution. The 5.2 upgrade and the
   allowlist are technically independent, but operationally they
   are a single decision and must merge in the right order.

6. **CVE clustering still demands a release-train (carried).** This
   pass adds Prometheus, ArgoCD v3.4, and the openclaw 5.2 hop on
   top of the prior W18 wave. Rolled individually, `admins` would
   see ~10 sync events in 7 days. **Recommendation (carried,
   escalated again):** `mctl-gitops` packages **two** synchronised
   release windows for W18 — (a) the cluster security release
   (ArgoCD audit + Argo Workflows + Helm + ESO + Vault + Backstage +
   Grafana + **Prometheus**), (b) a follow-on data-plane release
   (**openclaw 2026.5.2** + portal Backstage v1.50.4 + mctl-agent
   Go 1.24.8 patch). One ApplicationSet sync per release, one
   rollback plan per release.

7. **Duplicate clusters keep growing (carried, escalated).** This pass
   adds three new duplicates in `mctl-agent` alone (`go-runtime-upgrade`,
   `chi-open-redirect`, `sqlite-upgrade`). The `supersedes:` field in
   the proposal template is now blocking — without it, the deduplication
   tax for the next mentor cycle will exceed the new-content tax.

8. **MCP visibility still missing.** `mcp__mctl__*` is unavailable.
   No reconciliation against (a) production image tags, (b) open
   incidents, (c) `labs` RAM headroom, (d) PRs that already merged
   (we have to read `.status.yaml` markers manually). Restoring MCP
   visibility is a prerequisite for the mentor cycle to be
   *reconciled* rather than *inferred*.

9. **Dev-time secrets are production secrets.** The four tokens in
   `.dev.vars` (Top-5 #2) are not "dev shadow" tokens — they are the
   real production credentials. The platform's threat model has
   silently assumed dev workstations are isolated; in practice they
   share networks (cafés, co-working, CI). Either the dev
   credentials become true shadow tokens with their own scope, or
   the dev server must default to loopback-only. `cloudflare-vite-plugin
   ≥ 1.6.0` closes the immediate CVE; the architectural decision is
   still open.

10. **Backstage v1.50.4 is now a four-proposal bundle.** The W18 Top-5
    already names the security tightening (`fetchurlreader-ssrf` +
    `scaffolder-symlink-traversal-v2`). This pass adds two more
    targeted to the same v1.50.4 line (`catalog-unprocessed-security-patch`
    + `catalog-facets-perf-fix`). All four are a **single
    `yarn up '@backstage/*@^1.50.4'`** — one PR, one rebuild, one
    rollout. Splitting them would multiply the review-and-soak load
    for no benefit.
