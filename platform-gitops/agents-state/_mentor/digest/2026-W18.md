### 1. `mctl-gitops/eso-cross-namespace-secret-access` — multi-tenant isolation breach in the secret-store layer

- **What:** upgrade ESO Helm chart to **2.4.1** (controller + CRDs)
  closing **CVE-2026-22822** — the controller did not validate that an
  `ExternalSecret`'s namespace matches the Vault path namespace, so a
  manifest in tenant `labs` could resolve secrets from a `admins`-only
  Vault path. Distinct from `eso-dns-exfil-patch` (DNS exfil via
  `getHostByName`) and `eso-cve-patch` (still embargoed).
- **Why:** ESO is the **sole secret delivery channel** for every
  tenant. The platform's `labs` / `admins` separation is enforced by
  Kubernetes namespace + Vault policy; this CVE breaks the policy half
  of that pair. Anyone able to merge an ExternalSecret to `labs` could
  read `admins`-scoped material (Vault tokens, DSNs, GitHub App
  credentials). The mctl tenancy model is built on the assumption that
  has just been falsified.
- **Impact / Effort:** **5 / 2**.
- **Conflicts:**
  - **Distinct** from prior W18 ESO proposals. Bundle all three ESO
    fixes (`eso-dns-exfil-patch`, `eso-cve-patch`, this one) into the
    same chart bump (`helm-chart-2.4.1`) **only if** that chart in
    fact carries every patch — verify the Chart.yaml + controller image
    tag before merge (the inbox flagged the chart release notes failed
    to load on 2026-05-01).
  - The new policy enforces a tighter scope than current Vault
    policies assume: Top-5 #4-prior (ESO + Vault paired audit) must
    pre-walk every existing `ExternalSecret` for cross-namespace
    resolves and either move them or whitelist them explicitly before
    upgrade — otherwise legitimate workloads will break.
- **Pre-flight:**
  - Audit `platform-gitops/argo-workflows/secrets/` and every tenant
    overlay for `ExternalSecret.spec.data[].remoteRef.key` paths whose
    Vault path namespace differs from the manifest's `metadata.namespace`.
  - Snapshot Vault before the chart bump (consistent with Top-5 #4-prior).
  - Roll behind a flag in `labs` first; the cross-namespace resolves
    that fail post-upgrade are exactly the previously-undetected
    policy violations and must be tracked, not silenced.
- Link: `mctl-gitops/proposals/eso-cross-namespace-secret-access/`.

### 2. `mctl-portal/nodejs22-security-batch` — canonical Node.js base-image pin (2 Highs on a public surface)

- **What:** pin the Backstage Docker base image to **`node:22.22.2-alpine`**
  in `mctl-portal`. Closes:
  - **CVE-2026-21637** — TLS SNI crash (High; remote DoS).
  - **CVE-2026-21710** — `__proto__` HTTP header prototype-chain
    corruption (High).
  - Four Mediums (HTTP/2 leak, crypto timing, HashDoS, Permission Model bypass).
  - Two Lows.
- **Why:** the portal serves the public surface at `app.mctl.ai`. Both
  Highs are remotely reachable from that surface — TLS SNI crash is a
  one-packet DoS, `__proto__` header pollution can corrupt downstream
  request handlers in the Backstage backend pod that also mounts a
  Vault token, the Postgres DSN, and the GitHub App credential. This
  is the same blast-radius pile flagged in Top-5 #3-prior.
- **Impact / Effort:** **5 / 2**.
- **Conflicts:**
  - **Subsumes** `mctl-portal/nodejs-security-upgrade`,
    `nodejs-process-crash-cves`, and the older `nodejs-security-patch`.
    Adopt **`nodejs22-security-batch`** as the canonical Node-pin
    proposal; close the other three after the canonical PR description
    enumerates every CVE ID they covered.
  - Architecture choice: the prior canonical was
    `node:22.22.2-bookworm-slim`; this one switches to
    `-alpine`. Confirm `backstage-cli` and the native dependencies
    (sharp, sqlite3, isolated-vm if used) build on Alpine before
    accepting the switch — otherwise keep `-bookworm-slim`.
- **Pre-flight:**
  - Confirm Alpine native-build compatibility for every TechDocs /
    scaffolder action that compiles native code.
  - Carry the pre-flight already required by Top-5 #3-prior (Vault
    token + Postgres DSN + GitHub App rotation; `backend.reading.allow`
    audit; TechDocs build smoke).
  - Bundle this image bump into the **same PR** as Top-5 #3-prior
    (catalog + scaffolder + fetchurl + this) — five-PR fragmentation
    on one pod is unacceptable (Platform risks #5).
- Link: `mctl-portal/proposals/nodejs22-security-batch/`.

### 3. `mctl-openclaw/cve-addendum-auth-bypass-exec-bypass` — expedited `ovk` window for openclaw 2026.4.29

- **What:** documentation-only EARS addendum to `upgrade-to-2026-4-29`
  naming three further CVEs that the **same** image already closes:
  - **CVE-2026-41394** — unauthenticated plugin-auth path reaches
    `operator.write` (High).
  - **CVE-2026-41395** — Plivo V3 webhook replay (Medium).
  - **CVE-2026-41390** — exec allowlist bypass (Medium).

  Recommends **expediting** the `ovk` rollout step rather than waiting
  for the standard ADR-0001 24-hour soak per tenant.
- **Why:** the CVE-2026-41394 path is the highest-severity
  openclaw item to surface in W18. The `operator.write` permission
  controls multi-tenant plugin lifecycle on `ovk` and is the same
  permission boundary that the 4.29 image just hardened in three
  unrelated places. Holding `ovk` at 2026.3.14 for the full 24-hour
  soak after `admins` is now an **explicit risk decision**, not a
  default-safe path.
- **Impact / Effort:** **4 / 1** (doc-only; rides Top-5 #2-prior).
- **Conflicts:**
  - Companion to `cve-addendum-ssrf-eval-bypass` (prior). Both
    addenda decorate `upgrade-to-2026-4-29`. Acknowledge in a single
    PR description.
  - The "expedite `ovk`" recommendation conflicts with ADR-0001's
    standard cadence. Treat as a one-off exception requested by the
    addendum author; document the exception explicitly in the rollout
    PR and on the postmortem-ready checklist. Do **not** quietly
    shorten the soak — name the deviation.
- **Pre-flight:**
  - Pre-stage the `ovk` Helm values bump in mctl-gitops behind a
    paused ApplicationSet so that "expedited" still goes through the
    GitOps surface, not a console kubectl.
  - Re-verify that the labs RAM measurement (Platform risks #3) has
    actually been taken — accelerated `ovk` promotion without that
    baseline is a second risk decision that should not stack on top
    of an ADR exception.
- Link: `mctl-openclaw/proposals/cve-addendum-auth-bypass-exec-bypass/`.

### 4. `mctl-portal/fetchurlreader-ssrf` + `scaffolder-symlink-traversal-v2` — tightened Backstage pins on top of Top-5 #3-prior

- **What:** two strictly-tightened pins on the v1.50.4 bundle:
  - `fetchurlreader-ssrf` — bump `@backstage/backend-defaults`
    **≥ 0.15.0** to add per-hop redirect validation against
    `backend.reading.allow` (3xx redirects from a whitelisted URL were
    previously followed without re-checking the new destination —
    catalog-import, TechDocs, proxy plugin all reach internal
    `mctl-api` / Vault / Prometheus).
  - `scaffolder-symlink-traversal-v2` — raise the
    `@backstage/plugin-scaffolder-backend` pin from
    **≥ 3.1.1** (prior) to **≥ 3.1.5** to cover three additional
    handlers (`debug:log`, `fs:delete`, archive-extraction). The
    container-escape blast radius is the same Vault-mounted secret
    pile.
- **Why:** these are not new CVE classes — they are tighter version
  ceilings on the **same** plugins already targeted by the Top-5 #3
  bundle. Land as part of the same image rebuild (one PR, one rollout).
  Refusing to tighten the version ceiling because the original pin
  technically already merged would leave the three additional handlers
  open until the next Backstage cycle.
- **Impact / Effort:** **4 / 1** (paired; combined into the bundle).
- **Conflicts:**
  - `fetchurlreader-ssrf` is **distinct** from `fetchurl-ssrf` (the
    prior fetchurl proposal targeted package bumps + a
    `backend.reading.allow` audit; this targets per-hop redirect
    re-validation specifically). Both fix Backstage SSRF surfaces;
    bundle, do not pick one.
  - `scaffolder-symlink-traversal-v2` **supersedes**
    `scaffolder-symlink-traversal` and the umbrella
    `scaffolder-path-traversal` for the version-ceiling decision.
    Close the older two as folded into v2.
- **Pre-flight:**
  - `yarn backstage-cli versions:check` after the bulk bump — both
    bumps cross peer-dep boundaries.
  - Run the dry-run path-traversal probes from
    `scaffolder-path-traversal` against the new pin before merge.
  - Audit `backend.reading.allow` for entries that were tolerable
    only because the redirect chain was previously followed (e.g.
    short-link rewriters); these will fail post-bump and that is
    correct behaviour.
- Links:
  - `mctl-portal/proposals/fetchurlreader-ssrf/`
  - `mctl-portal/proposals/scaffolder-symlink-traversal-v2/`
  - prior umbrella: `mctl-portal/proposals/fetchurl-ssrf/`,
    `mctl-portal/proposals/scaffolder-symlink-traversal/`,
    `mctl-portal/proposals/scaffolder-path-traversal/` (close the
    last two as folded into v2; keep `fetchurl-ssrf` for the

  - Inspect `package-lock.json` for any string match of

  - Add the `.npmrc` scope `:registry=null` for the Discord-adjacent

   reading. As of 2026-05-02 the rollout has not yet shipped, so the
   baseline still does not exist and `nodejs-heap-cap-labs` cannot be
   sized. Multiple proposals continue to depend on it. The `Alpine`
   switch in the new `nodejs22-security-batch` is **not** a
   substitute — different service, different node version, different
   tenant.

4. **Multi-tenant isolation is now under simultaneous attack at three
   layers.** ESO cross-namespace (Top-5 #1 this update), ArgoCD
   project-scoped tokens (Top-5 #1-prior), Vault token-forwarding via
   `Authorization` pass-through (Top-5 #4-prior). The platform's
   isolation model — namespace + project + Vault policy — has had a
   CVE land on each leg in the same week. Even with all three patches
   shipped, a postmortem-grade re-validation of "what tenant `labs`
   can read from `admins`-scoped material" is overdue.

5. **Supply-chain risk is now hit at five layers in one week.** CI
   tooling (wrangler), npm transitive deps (baileys + npm-audit), npm
   typosquats (Discord — new this update), marketplace (ClawHub
   allowlist), axios (`axios-supply-chain-pinning`). One shared CI
   step in `mctl-gitops` enforcing `resolved` URL allowlist + name
   deny-list + `:registry=null` for typosquat scopes is the
   minimum-viable consolidation.

6. **CVE clustering still demands a release-train.** This update adds
   ESO cross-namespace, Terraform, Node 22, and Go 1.24.8 patches on
   top of the W18 wave. Rolled individually, `admins` will see 9+
   sync events in 7 days. **Recommendation (carried, escalated):**
   `mctl-gitops` packages **two** synchronised release windows for
   W18 — (a) the cluster security release (ArgoCD + Argo Workflows +
   Helm + ESO + Vault + Backstage + Grafana), (b) a follow-on data
   plane release (openclaw 2026.4.29 + portal Node 22 batch +
   mctl-agent Go 1.24.8 patch). Each is one ApplicationSet sync, one
   rollback plan.

7. **MCP visibility is still missing.** `mcp__mctl__*` is unavailable.
   Live cross-checks against (a) actual production versions, (b)
   currently open incidents, (c) `labs` RAM headroom, (d)
   deduplication against patches that already shipped — none are
   reconciled here. This digest remains a sanity-check of proposal
   text. Restoring MCP visibility is a prerequisite for the mentor
   cycle to be **reconciled** rather than **inferred**.

8. **Two-track Go upgrade is the right answer; one-track was wrong.**
   `mctl-agent/go-runtime-patch` (1.24.8) closes 10 stdlib CVEs at
   patch-level risk. The prior single-track 1.26.2 plan held those
   CVEs hostage to a minor hop with TLS cipher-suite default changes
   and Green Tea GC behaviour. Ship the patch on its own merits in
   W18; revisit 1.26.2 as a W19 platform-aligned decision owned by
   `mctl-gitops`. Do **not** wait for the minor hop to close
   patch-level CVEs.

9. **`fetchurlreader-ssrf` reveals an architectural choice.**
   Backstage `FetchUrlReader` redirect re-validation belongs in
   `backend-defaults`, not in each plugin. The new pin (≥ 0.15.0)
   centralises the check; expect future Backstage CVEs in this class
   to be fixed at the same layer. Maintain the
   `backend.reading.allow` audit as the **policy** input to the
   centralised check, not as scattered per-plugin allowlists.

10. **Proposal counts are growing faster than triage capacity.** W18
    received ~60 net proposals across 7 services; this update alone
    added 17. Weekly proposal volume per service has tripled since
    the policy of "submit per CVE" was adopted. The sustainable
    primitive is a CVE-batch proposal (cf. `nodejs22-security-batch`)
    rather than per-CVE proposals — recognise the pattern in the
    template and reward the consolidation explicitly.
