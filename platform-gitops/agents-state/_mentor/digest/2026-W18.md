### 1. `mctl-gitops/argocd-repo-cred-leak-patch` — CVSS 10.0 in the platform's sync engine

- **What:** upgrade ArgoCD to **v3.3.9** (released 2026-04-30) to close
  CVE-2025-55190. The same image bump removes the
  ApplicationSet-generator panic, the Redis cache regression, and the
  pod-log UI crash that yesterday's `argocd-informer-cache-patch`
  wanted to fix at v3.3.8 — v3.3.9 is **strictly newer** and carries
  every fix in that proposal.
- **Why:** any project-scoped ArgoCD API token can call
  `/api/v1/projects/{project}/detailed` and read every repository
  credential stored on that project (deploy keys, GitHub App bot
  password) without any explicit `projects, get-secret` grant. ArgoCD
  is the **single sync engine** for the entire platform via the
  App-of-Apps + three ApplicationSets topology; a credential dump here
  is a write-path compromise of every tenant. CVSS 10.0.
- **Impact / Effort:** **5 / 2**.
- **Conflicts:** **supersedes** `mctl-gitops/argocd-informer-cache-patch`
  — close the v3.3.8 proposal as superseded; ride the same patch
  window. Pair with Top-5 #2 below (Argo Workflows v3.7.14): both bumps
  belong in the same App-of-Apps sync window so `admins` is rebooted
  once, not twice.
- **Pre-flight:**
  - Apply CRDs first, then controller (standard ArgoCD upgrade).
  - Verify all ApplicationSets re-render; expected ~10 min cold-cache
    reconciliation after first boot.
  - Add a CI gate (`conftest`) rejecting any future ArgoCD pin below
    v3.1.2.
  - **Verify the v3.3.9 image bundles Helm v4.1.4** (the
    `helm-security-patch` chart-extraction CVEs are otherwise unfixed —
    Platform risks #6, prior digest).
  - **Trigger consumer-side rotation:** any mctl-api / mctl-agent /
    automation that holds a project-scoped token must be assumed
    compromised until rotated; this is what the new
    `mctl-api/argocd-token-scope-audit` is for (Block C).
- Link: `mctl-gitops/proposals/argocd-repo-cred-leak-patch/`
  (close `mctl-gitops/proposals/argocd-informer-cache-patch/` as superseded).

### 2. `mctl-openclaw/upgrade-to-2026-4-29` — supersedes the entire openclaw upgrade ladder

- **What:** move openclaw image tag 2026.3.14 → **2026.4.29** in each
  tenant's Helm values via mctl-gitops, following ADR-0001
  `labs → admins → ovk` with the ADR-0002 s3-sync-canary stop /
  delayed-restart and `restore-state` probe guards. 24-hour soak per
  tenant; `ovk` only inside the pre-approved low-traffic window.
- **Why:** 2026.4.29 became upstream-stable on 2026-04-30 and now
  supersedes **four** earlier upgrade proposals: 4-8, 4-25, 4-26, and
  yesterday's Top-5 #2 4-27. It closes the same six HIGH CVEs called
  out yesterday (CVE-2026-41342 / 41349 / 41352 / 41353 / 41359 /
  41371) **plus** the seven CVEs explicitly enumerated in the upstream
  release notes for the 4.28+ window — see the
  `cve-addendum-ssrf-eval-bypass` proposal for the EARS-form list:
  - CVE-2026-42422 (8.8) — token-role bypass via `device.token.rotate`
  - CVE-2026-42426 (8.8) — improper authz in `node.pair.approve`
  - CVE-2026-42428 (7.1) — missing plugin integrity verification
  - CVE-2026-42429 (7.1) — gateway plugin HTTP auth privilege escalation
  - CVE-2026-42423 — `strictInlineEval` approval-timeout bypass (sandbox escape)
  - CVE-2026-41912 — SSRF policy bypass
  - CVE-2026-41914 — QQ Bot media SSRF (channel in active use)

  2026.4.29 also delivers the **gateway slow-startup / stale-session
  recovery** fixes that directly reduce `restore-state` probe timeout
  risk during the rollout itself — a behaviour that has bitten
  prior openclaw promotions.
- **Impact / Effort:** **5 / 3**.
- **Conflicts:**

    allowlist must land first **and** be non-empty after merge

    `@backstage/backend-defaults` ≥ 0.15.0.

### 4. `mctl-gitops` secret-store layer — ESO v2.2.1 + Vault v2.0.0 paired audit

- **What:** two paired upgrades on the platform's secret-distribution
  layer, scheduled into the same cluster security release as Top-5 #1:
  - **ESO → v2.2.1** (chart `helm-chart-2.4.1`, **patch inclusion to
    be verified before merge**) closing CVE-2026-34984 — the v2
    template engine exposes `getHostByName` to template authors,
    permitting covert DNS-based exfiltration of any secret material
    visible to ESO. Adds an OPA/conftest gate rejecting future
    ExternalSecret manifests that reference `getHostByName`.
  - **Vault → v2.0.0 Community** closing CVE-2026-4525 (token
    forwarding via `Authorization` pass-through) and CVE-2026-5807
    (unauthenticated `sys/rekey` slot monopolisation, admin DoS).
    Step 1 is a pre-upgrade audit of every auth mount for
    `passthrough_request_headers` containing `Authorization`; that
    audit closes CVE-2026-4525 even before the upgrade lands. Step 3
    verifies `sys/rekey`, `sys/generate-root`, `sys/rekey-recovery-key`
    return 403 unauthenticated post-upgrade.
- **Why:** ESO is the **sole secret delivery channel** for every
  tenant; Vault is the upstream of ESO. A compromise on either is a
  platform-wide credential exposure. Both patches arrive in the same
  release window, share an audit (auth mounts + ExternalSecret
  templates), and are best applied with a single rollback plan.
- **Impact / Effort:** **5 / 3** (paired); each in isolation is 4 / 2.
- **Conflicts:**
  - The new `eso-dns-exfil-patch` is **distinct** from the prior
    `eso-cve-patch` (different CVE class — DNS exfil via template
    function vs the embargoed v2.4.0 fix). Both should ship together
    in the ESO bump, but the `eso-cve-patch` embargo signal is still
    not public — track separately.
  - The new `vault-cve-2026-token-exposure` is **distinct** from the
    prior `vault-ssrf-acme-patch` and `vault-kvv2-deletion-bypass`.
    Bundle all three Vault-side proposals into the v2.0.0 cutover so
    that the breaking-change review and ESO-auth validation only
    happen once.
- **Pre-flight:**
  - Inspect `helm-chart-2.4.1` `Chart.yaml` and ESO controller image
    tag — verify v2.2.1+ is bundled. Today's inbox flagged the chart
    release notes as failing to load; this verification is mandatory.
  - Audit every ExternalSecret in `platform-gitops/argo-workflows/secrets/`
    for `getHostByName` (none expected; required regardless).
  - Take a verified Vault snapshot before the v2.0.0 cutover; storage
    migration is one-way.
  - Validate the ESO `vault-backend` ClusterSecretStore reauthenticates
    successfully against Vault v2.0.0 in a non-production validation
    step before production cutover.
  - Audit any monitoring / health-check tooling that calls Vault
    unauthenticated — v2.0.0 may require token-gating.

### 5. MCP write-tool hardening — `mctl-api` server-side + `mctl-agent` allowlist

- **What:** two complementary application-layer fixes on the MCP
  attack surface:
  - `mctl-api/mcp-rce-server-hardening` — per-tool authz wrapper for
    the 13 write tools (config in `config/mcp-tool-permissions.yaml`),
    `http.MaxBytesReader` 64 KB cap on the MCP endpoint,
    per-`(identity, write)` rate limit (default 30/min, env-tunable
    via `MCP_WRITE_RATE_LIMIT`), and a `mcp_write_audit` Postgres
    table recording SHA-256 of each write call's arguments for an
    immutable audit trail.
  - `mctl-agent/mcp-input-validation` — per-tool allowlist schemas
    (regex + length + enum) for the agent's six MCP tools (`create_pr`,
    `list_tickets`, `get_ticket`, `silence_alert`, `query_alerts`,
    `update_ticket_state`); reject-by-default on unknown tool names;
    structured `slog.Warn` rejection logs.
- **Why:** Top-5 #5 in the prior draft (the `mcp-go-upgrade-v0.50` +
  pgx/chi line-level CVE bump) closes **library-level** issues on
  `mctl-api`. These two new proposals close the **application-level**
  layer that sits above the library — the surface where the
  CVE-2025-49596 family lives, and the surface that
  `mcp-go-upgrade-v0.50`'s `WithInputSchemaValidation()` hook is
  expected to harden in coordination with. Until both layers exist,
  every write tool relies on per-handler validation written in 2025
  (Platform risks #9, prior digest).
- **Impact / Effort:** **4 / 2** (paired).
- **Conflicts:**
  - `mctl-api/mcp-rce-server-hardening` is **complementary**, not
    competing, with `mcp-go-upgrade-v0.50`. Land library-level
    schema validation first (Top-5 #5 prior draft), then the
    application-level authz / size cap / rate limit / audit log on top.
  - `mctl-agent/mcp-input-validation` and the prior
    `mctl-agent/mcp-command-injection-hardening` overlap on intent
    but at different layers (allowlist regex schemas vs command
    surface hardening). Land the regex layer first; revisit the
    command-injection proposal with the allowlist in place.
- **Pre-flight:**
  - Validate every Claude-API-generated argument shape for each of the
    13 + 6 = 19 MCP tools against the proposed regex + length + enum
    rules, using the production logs of the last 14 days. Adjust
    schemas to accept all currently-known-valid inputs **before**
    deploy; the `-32602` failure mode is not graceful.
  - One additive Postgres migration (`mcp_write_audit` + two indexes)
    on the `mctl-api` database; runs via the existing `migrations/`
    directory. Verify the migration is reversible.
  - Confirm the `httprate` 0.15 store extension does not regress the
    existing global limit; integration test against the 60-second
    rollover boundary.

   - mctl-web vue-router: `vue-router-v5-migration` (planning) /
     **`vue-router-v5-migration-v2`** (execute) — these are companions,
     not duplicates, but both exist as separate proposals.
   - Plus all eight clusters carried over from yesterday (pgx triple,
     chi pair, mcp-go pair, mctl-api Go-upgrade triple, wrangler
     triple, Nuxt triple, openclaw outbound docs pair, Argo Workflows
     cross-service).

   **Recommendation (carried over):** add a `supersedes:` field to the
   proposal template; require services to close superseded proposals
   before submitting new ones. Without preflight dedup, the reviewer
   reads ~30 proposals for ~12 fixes this week.

2. **Cross-service shared-component ownership is still unwritten.**
   Argo Workflows (gitops vs agent), ArgoCD (gitops + consumer-side
   token scope on api), Backstage (portal + docs), Vault (gitops + api
   indirectly through ESO) keep arriving from multiple owners. A
   platform ADR with an explicit owner registry plus a
   `shared-component-owner:` proposal-template field is overdue.

3. **`labs` RAM gate now has both a measurer AND a mitigation.** Top-5
   #2 (openclaw 2026.4.29) takes the first reading and publishes the
   baseline. The new `nodejs-heap-cap-labs` is the **first concrete
   structural mitigation** — explicit V8 heap ceiling on the labs pod
   only, using the Node.js 24.15.0 LTS stable `--max-heap-size` flag.
   The cap value `N` depends on the baseline, so the dependency goes
   one way: Top-5 #2 baseline → `nodejs-heap-cap-labs` profiling.
   Multiple proposals still depend on the baseline:
   `nodejs-runtime-upgrade`, `egress-network-policy`, `eso-cve-patch`,
   `vault-ssrf-acme-patch` Path B, `go-upgrade-1-26`. Without the
   baseline, "no more than +50 MB" remains unanchored.

4. **Supply-chain risk is now hit at four layers in one week.** CI
   tooling (wrangler + the new `wrangler-ci-injection-audit`), npm
   transitive deps (`baileys-registry-lockdown` + `npm-supply-chain-audit`),
   marketplace (ClawHub allowlist), and axios (`axios-supply-chain-pinning`).
   The recommendation from yesterday stands: a single shared CI step
   in `mctl-gitops` (allowlist `resolved` URLs, deny-list names,
   fail-closed default) referenced from every service Dockerfile / CI.
   `baileys-registry-lockdown`'s `.npmrc` scope pin remains the
   first concrete primitive ready for promotion upward.

5. **Backstage scaffolder pod stays a credential concentration point.**
   Top-5 #3 collects the W18 wave; the architectural concern is
   unchanged — one pod, one Vault token + Postgres DSN + GitHub App
   credential mount, every CVE class lifts the whole pile. Backlog
   ADR (next week, not this week): least-privilege per scaffolder
   action, or split the scaffolder pod from co-resident secrets.

6. **CVE clustering still demands a release-train.** A single week
   carries patches for ArgoCD, Argo Workflows, Helm, ESO, Vault,
   Backstage, openclaw, pgx, chi, mcp-go, Go toolchain, Node.js,
   wrangler, sqlite, Nuxt, vue-router, Vue, sass — and now adds
   ArgoCD CVSS 10.0 on top. Rolled individually, `admins` will see 8+
   sync events in 7 days. **Recommendation (carried over):**
   `mctl-gitops` packages a "W18 cluster security release" — one
   ApplicationSet sync, defined window, single rollback plan, bundling
   ArgoCD + Argo Workflows + Helm + ESO + Vault + Backstage + Grafana.
   Cuts the blast-window and consolidates canary observation.

7. **MCP visibility is still missing.** `mcp__mctl__*` is not exposed
   to this run. Live cross-checks against (a) actual production
   versions, (b) currently open incidents, (c) labs RAM headroom,
   (d) deduplication against patches that already shipped — none are
   reconciled here. This digest remains a sanity-check of proposal
   text. Restoring MCP visibility is a prerequisite for the mentor
   cycle to be **reconciled** rather than **inferred**.

8. **Proposals tree read-path: stable via `Glob`+`Read`, broken via
   `ls`.** Direct `ls` of per-service `proposals/` is still blocked
   (gitops state-tree symlinks). `Glob` and `Read` against file paths
   work, so this run enumerated cleanly. The orchestrator should
   still mirror the gitops state into the mentor working tree (or
   resolve symlinks at mount time) so a future run does not silently
   degrade to "no items."

9. **MCP write-tool surface now has a 3-layer hardening plan.**
   Library layer (`mcp-go-upgrade-v0.50` schema validation),
   application layer (Top-5 #5 — per-tool authz + size cap + rate
   limit + audit log on `mctl-api`; allowlist regex on `mctl-agent`),
   and the still-open CVE-2026-27896 question (JSON-RPC field-name
   smuggling, confirmed in the official `go-sdk`). All three must
   land before the per-handler 2025 validation can be retired.

10. **ArgoCD CVSS 10.0 invalidates every consumer-side token's
    confidentiality assumption.** Even after Top-5 #1 lands,
    every project-scoped ArgoCD API token issued before this week
    must be assumed leaked — the consumer-side rotation in
    `mctl-api/argocd-token-scope-audit` is **mandatory** and
    blocking, not advisory. Other services that hold ArgoCD tokens
    (mctl-agent? mctl-portal?) need the same audit; the inboxes
    surveyed for this digest do not reveal any other holders, but
    that absence is itself unverifiable without `mcp__mctl__*`.
