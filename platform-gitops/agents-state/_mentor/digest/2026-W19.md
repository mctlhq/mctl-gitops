### 2. `mctl-portal/techdocs-mkdocs-code-exec` ŌĆö Backstage RCE in TechDocs MkDocs subprocess

- **What:** bump `@backstage/plugin-techdocs-node` to **^1.14.3** in
  `packages/backend/package.json`. Versions before 1.14.3 do not validate
  the `plugins` and `hooks` sections in user-supplied `mkdocs.yml` against
  the allowlist before spawning the MkDocs subprocess, allowing injection
  of arbitrary Python callables. **The exploit prerequisite is catalog
  write access** ŌĆö i.e., any authenticated user able to open a PR to a
  catalog component's repo.
- **Why:** the Backstage backend pod runs as the `mctl-portal`
  ServiceAccount, which holds a Vault token with read access to platform
  secrets. A successful code-exec therefore grants full read of the
  platform's secret bundle (Vault token, Postgres DSN, GitHub App creds
  ŌĆö same bundle as the W19 `scaffolder-secret-leak` blast radius).
  Distinct CVE class from `techdocs-path-traversal-fix` (CVE-2026-23947,
  symlink-escape) already in earlier-W19 Top-5 #4 ŌĆö both fold into the
  v1.50.4 yarn-up. The Backstage v1.50.4 bundle now carries **eight**
  CVEs in one PR.
- **Impact / Effort:** **5 / 1.**
- **Conflicts:**
  - **Bundles** with the in-flight v1.50.4 yarn-up (W18 Block C +
    earlier-W19 Top-5 #4). Verify the `^1.50.4` resolution actually
    pulls `@backstage/plugin-techdocs-node Ōēź1.14.3`; if not, add an
    explicit `resolutions` override.
  - No conflict with `auth-backend-redirect-bypass` /
    `auth-backend-metadata-ssrf` (Block C below) ŌĆö same yarn-up cycle,
    different package. Bundle all three in the v1.50.4 PR.
- **Pre-flight:**
  - Run a dry-run TechDocs build against every catalog component in
    staging post-bump; surface any mkdocs-plugin allowlist rejections to
    the owning teams **before** production cutover.
  - Add `yarn audit --level high` to the CI pipeline gated on the
    TechDocs build job (per the proposal's task 3).
  - We run `techdocs.generator.runIn: local` (per the
    `techdocs-path-traversal-fix` proposal's audit), so the backend pod
    is in scope.
  - Treat the Vault-token blast radius as motivation to file a

### 3. `mctl-portal/auth-backend-redirect-bypass` + `auth-backend-metadata-ssrf` ŌĆö paired auth-backend bump (Ōēź0.27.1)

- **What:** bump `@backstage/plugin-auth-backend` to **^0.27.1** in
  `packages/backend/package.json` and ship in **one PR** that closes both
  proposals. The two CVE classes are:
  - **Redirect-bypass** ŌĆö versions <0.27.1 in the experimental OIDC
    provider use a string-equality `redirect_uri` check that is bypassed
    by encoded / fragment / parameter-pollution variants, sending the
    OAuth code to attacker hosts before allowlist re-validation. Patch
    normalises the URI (decode, strip fragment, canonicalize host)
    before comparison.
  - **Metadata-SSRF** ŌĆö `experimentalClientIdMetadataDocuments` follows
    redirects without re-validating against the allowlist. The Backstage
    backend has outbound HTTP to Vault, mctl-api, and the kube-API via
    in-cluster DNS, so the SSRF can enumerate / exfil from those
    endpoints with the SA token. Defence-in-depth: explicit
    `auth.experimentalClientIdMetadataDocuments.enabled: false` plus a
    CI YAML drift-check.
- **Why:** these two CVEs are auth-class ŌĆö they bypass the **identity
  gate** of the entire portal stack. The blast radius is asymmetric
  with the rest of the v1.50.4 bundle: a successful redirect-bypass
  exfiltrates an authorization code that grants Dex SSO impersonation,
  and the SSRF then walks the in-cluster trust boundary. Same package
  bump for both; **always travel together** per the proposal authors,
  with the rollback procedure pre-documented as atomic.
- **Impact / Effort:** **5 / 1** (single yarn change, paired CI step).
- **Conflicts:**
  - **Bundle into the v1.50.4 yarn-up** (now eight-wide:
    `fetchurlreader-ssrf` + `scaffolder-symlink-traversal-v2` +
    `catalog-unprocessed-security-patch` + `catalog-facets-perf-fix` +
    `integration-credential-leak-fix` + `scaffolder-secret-leak` +
    `techdocs-path-traversal-fix` + `techdocs-mkdocs-code-exec` + this
    auth-backend pair). Yes, that brings the bundle to **ten CVE-class
    proposals** in a single yarn-up ŌĆö see Platform risks #4.
  - The CI YAML drift-check (`enabled: false`) must use a structured
    parser, not raw `grep` (per the proposal's risk note); one comment
    line containing `true` would otherwise produce a false positive.
- **Pre-flight:**
  - Run the full Playwright e2e Dex-SSO login flow on staging before
    promoting to `admins`. Test with **all** configured OAuth clients,
    not just the canonical one ŌĆö the strict normalisation could reject
    a legitimate redirect URI used by a Backstage plugin we have not
    audited.
  - `grep` the codebase + config history for any reference to
    `experimentalClientIdMetadataDocuments`. If anything still
    references it, surface that as a separate decision before flipping
    `enabled: false`.
  - The atomic-rollback property of the paired PR means **either both
    fixes revert or neither**. Document this in the PR body so a
    panicked partial revert does not happen during incident response.
- Link: `mctl-portal/proposals/auth-backend-redirect-bypass/`,

### 4. `mctl-agent/anthropic-sdk-timeout-default` ŌĆö close the silent-hang gap in the platform's only self-healing channel

- **What:** upgrade `anthropic-sdk-go` to **v1.39.0** (10-min default
  HTTP timeout) in `mctl-agent`'s `go.mod`, and override with a
  **5-minute application-level `http.Client` timeout** in the
  `LLMDiagnosis` skill. Tunable via `ANTHROPIC_TIMEOUT` env var on the
  ArgoCD Application manifest.
- **Why:** mctl-agent's circuit breaker
  ("auto-disables after N consecutive fails") **only fires on completed
  errors**. A hung in-flight Anthropic call produces no error and
  therefore never trips the breaker ŌĆö the alert pipeline silently
  blocks indefinitely. This is the platform's **only** self-healing
  channel (PR creation against `mctl-gitops` ŌåÆ ArgoCD reconcile). When
  it stalls, the platform stops repairing itself, and there is no
  louder failure signal because the breaker stays closed. The fix is
  one go.mod bump plus ~6 lines of skill-init code; the cost/benefit
  is unambiguous. This is the same failure-shape as Platform risks #8
  carried (silent-failure surfaces) ŌĆö third instance this week.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - No conflict with the Go-toolchain track (`go-toolchain-upgrade`
    1.26.x in W18, `go-toolchain-ace-cve-27140` in this pass) ŌĆö the
    SDK bump compiles on 1.24 just as well as 1.26.x. Land
    independently of the toolchain decision.
  - Companion (not duplicate) to `mcp-command-injection-hardening`
    (earlier-W19 Block E) ŌĆö same agent-binary, different surface.
- **Pre-flight:**
  - Confirm the SDK v1.39.0 release does not change any public Go API
    used by `LLMDiagnosis`. Per the proposal, no public-API breaks
    shipped, but verify `go vet ./...` is clean.
  - Set `ANTHROPIC_TIMEOUT` explicitly in the `admins`-tenant ArgoCD
    Application manifest at the same value (`5m`) as the hardcoded
    default. Explicit beats implicit; future ops change becomes a
    one-line edit, not a rebuild.
  - When the timeout fires, `LLMDiagnosis` returns a non-nil error,
    which **does** count toward the existing skill-runner circuit
    breaker. Verify this counting path on staging: send a request that
    will deliberately hang, confirm the breaker increments after
    timeout.
  - Record in the PR description that the 5-minute default covers the
    P99 of observed diagnose calls; if this assumption is wrong,
    expect spurious breaker trips during legitimate large-log analysis
    and tune via the env var.
- Link: `mctl-agent/proposals/anthropic-sdk-timeout-default/`.

## Platform risks

1. **The openclaw direction has a clean resolution ŌĆö finally, but only
   if the close-out happens.** Three weeks of churn (W17 4-x ladder ŌåÆ
   W18 5.2 direct ŌåÆ earlier-W19 4.8 emergency ŌåÆ this pass 5.3) end with
   2026.5.3 being a **strict superset** of every alternative on CVE
   coverage and a **strict superset** of 5.2 on regression closure. The
   risk now is not the choice ŌĆö it is the cleanup. Without an explicit
   "close five superseded slugs in the same gitops PR" step, the inbox
   carries four contradictory canonical targets and the next mentor
   cycle will see a fifth divergent proposal. **Recommendation:** the
   2026.5.3 PR description must list every superseded slug by name and
   reference the CVE IDs each one covered.

2. **`labs` RAM gate is still measurement-blocked, but 2026.5.3 reopens
   the pathway.** The earlier-W19 reversal to 4.8 closed off the
   negative-RSS expectation that the 5.x peer-dep externalisation
   creates. 2026.5.3 inherits it. The **only** remaining `labs`
   measurement opportunity in flight is the openclaw soak (the ArgoCD
   v3.4 RC7 staging from W18 is independent and orthogonal). Owners
   must record the 1 h / 6 h / 24 h RSS readings in `_mentor/digest/`
   so the value survives beyond the PR review window. Without that,
   `nodejs-heap-cap-labs` stays blocked for a fourth digest.

3. **MCP visibility still missing (carried, escalated for the third
   pass).** This refresh ŌĆö like both prior W19 and W18 passes ŌĆö reasons
   textually about service versions, open incidents, and tenant
   resource state because `mcp__mctl__*` is unavailable. The 2026.5.3
   decision in particular *needs* MCP visibility to confirm whether
   upstream issues #77095 / #77116 are resolved against our deployment
   posture; we are taking the proposal's word for it. Restoring
   `mcp__mctl__*` is now blocking three consecutive Top-5 #1 items.

4. **The Backstage v1.50.4 yarn-up is now ten CVE-class proposals
   wide.** W18 named four; earlier-W19 added three; this pass adds
   three more (`techdocs-mkdocs-code-exec` +
   `auth-backend-redirect-bypass` + `auth-backend-metadata-ssrf`). At
   ten, the bundle is past the point where "splitting it" helps ŌĆö
   audit value comes from the changelog/PR description, not from
   per-CVE PRs. **But** the **identity-gate class** of the auth-backend
   pair (Top-5 #3) is qualitatively different from the rest of the
   bundle, and the rollback procedure must explicitly call out that
   reverting the bundle reverts auth-backend too. Recommend: **one PR,
   ten CVEs, with a rollback runbook that pre-stages the auth-backend
   downgrade as a separate revert commit ready to fire if SSO breaks.**

5. **Cross-repo proposal duplication is structural, not coincidental.**
   This pass adds three more cross-repo / dual-slug duplicates:
   `argocd-secret-leakage-patch` Ōåö `argocd-serversidediff-secret-leak`
   (different CVE numbers, same fix); `argo-workflows-cve-bundle-upgrade`
   supersedes the earlier-W19 cross-repo pair
   (`mctl-agent/argo-workflows-cve-patch` +
   `mctl-gitops/argo-workflows-cve-patch(-v2)`); and the implicit
   coordination between `mctl-agent`'s W18 `go-toolchain-upgrade` and
   `mctl-api`'s `go-toolchain-ace-cve-27140` (same target version,
   different repos). **Recommendation, escalated from W18-W19:** the
   `supersedes:` field in the proposal template is no longer optional
   for any proposal touching a shared cluster component (Argo Workflows,
   ArgoCD chart, ESO chart, Vault chart, Prometheus, Loki, Grafana, Go
   toolchain). Without that field, the deduplication tax is now
   dominating mentor cycles for the third week running.

6. **The `mctl-agent/POST /mcp` endpoint is still unauthenticated.**
   Carried from earlier-W19 Platform risks #9. The
   `mcp-command-injection-hardening` proposal explicitly carved auth
   "out of scope"; no follow-up auth proposal exists. This pass adds a
   **second** unauthenticated MCP attack surface gap by way of
   `mcp-csrf-origin-validation`'s default ŌĆö if `MCP_ALLOWED_ORIGINS` is
   not set, all browser Origins are forbidden but the endpoint is still
   reachable to non-browser callers without auth. The architectural
   question ŌĆö should `mctl-api/MCP` and `mctl-agent/POST /mcp` share the
   same authentication scheme? ŌĆö has no owner. **Surface in W20 with
   forced ownership.**

7. **Silent-failure surfaces still accumulating; the Anthropic-SDK
   case is the most operationally severe yet.** Earlier-W19 Platform
   risks #8 named five silent-failure patterns (AlertManager schema
   drift, TechDocs symlink-escape served silently, ESO via DNS exfil,
   Wrangler `--commit-hash` shell injection invisible to GHA logs, ESO
   v2 `getHostByName` accepted at apply time). This pass adds a sixth
   that is qualitatively worse: **the platform's only self-healing
   channel hangs without firing the circuit breaker** because the
   Anthropic SDK had no default HTTP timeout (Top-5 #4). The cluster
   continues to reconcile, alerts continue to flow into mctl-agent, and
   the agent silently produces no output. The fix in this PR is the
   per-component patch; the systemic fix ŌĆö a "no-progress watchdog" at
   the skill-runner level ŌĆö is still missing. **Recommendation, per
   carry-forward:** the next architecture review must produce a silent-
   failure taxonomy and a generic "skill must produce progress within
   N seconds or the breaker increments" rule.

8. **CVE clustering still demands a release-train (carried, escalated
   for the third pass).** The two release-window shape from W18 holds
   exactly: (a) **cluster security** = ArgoCD (audit + floor
   annotation, no v3.4 yet) + Argo Workflows v3.7.14 + ESO chart 2.4.1
   (now three CVEs in one bump) + Vault release window (W19 four-CVE
   set) + Prometheus v3.11.3 + Loki v3.7.1 + Grafana Ōēźv12.1.10
   (conditional); (b) **data plane** = openclaw 2026.5.3 (Top-5 #1) +
   portal Backstage v1.50.4 (now ten-bundle, Top-5 #2 + #3) +
   mctl-agent Go-toolchain + mctl-api Go-toolchain (Block D) + the
   mctl-agent reliability patch (Top-5 #4). One ApplicationSet sync
   per release. Splitting further multiplies the soak load for no
   audit benefit.

9. **The dependency-tracking tier of the platform is structurally
   under-instrumented.** `automated-dep-updates` (Block F) names a
   concrete failure: the manual researcher missed workerd v1.20260505.1
   within 24 hours of v1.20260504.1, and the W19 `wrangler-cve-2026-0933`
   triage missed the same 48-hour CVE-disclosure lag. Renovate on
   `mctl-web` is the proportionate fix for the one repo that owns its
   own deploy pipeline; the **platform-level question** ŌĆö does every
   service get Renovate, or do we centralise dependency tracking at
   `mctl-gitops` ŌĆö is unanswered and should be its own W20 proposal.

10. **Multi-tenant isolation re-validation: still overdue, now five
    layers deep.** Carried from earlier-W19 Platform risks #10. ESO
    cross-namespace, ArgoCD project tokens, Vault token forwarding,
    openclaw `git:` plugin scheme, scaffolder GitHub token. This pass
    adds a sixth implicit layer via the auth-backend SSRF
    (`auth-backend-metadata-ssrf` Top-5 #3): the Backstage backend pod
    can SSRF the in-cluster Vault, mctl-api, and kube-API endpoints
    using its own SA token. The systemic re-validation has been
    overdue for three pass-es now. The per-component fixes are
    correct individually but do not generalise.
