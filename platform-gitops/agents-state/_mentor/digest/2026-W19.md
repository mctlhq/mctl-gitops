### 1. `mctl-gitops/loki-ruler-path-traversal` — unauth SA-token read off the Loki pod, single chart pin

- **What:** pin the Loki Helm chart to **v3.7.1** in
  `platform-gitops/services/loki/values.yaml`. v3.7.1 closes
  **CVE-2026-21726** — a double-URL-encoded path traversal in the Ruler
  API that lets an unauthenticated caller read arbitrary files off the
  Loki pod, including the projected service-account token at
  `/var/run/secrets/kubernetes.io/serviceaccount/token`.
- **Why this is the #1 add-on item:** the CVE is **pre-auth** and the
  endpoint is reachable from any in-cluster network peer that can hit
  the Loki Service. With the SA token in hand, the attacker pivots
  straight into the Kubernetes API at whatever permissions the Loki
  SA holds — which historically includes `secrets` read in the
  monitoring namespace. Loki currently runs on **`labs`** as part of
  the observability stack; a read of the labs Loki SA token therefore
  bridges directly into the labs tenant boundary that Wednesday's
  Platform risks #10 already flagged as "overdue for re-validation
  five passes running." Patch shape is a single chart-version bump —
  the audit cost is in confirming the rollout, not in the change.
- **Impact / Effort:** **5 / 1.**
- **Conflicts:**
  - **Bundle hint:** rides cleanly inside the cluster-security
    release window already named in W18 Platform risks #6 (carried,
    escalated through Wednesday's #8). One ApplicationSet sync,
    one rollback plan.
  - No conflict with `argocd-v3-4-upgrade-plan-v2` (Wednesday Top-5
    #5) — different chart, different reconciliation pass; bundle if
    convenient, ship independently otherwise.
  - Distinct from the carried-forward `prometheus-security-patch`
    (W18 Top-5 #5): different component, different CVE family —
    do not merge into the same audit list.
- **Pre-flight:**
  - Take the **`labs` RSS reading on the Loki pod** before and
    after the chart bump. This is the third independent labs-RAM
    measurement opportunity in flight (after openclaw 2026.5.4
    and ArgoCD v3.4 GA); record alongside the other two so
    `nodejs-heap-cap-labs` finally has three corroborating data
    points.
  - Confirm rolling-update strategy keeps at least one Loki
    replica available throughout the self-upgrade.
  - Rotate the Loki SA token after the rollout completes — even
    though the patch closes the read, treat any window of
    exposed-and-reachable Loki as a possible exposure event
    (mirrors the W18 `cloudflare-vite-plugin` rotation precedent).
  - Verify the Ruler API responds 404, not 200, for the test
    payloads in the proposal's CI smoke step.
- Link: `mctl-gitops/proposals/loki-ruler-path-traversal/`.

### 2. `mctl-api/argocd-token-scope-audit` — close the CVSS-10.0 ArgoCD authz bypass on the client side

- **What:** rotate the `ARGOCD_TOKEN` that `mctl-api` holds onto a
  minimal-scope project RBAC role; document the rotation in Vault;
  add a CI probe that fails the build if the running token can read
  outside the projects it owns. Three-phase: (1) generate a new
  scoped token in a non-prod project; (2) rotate the in-use token
  via Vault dynamic credentials; (3) revoke the legacy admin-equivalent
  token.
- **Why:** **CVE-2025-55190 (CVSS 10.0)** — the upstream advisory
  documents a complete authz bypass on the ArgoCD server side.
  Wednesday's Top-5 #5 promotes the platform to v3.4.0 GA, which
  closes the **server**-side issue. This proposal closes the
  symmetric **client**-side risk: `mctl-api` currently holds an
  ArgoCD token with admin-equivalent scope, so any compromise of
  the `mctl-api` pod yields full cluster sync authority. Scoping
  the token down means a future `mctl-api` compromise is bounded
  to the projects `mctl-api` legitimately operates on — which is
  the CVSS-10.0 reduction we want, not just the CVE patch.
- **Impact / Effort:** **5 / 3.**
- **Conflicts:**
  - **Pairs** with `argocd-v3-4-upgrade-plan-v2` (Wednesday Top-5
    #5). Schedule the token rotation for the **post-`labs`-soak,
    pre-`admins`-promotion** window of that proposal: the new
    token's project scope is verified against v3.4.0 GA semantics
    before it lands on `admins`.
  - Distinct from `mctl-api/argocd-ghsa-3v3m-patch` (verification-only,
    covered) — that proposal verifies the server is on v3.3.9; this
    one rotates the client token regardless of server version.
  - The new CI probe complements (does not replace) the
    `mcp-rce-server-hardening` audit table (Top-5 #3 here): both
    are observability layers on `mctl-api`, but at different
    failure surfaces.
- **Pre-flight:**
  - Document the existing token's actual scope before rotation —
    the operator should be able to show that the new project-scoped
    token can do everything `mctl-api` legitimately needs, and
    nothing else. A diff in Vault is the proof artifact.
  - Smoke-test every `mctl-api` ArgoCD caller (sync, app list,
    project list, app rollout) under the new token in a staging
    namespace before promoting.
  - Add an alert if `mctl-api` requests an out-of-project resource
    — that signal would have caught any latent over-reach.
- Link: `mctl-api/proposals/argocd-token-scope-audit/`.

### 3. MCP-hardening triple — `mctl-api/mcp-rce-server-hardening` + `mctl-agent/mcp-command-injection-hardening` + `mcp-input-validation`

- **What:** treat the three proposals as **one platform-wide MCP
  hardening release-train v1**. `mctl-api/mcp-rce-server-hardening`
  adds per-tool authz, a 64 KB request-body cap, write-tool rate
  limiting, and a Postgres `mcp_write_audit` append-only table.
  `mctl-agent/mcp-command-injection-hardening` adds per-tool
  allowlist middleware, the same body cap, a tool-name allowlist,
  and a 30-second `http.TimeoutHandler` wrapping the chi router.
  `mctl-agent/mcp-input-validation` ships a per-tool regex / enum /
  length schema registry inside `POST /mcp` for all six exposed
  tools. The pieces compose: ratelimit + body cap + per-tool authz +
  per-input schema = end-to-end input validation on both public
  `/mcp` surfaces.
- **Why:** **CVE-2025-49596 (CVSS 9.4)** is the nested-payload
  memory-exhaustion class against the MCP protocol; **CVE-2026-30623**
  is the stdio command-injection class. Both are exercised against
  the same external attack surface — anyone who can reach
  `mctl-api/MCP` or `mctl-agent/POST /mcp` can attempt them.
  Wednesday's Top-5 #3 (`mcp-go-v052-upgrade`) bumps the underlying
  library; the symmetric `mcp-csrf-origin-{validation,check}` pair
  (Wednesday Top-5 / earlier-W19) covers Origin-allowlist CSRF.
  Neither closes the RCE / injection class — that requires
  application-level authz, body limits, schema validation, and
  audit. This triple is the missing layer; its absence is the
  reason Wednesday's Platform risks #6 still sits at "open and
  needs a forced owner."
- **Impact / Effort:** **5 / 4** (composite — three repos, but each
  PR is small; the integration cost is in shared schema definitions
  and the audit table migration).
- **Conflicts:**
  - **Single ADR.** Document the three proposals as one
    architectural decision: shared schema registry shape, shared
    `MCP_ALLOWED_*` env-var family (extends the existing
    `MCP_ALLOWED_ORIGINS`), shared audit-table convention. One
    ADR avoids a future drift between the two endpoints.
  - **Coordinates** with `mcp-go-v052-upgrade` (Wednesday Top-5
    #3) and the symmetric `mcp-csrf-origin-{validation,check}`
    pair. Order: ship `mcp-go-v052-upgrade` first (library
    floor), then the CSRF pair (cheap, defensive), then the
    hardening triple (the heavy item). All three layers stack;
    none substitutes for the others.
  - The mctl-agent pair (`mcp-command-injection-hardening` +
    `mcp-input-validation`) **must** ship in the same PR — the
    schema registry references the allowlist middleware. Splitting
    them yields a half-validated endpoint that is worse than no
    hardening (false sense of coverage).
  - `mctl-api/mcp-rce-server-hardening` is independent of the
    mctl-agent pair at the code level — same pattern, separate
    repos, separate gitops PRs — but the ADR unifies them.
- **Pre-flight:**
  - The Postgres `mcp_write_audit` table needs a sizing decision
    before the PR opens (retention window + index strategy +
    GC policy). Treat as a separate decision artifact in
    `_mentor/digest/`.
  - Per-tool allowlist values must be enumerated before the
    middleware lands — there are six tools on `mctl-agent`, and
    the proposal does not enumerate the allowlist content. Owner
    must publish the list with the ADR.
  - 64 KB body cap is conservative; verify it does not regress
    the legitimate `tool/call` payloads (some MCP tool inputs
    can carry small files). A pre-merge log review on the prior
    week's `/mcp` traffic should confirm the 99.9th percentile
    request size sits below the cap.
  - Add the `runtime.Version()` + `mcp-go.Version()` startup
    log in both binaries (extends the Wednesday Top-5 #4
    pattern) so the deployed library + hardening versions are
    log-observable.
- Links: `mctl-api/proposals/mcp-rce-server-hardening/`,
  `mctl-agent/proposals/mcp-command-injection-hardening/`,
  `mctl-agent/proposals/mcp-input-validation/`.

### 4. `mctl-gitops/helm-security-patch` — three GHSAs in the Helm binary, fan-out across ArgoCD + Workflows + scaffolder

- **What:** update the **Helm binary** to **v4.1.4** in three
  places: (a) the ArgoCD container image, (b) the Argo Workflows
  step images that exec `helm`, (c) the Backstage scaffolder
  template runners that exec `helm`. Closes **GHSA-vmx8-mqv2-9gmg**
  (template-rendering symlink escape), **GHSA-hr2v-4r36-88hr**
  (chart-fetch path traversal), **GHSA-q5jf-9vfq-h4h7** (release
  history information disclosure).
- **Why:** Helm is shared across three of the platform's most
  privileged pods (ArgoCD has cluster-admin sync rights; Workflows
  steps run with project-scope service accounts; Backstage
  scaffolder runs with token-write capabilities). A vulnerability
  in the binary fans out across all three — a single malicious
  chart can pivot through any one of them. This is a
  **release-train-class** patch: the same `Chart.yaml`-line change
  closes three advisories at three separate failure surfaces.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - **Bundle into the cluster-security release window** alongside
    Wednesday Top-5 #5 (ArgoCD v3.4.0 GA), the W18-carried ESO
    chart 2.4.1, Vault, Prometheus v3.11.3, **Loki v3.7.1**
    (Top-5 #1 here), and Argo Workflows v3.7.14 (Wednesday Top-5
    #5 of the prior pass). One ApplicationSet sync, one rollback
    plan.
  - The Backstage scaffolder template paths overlap with the
    Wednesday Top-5 #2 yarn-up bundle (twelve-CVE Backstage
    rebuild). Verify the scaffolder image's `helm` binary version
    is the same as the cluster's after both PRs land — drift
    between the two would silently leave one of the three GHSAs
    open.
  - Distinct from `cert-manager-dns-dos-patch` (Top-5 #5 here) —
    different chart, different binary; bundle in the same
    cluster-security release window for review efficiency.
- **Pre-flight:**
  - Walk every `platform-gitops/services/*.yaml` and
    `platform-gitops/helm-charts/*.yaml` for the current Helm
    binary version reference; surface the audit list in the PR
    description.
  - Verify v4.1.4's chart-rendering output is byte-identical
    against a known-good chart in the test cluster — the
    template-symlink fix can in principle change rendered output
    if a chart relied on the previous behaviour.
  - Smoke-test the Argo Workflows step images that exec `helm`
    (typically `alpine/helm` derived) against a representative
    workflow; the binary swap must not change exit codes.
- Link: `mctl-gitops/proposals/helm-security-patch/`.

### 5. `mctl-gitops/cert-manager-dns-dos-patch` — DNS-triggered controller panic, one chart bump

- **What:** bump the cert-manager Helm chart to **v1.20.2** in
  `platform-gitops/services/cert-manager/values.yaml`. Closes
  **GHSA-gx3x-vq4p-mhhv** — a DNS-triggered controller panic that
  halts cluster-wide certificate renewal until the controller is
  manually restarted.
- **Why:** cert-manager is a single-replica controller in the
  current configuration. A panic stops every certificate renewal
  in the cluster; the failure mode is silent (renewals just stop)
  until a certificate near its expiry triggers an alert. The
  trigger is a malformed DNS response, which means any DNS
  provider compromise — or any DNS-spoofing position on the
  network path — can crash the controller without the attacker
  needing direct cluster access. Mitigation is a one-line chart
  pin; the soak cost is verifying renewals continue to succeed
  through the controller restart.
- **Impact / Effort:** **4 / 1.**
- **Conflicts:**
  - **Bundle** into the cluster-security release window (with
    Loki, Helm, ArgoCD GA, ESO 2.4.1, Vault, Prometheus, Argo
    Workflows). One sync, one rollback plan.
  - Independent of every other certificate-related proposal in
    flight — there is no overlap with Vault PKI, ESO, or
    cert-manager configuration changes.
- **Pre-flight:**
  - Confirm the chart values file does not need schema updates
    between the prior pinned version and v1.20.2.
  - Verify the controller restart does not lose the
    in-progress order list (cert-manager persists orders to
    its CRDs, but the operational check is still worth the
    five minutes).
  - Watch a short renewal cycle on a low-stakes cert in `labs`
    after the bump — the panic class is "happens hours later"
    on real DNS noise, so the post-deploy observation window
    should run for at least one full DNS-resolution cycle.
- Link: `mctl-gitops/proposals/cert-manager-dns-dos-patch/`.
