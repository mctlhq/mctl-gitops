### 3. `mctl-api/mcp-go-v052-upgrade` — fix the fd-leak that is silently degrading both `admins` and `labs`

- **What:** bump `mark3labs/mcp-go` from **v0.31 → v0.52.0** in
  `mctl-api/go.mod` and replace the manual chi `Post`/`Get` route
  registration on `/mcp` with the new transport-agnostic
  `(*StreamableHTTPServer).Handle(mux)` entry point. The single
  upstream change that matters is the fix for unclosed response bodies
  in the retry loop — every retried 404 in the prior v0.31 implementation
  leaked an fd and held a half-open TCP connection.
- **Why:** this proposal is **net resource-positive for both tenants**.
  `admins` was on a slow path to `EMFILE`/`ENFILE` (`accept(2)` failures
  → `/mcp` unavailable). `labs` was accumulating ~4–8 KB per leaked
  connection, which under typical retry traffic compounds to several MB
  of wasted memory — i.e., a **negative-RSS-delta** fix on a tenant
  already over-budget. This is the second `labs`-RAM-positive proposal
  in flight (the first is openclaw 2026.5.4). It also subsumes the
  earlier-W19 dual-track (`mcp-go-cve-waf-bypass`,
  `mcp-json-smuggling-investigation`) **if** v0.52.0 carries CVE-2026-27896
  fixes — verify in the upstream changelog. v0.52 also obsoletes the
  earlier-W19 `mcp-go-v051-upgrade` slug as the canonical mcp-go target.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - **Supersedes** earlier-W19 `mcp-go-v051-upgrade`. Adopt v0.52.0 as
    the canonical mcp-go target; close v0.51.
  - Coordinates (does not duplicate) with `mcp-csrf-origin-validation`
    (earlier-W19 Block D) and the symmetric `mctl-agent/mcp-csrf-origin-check`
    (Block E here). Either ship the chi middleware as the **interim**
    while v0.52 is staged (one source of truth: `MCP_ALLOWED_ORIGINS`),
    or ship v0.52 first and configure its built-in CORS with the same
    allowlist. ADR update documents the choice.
  - Independent of `go-upgrade-1262` (Top-5 #4 below) — the SDK bump
    compiles on Go 1.24 just as well as 1.26.x.
- **Pre-flight:**
  - ADR-0001 mandates MCP Inspector re-validation of all 24 tools on
    any mcp-go version bump. This is the heavy item in the effort
    score; do not skip.
  - Confirm v0.52.0 changelog references CVE-2026-27896. If yes, close
    `mcp-go-cve-waf-bypass` and `mcp-json-smuggling-investigation`. If
    no, ship the investigation's strict-case parser as a defensive
    monkeypatch.
  - Stage on `admins` first (mctl-api runs only there); record the fd
    count before/after via `lsof` or `/proc/<pid>/fd` — the proof point
    for the fix.
- Link: `mctl-api/proposals/mcp-go-v052-upgrade/`.

### 4. Go 1.26.2 toolchain — `mctl-api/go-upgrade-1262` + `mctl-agent/go-runtime-upgrade-v2` coordinated release

- **What:** bump both `mctl-api` and `mctl-agent` from Go **1.24 → 1.26.2**
  in the same release window. Each repo updates `go.mod` (the `go`
  directive plus a `toolchain go1.26.2` line), the Dockerfile builder
  base (`golang:1.26.2-alpine`), the CI matrix pin, and runs
  `go mod tidy`. `mctl-agent`'s proposal **bundles** chi v5.2.1 → v5.2.5
  in the same PR (no breaking changes; chi v5.2.5 requires Go ≥ 1.22,
  satisfied by 1.26.2).
- **Why:** ten stdlib CVEs are unpatched on Go 1.24, including
  CVE-2026-32283 (TLS 1.3 deadlock DoS, exercised on every outbound TLS
  call to Anthropic, GitHub, Vault, ArgoCD, kube-API), CVE-2026-32280 /
  32281 (x509 chain-build DoS), CVE-2026-32289 (html/template XSS), and
  the highest-severity item — CVE-2026-27140 (`cmd/go` arbitrary code
  execution at build time via crafted SWIG file names). The build-time
  ACE is a supply-chain risk: any malicious PR that lands in a
  build-stage path could exfiltrate Vault-mounted build secrets via
  the toolchain itself. This is the same Go-toolchain coordination
  flagged in earlier-W19 Platform risks #5; this proposal pair gives
  it a concrete shape. Go 1.26 also ships the Green Tea GC, which
  reduces GC CPU overhead 10–40% for long-running services with
  variable load — a free benefit on a security-driven bump.
- **Impact / Effort:** **4 / 2** (per-repo). Coordinated release
  window adds a small coordination tax, but amortises over both
  binaries.
- **Conflicts:**
  - **Supersedes** the W18 `mctl-agent/go-toolchain-upgrade` and
    `go-upgrade-1-26` slugs (same target). Adopt
    `go-runtime-upgrade-v2` as the canonical `mctl-agent` slug;
    close the W18 duplicates.
  - Cross-repo coordination: same target version, same release
    window, but separate gitops PRs. Schedule the `mctl-api` and
    `mctl-agent` deploys consecutively to amortise the rebuild and
    CI churn.
  - The W18 `mctl-api/go-toolchain-ace-cve-27140` slug names the
    same target version and CVE; that proposal can either be
    closed as a duplicate of `go-upgrade-1262` (canonical here) or
    folded as the CVE-rationale field on the canonical PR.
- **Pre-flight:**
  - Add `govulncheck` as a CI step on both repos; fail the build if
    any of the ten patched CVEs is detected post-bump.
  - Run integration tests covering OIDC login (`mctl-api`), Vault
    mTLS (both), and the AlertManager webhook flow (`mctl-agent`)
    before promoting. The crypto/tls and crypto/x509 changes are
    on every outbound TLS path.
  - Emit `runtime.Version()` at INFO level on startup in both
    binaries — the deployed version becomes log-observable.
  - Schedule the ArgoCD sync during a maintenance window.
- Link: `mctl-api/proposals/go-upgrade-1262/`,
  `mctl-agent/proposals/go-runtime-upgrade-v2/`.

## Platform risks

1. **The openclaw direction has churned for the fourth time in three
   weeks.** W17 4-x ladder → W18 5.2 direct → earlier-W19 4.8 emergency
   → earlier-W19 refresh 5.3 → this refresh **5.4**. 2026.5.4 is the
   first stable, strict-superset target in the chain; adopting it
   stops the churn — but only if the close-out step happens. The
   pattern itself is the risk: each direction shift carries an
   implicit "next week we'll choose differently" expectation. The
   2026.5.4 PR description **must** name every superseded slug and
   the CVEs each closure picks up; without that artifact, slug
   accretion will continue.

2. **`labs` RAM gate has now had four measurement opportunities
   come and go without producing a durable reading.** Both Top-5 #1
   (openclaw 5.4 with externalised peer deps; expected
   negative-RSS-delta) and Top-5 #5 (ArgoCD v3.4 GA in `labs`)
   require operator-recorded RSS values. The earlier-W19 refresh
   Platform risks #2 already escalated this; the same escalation
   applies. Without a durable record in `_mentor/digest/`,
   `nodejs-heap-cap-labs` stays blocked for a fifth digest, every
   future `labs`-touching feature proposal carries the same RAM
   uncertainty, and the platform's quantitative answer to "is `labs`
   over budget?" remains "we keep meaning to measure it."

3. **MCP visibility still missing — fourth pass blocked.**
   `mcp__mctl__*` tools remain unavailable. Three Top-5 #1 items in a
   row (and now four) have been chosen against the proposal's text
   alone, with no reconciliation against actual production image
   tags or open incidents. **Restoring `mcp__mctl__*` is now blocking
   four consecutive mentor cycles.** If it is not restored before
   W20, the digest cannot meaningfully audit upstream issues
   #77095 / #77116 against our deployment posture.

4. **The Backstage v1.50.4 yarn-up is now twelve CVE-class proposals
   wide.** W18 named four; earlier-W19 added three; earlier-W19
   refresh added three (techdocs RCE, auth-backend pair); this pass
   adds two more (`backstage-1-50-4-patch` + `scaffolder-cve-2026-24046-path-traversal`).
   At twelve CVEs in one rebuild, the yarn-up is past the splitting
   point. The auth-backend pair remains qualitatively
   different (identity-gate class) — the rollback runbook must
   pre-stage an auth-backend revert as a separate commit ready to
   fire on SSO breakage. Carried from earlier-W19 Platform risks #4,
   now compounded.

   structural.** This pass adds five more duplicate or near-duplicate
   pairs:
   (a) `upgrade-to-2026-5-4` ↔ `upgrade-to-2026-5-3` ↔ ... ↔ the

   week's mentor cycle is now spending more time on de-duplication
   than on net-new triage. Until the template change lands, the
   weekly digest can only paper over the deduplication tax, not
   eliminate it.

   is still open and still needs a forced owner.

   interactive-prompt class is the new addition.**

   that triggers operator attention. The fix is per-component;
   the systemic "no-progress watchdog at the deploy / skill-runner

   #7.

   escalated).** The two release-window shape from W18 / earlier-W19

   mctl-api MCP (Top-5 #3) + mctl-web wrangler 4.88.0 trio (Block

    Vault-backed ExternalSecret). The per-component fixes are
    correct individually but do not generalise. The systemic
    re-validation is overdue for five mentor passes; without it,
    the Nth+1 layer is already implicit in any new feature
    proposal that touches a tenant boundary.
