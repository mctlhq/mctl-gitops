### 1. `mctl-openclaw/emergency-cve-patch` — direction reversal: 2026.4.8 vs the just-adopted 2026.5.2

- **What:** upgrade every tenant from **2026.3.14 → 2026.4.8** (not 2026.5.2,
  as the W18 Top-5 #1 currently directs). The proposal cites two upstream
  regressions in 2026.5.2 — issue #77095 (gateway crash at ~12 h uptime) and
  issue #77116 (Feishu channel incompatibility) — and argues that 2026.4.8 is
  the **first stable release** that closes the four highest-boundary CVEs
  (CVE-2026-42422, 42426, 42428, 41914) without inheriting the 5.2
  regressions. WebSocket DoS (CVE-2026-41399) is intentionally **not**
  addressed by this hop and is delegated to `ws-dos-rate-limit` (Block A).
- **Why this is the #1 item of the week:** the W18 digest *just* reversed
  course in the opposite direction (4-x ladder → direct 5.2 hop). Two
  successive reversals on the canonical openclaw target inside two weeks is
  unsustainable; the inbox is now carrying **three** contradictory signals
  (4-x ladder, 5.2 direct, 4.8 emergency). The mentor's job here is not to
  pick a winner unilaterally — it is to surface the conflict and demand a
  **single decision artifact** before any openclaw PR opens.
- **Impact / Effort:** **5 / 3** as a discrete upgrade; **5 / 5** as a
  decision because the input data (gateway crash, Feishu incompat) needs
  reconciliation against `mcp__mctl__*` (currently unavailable) and against
  whether `ovk` actually uses Feishu.
- **Conflicts:**
  - **Direct conflict** with W18 Top-5 #1 (`upgrade-to-2026-5-2`) and with
    the four superseded 4-x rungs (4-25, 4-26, 4-27, 4-29) it consolidates.
    These six proposals together name **three** different canonical targets
    (4.8, 4.29, 5.2). Owners must close five of the six.
  - **Subsumes** the W18 cve-addenda (`cve-addendum-auth-bypass-exec-bypass`
    + `cve-addendum-ssrf-eval-bypass`) only **partially** — 2026.4.8 closes
    the CVE-2026-414xx set named in the addenda, but does **not** close
    CWE-532 (credential-in-logs); the openclaw 5.2 path did. If 4.8 is
    chosen, `cwe532-log-credential-sanitization` (W18 Block A) becomes a
    permanent control rather than a time-bounded interim.
  - **Independent** of `git-plugin-install-allowlist` (W18 Top-5 #4) — the
    `git:` plugin scheme only exists in 2026.5.2; if the platform takes 4.8
    instead, the allowlist becomes a future-proofing item (still worth
    landing, but no longer a release-window prerequisite).
- **Pre-flight:**
  - **Decision artifact required** (one-page note in `_mentor/digest/`):
    which target version, which evidence, which CVEs each version closes,
    which regressions remain. Without that, no PR should open.
  - Verify the 5.2 regressions cited by the proposal (#77095, #77116). If
    they are fixed in a 5.2.x patch already, the W18 5.2 path stands.
  - Confirm `ovk` Feishu usage — if `ovk` does not use Feishu, the #77116
    argument against 5.2 collapses for our platform.
  - Audit `device.token.rotate` response-parsing tooling **before** the
    `ovk` step (this prerequisite gate is unchanged regardless of which
    target is chosen).
  - If 4.8 is chosen, **promote** `cwe532-log-credential-sanitization` from
    "interim" (W18 Block A) to a permanent DaemonSet control, since 4.8
    does not ship the in-process credential redaction that 5.2 carries.
- Link: `mctl-openclaw/proposals/emergency-cve-patch/` (vs
  `mctl-openclaw/proposals/upgrade-to-2026-5-2/`).

- **What:** record CVE-2026-43824 (CVSS 9.6, GHSA-3v3m-wc6v-x4x3) as the

  ArgoCD users can extract plaintext Kubernetes Secret content via the
  ServerSideDiff endpoint when `IncludeMutationWebhook=true`. Fixed in
  v3.3.9 / v3.2.11.
- **Why:** the platform is **already on v3.3.9** (per W18 carry-forward), so
  this CVE is **already remediated by the existing pin**. The proposal's
  remaining value is (a) version-floor annotation in the Application
  manifest (so a future revert below v3.3.9 fails CI), and (b) an explicit
  audit artifact for the CVE-2026-43824 paper trail. The blast radius if a
  revert ever silently lowered the pin is platform-wide secret exposure
  across both tenants — an annotation that prevents that is cheap and
  durable.
- **Impact / Effort:** **4 / 1** (impact captures the prevented-revert
  scenario; effort is a single PR).
- **Conflicts:**
  - **Bundles** with `argocd-v3-4-upgrade-plan` (W18 Top-5 #3) and
    `argocd-xss-version-verify` (W18 Block D): one Application manifest,
    one PR, three CVE annotations stacked (CVE-2025-47933 floor v3.3.0,
    CVE-2026-43824 floor v3.3.9, plus the v3.4 staging change).
  - **Distinct** from `argocd-image-updater-ns-escalation` and
    `argocd-repo-cred-leak-patch` (W18) — those are component-level
    patches at the v3.3.x line and ship independently.
  - `argocd-informer-cache-patch` is **superseded** by the existing v3.3.9
    pin (it targets v3.3.7 → v3.3.8). Close it in the same PR.
- **Pre-flight:**
  - Add a CI rule that blocks any commit lowering the ArgoCD image pin
    below v3.3.9. The annotation alone is documentation; the CI rule is
    enforcement.
  - Confirm no test or staging Application manifest still pins ≤v3.3.8
    anywhere under `platform-gitops/`.
- Link: `mctl-gitops/proposals/argocd-serversidediff-secret-leak/`.

### 3. `mctl-gitops/eso-dns-exfil-patch` — secret exfil via ESO v2 template Sprig function

- **What:** upgrade ESO to v2.2.1+ (Helm chart `helm-chart-2.4.1`, already in
  flight per W18) to remove `getHostByName` from user-accessible
  ExternalSecret templates. CVE-2026-34984 (High) lets any contributor with
  PR access to `platform-gitops/argo-workflows/secrets/` craft a templated
  ExternalSecret that encodes a Vault secret into a DNS hostname; the ESO
  controller then performs a DNS lookup and the secret is exfiltrated
  through DNS (which the egress firewall allows by default).
- **Why:** the entire platform's secret distribution depends on the
  `vault-backend` ClusterSecretStore. A controller-side DNS-side-channel
  bypasses *every* existing egress control because DNS itself is
  whitelisted. The fix is a Helm chart bump, not a code change. This folds
  cleanly into the W18-named `helm-chart-2.4.1` upgrade rather than being a
  standalone PR.
- **Impact / Effort:** **5 / 1.**
- **Conflicts:**
  - **Bundles** with the carry-forward `helm-chart-2.4.1` ESO upgrade and
    with `eso-cross-namespace-secret-access` (W18). Same chart, same
    rollout — one PR.
  - `eso-cve-patch` is a generic-slug duplicate that almost certainly
    refers to the same fix; close as superseded by this proposal once the
    CVE list is reconciled.
- **Pre-flight:**
  - `grep` every `*.yaml` under `platform-gitops/argo-workflows/secrets/`
    and per-tenant service configs for any current use of
    `getHostByName` in ExternalSecret templates. Surface the audit list
    in the PR description; the fix rejects such templates with
    `Ready=False / TemplateFunctionDenied`, so any pre-existing legitimate
    use will break.
  - Confirm `helm-chart-2.4.1` is the version that ships ESO ≥v2.2.1.
- Link: `mctl-gitops/proposals/eso-dns-exfil-patch/`.

### 5. `mctl-agent/argo-workflows-cve-patch` — controller crash kills self-healing within 30 minutes

- **What:** upgrade Argo Workflows to **v4.0.5** (or v3.7.14 on the v3
  branch) to close CVE-2026-40886 (CVSS 7.7 High). Affected versions
  v3.6.5–4.0.4 panic the controller on a malformed `pod-gc-strategy`
  annotation, halting all Workflow processing cluster-wide.
- **Why:** mctl-agent's `cwft-rotate-github-token` CronWorkflow runs **every
  30 minutes** in the `admins` tenant. If the controller falls over: token
  rotation stops → within ≤30 min the GitHub token expires → `go-github`
  client returns 401 on every call → **PR creation (the agent's only
  self-healing channel) is fully unavailable**. The exploit prerequisite
  is the ability to set an annotation on any pod in the cluster — well
  within the trust boundary of any compromised tenant workload. Combined
  with the ArgoCD reconciliation surface, a single malformed annotation
  takes the platform's auto-remediation offline. Fix is a single image
  pin; CVE-2026-31892 and CVE-2026-28229 are **also** closed by v4.0.5.
- **Impact / Effort:** **4 / 2.**
- **Conflicts:**
  - **No conflict** with the parallel `mctl-gitops/argo-workflows-cve-patch`
    proposal — they are the same fix in two repos. Confirm whether the
    upgrade lives in `mctl-agent` (the controller is in the `admins`
    tenant, GitOps-managed) or in `mctl-gitops` (where the chart pin
    lives) and pick one canonical PR; close the other as duplicate.
  - `argo-workflows-cve-patch-v2` (in `mctl-gitops`) is the v2 slug for
    the same fix; pick one, close the other.
- **Pre-flight:**
  - Pin via `mctl-gitops` (chart values), not by editing `mctl-agent`
    directly.
  - Re-run the existing CronWorkflow smoke tests post-upgrade.
  - Verify no Workflow definitions in `platform-gitops/argo-workflows/`
    use a custom `pod-gc-strategy` annotation that would now be more
    strictly validated.
- Link: `mctl-agent/proposals/argo-workflows-cve-patch/` (and
  `mctl-gitops/proposals/argo-workflows-cve-patch-v2/`).

## Platform risks

1. **The openclaw direction has now changed twice in two weeks.** W17 named
   the 4-x ladder; W18 swung it to a direct 2026.5.2 hop; this pass surfaces
   `emergency-cve-patch` arguing for 2026.4.8 instead. The inbox now carries
   **three** contradictory canonical targets simultaneously. **Recommendation:**
   the operator must publish a one-page decision artifact in
   `_mentor/digest/` naming a single target, the evidence used to choose
   it, and the proposals to close. Without that artifact the next mentor
   cycle will see a **fourth** divergent proposal.

2. **`labs` RAM gate is still measurement-blocked.** W18 named two
   independent measurement opportunities (openclaw 2026.5.2 RSS delta;
   ArgoCD v3.4 RC7 RAM in `labs`) that would finally produce a quantitative
   `labs` headroom number. Neither has landed, and the W19 openclaw
   reversal pushes the openclaw measurement out further: 2026.4.8 has **no
   externalised peer deps**, so the RSS delta reasoning that justified the
   5.2 path does not transfer. The ArgoCD RC7 staging is now the **only**
   remaining `labs` measurement pathway in flight. `nodejs-heap-cap-labs`
   stays blocked.

3. **MCP visibility still missing (carried, escalated).** This is now the
   third digest in a row reasoning textually about service versions
   because `mcp__mctl__*` is unavailable. W19's Top-5 #1 (openclaw
   reversal) explicitly *needs* MCP visibility — we cannot tell from
   inside the proposals tree whether 2026.5.2 issues #77095 / #77116 are
   open against our deployment, fixed in a 5.2.x patch, or affect us at
   all. **Restoring MCP visibility is now blocking the highest-impact
   decision of the week.**

4. **The Vault stack has grown four CVE entries in one pass.**
   `vault-kvv2-deletion-bypass` (CVE-2026-3605), `vault-ssrf-acme-patch`
   (CVE-2026-5052 / HCSEC-2026-06), `vault-ldap-null-bind-bypass`
   (CVE-2025-13357), plus the W18 carry-forward `vault-cve-2026-token-exposure`.
   All four hit the same single secret store that the entire platform
   depends on. **Recommendation:** package the Vault upgrade as a single
   release window (Vault server v2.0.0 + Terraform-provider pin + KVv2
   policy tightening) rather than four serial PRs. The breaking-change
   handling for v2.0.0 amortises across all four CVEs, and the rollback
   surface stays single-rollback.

5. **MCP is now a three-surface attack-class** (mctl-api `/mcp`, mctl-agent
   `POST /mcp`, library-level `mark3labs/mcp-go`). Three CVE / hardening
   proposals hit those surfaces this pass:
   `mctl-api/mcp-go-cve-waf-bypass` (library, CVE-2026-27896),
   `mctl-api/mcp-rce-server-hardening` (server-side controls,
   CVE-2025-49596 class), `mctl-agent/mcp-command-injection-hardening`
   (mctl-agent endpoint, CVE-2026-30623). The proposals also reveal that
   `mctl-agent/POST /mcp` is **currently unauthenticated**, which the
   proposal explicitly carves "out of scope" for itself. **The auth gap is
   itself a separate platform-level proposal that does not yet exist** —
   surface it in W20 and require an owner.

6. **CVE clustering still demands a release-train (carried, escalated
   again).** This pass adds Argo Workflows v4.0.5, Vault v2.0.0 (+ TF
   provider + KVv2 policies), ESO v2.2.1+, ArgoCD v3.3.9 floor annotation,
   Loki v3.7.1, Grafana v12.1.10+ (conditional), and three more Backstage
   CVEs into the v1.50.4 yarn-up. Plus the W18 carry-forward (Prometheus,
   ArgoCD v3.4 RC7, openclaw 4.8/5.2, Backstage four-bundle). **Two
   release windows are still the right shape:** (a) **cluster security**
   (ArgoCD audit + Argo Workflows + ESO + Vault + Prometheus + Loki +
   Grafana-if-deployed + Helm), (b) **data plane** (openclaw chosen
   target + portal Backstage v1.50.4 seven-bundle + mctl-agent Go patch).
   Splitting further multiplies soak load.

7. **Duplicate clusters are still growing — and now span repos.** This
   pass adds: `eso-cve-patch` ↔ `eso-dns-exfil-patch`,
   `argocd-informer-cache-patch` ↔ current v3.3.9 pin,
   `mcp-json-smuggling-investigation` ↔ `mcp-go-cve-waf-bypass`,
   `mctl-agent/argo-workflows-cve-patch` ↔
   `mctl-gitops/argo-workflows-cve-patch(-v2)`,
   `wrangler-cve-2026-0933` ↔ merged `wrangler-upgrade-security`,
   `nodejs-security-march-2026` ↔ W18 `nodejs22-security-batch`. **The
   `supersedes:` field in the proposal template is now unconditionally
   blocking; without it, the W20 deduplication tax will dominate the
   mentor cycle.** Also: the cross-repo duplicates (Argo Workflows in
   both `mctl-agent` and `mctl-gitops`) need an explicit "ownership of
   shared cluster components" rule — proposals for shared components
   should originate in `mctl-gitops` only.

8. **Silent-failure surfaces keep accumulating.** AlertManager schema
   drift (`alertmanager-webhook-v032`), TechDocs symlink-escape served
   silently to all viewers (`techdocs-path-traversal-fix`),
   ESO-via-DNS exfil (`eso-dns-exfil-patch`), Wrangler `--commit-hash`
   shell injection invisible to GHA logs (`wrangler-cve-2026-0933`),
   ESO v2 template `getHostByName` accepted at apply time. The platform
   now has **at least five named silent-failure attack patterns**.
   **Recommendation:** the next architecture review should produce a
   dedicated taxonomy of silent-failure modes; per-component fixes are
   correct individually but do not generalise to the next case.

9. **The `mctl-agent/POST /mcp` endpoint is unauthenticated** (per its
   own hardening proposal). This is a footnote-level disclosure that
   should be a Top-5 item once it surfaces as its own proposal. For now,
   `mcp-command-injection-hardening` provides defence-in-depth (input
   validation + size caps + audit) but not authentication. Track this
   gap explicitly into W20.

10. **Cross-tenant blast radius via scaffolder GitHub token (new
    surface).** The `@backstage/integration` CVE
    (CVE-2026-29185, Top-5 #4 component) describes a path where a crafted
    SCM URL exfils the **scaffolder's GitHub App token**, which has
    write access to `mctl-gitops`, which is the source of truth for
    every ArgoCD-managed Application across both tenants. This is a
    **new** path in the multi-tenant isolation threat model (W18
    Platform-risks #3) and brings the layer count to **five**: ESO
    cross-namespace, ArgoCD project tokens, Vault token forwarding,
    openclaw `git:` plugin scheme, scaffolder GitHub token. The
    end-to-end isolation re-validation has been overdue for two pass-es
    now.
